## Общее правило для всех глав

Каждый пункт в «Типовые слабые места» должен отвечать на вопрос: **как это выглядит в реальности и как это найти?**

Формат замены:
```
Было:
- открытая рекурсия

Должно быть:
- открытая рекурсия — любой внешний хост может делать DNS-запросы через твой сервер,
  что используют для amplification DDoS и обхода политик.
  Проверить: `dig @YOUR_SERVER_IP any google.com` с машины вне твоей сети.
  Если ответ пришёл — рекурсия открыта.
```

Применить этот формат к разделам 1.2, 2.2, 3.2, 4.2, 5.2, 6.2, 7.2 — во всех книгах части 3.

### Глава 3 (`chapter-03.md`) — VLAN и сегментация

**Проблема:** Матрица доступов показана в виде текстовой схемы, но нет команд для проверки что изоляция реально работает.

**Добавить** в раздел 3.4 после Шага 2 — проверка изоляции между зонами:

```bash
# С машины в зоне users попробовать достучаться до management:
nc -vz MANAGEMENT_IP 22
# Ожидаемо: Connection refused или timeout (не открыто)

# С машины app попробовать достучаться до db:
nc -vz DB_IP 5432
# Ожидаемо: Connected (разрешено по матрице)

# С IoT попробовать достучаться до servers:
nc -vz SERVER_IP 443
# Ожидаемо: timeout (заблокировано)
```

Разница между `Connection refused` и `timeout`:
- `refused` — хост ответил, порт закрыт — трафик прошёл через firewall
- `timeout` — пакет отброшен — firewall/VLAN изолирует

**Добавить** раздел **3.5 «Захват трафика для проверки»** (lab-only):

```bash
# На хосте в зоне servers наблюдать трафик
sudo tcpdump -ni eth0 src MANAGEMENT_IP -c 10

# С management отправить пинг
ping SERVER_IP

# Пакеты должны быть видны в tcpdump → трафик management→servers разрешён
# Если пакетов нет → изоляция работает
```

### Глава 6 (`chapter-06.md`) — DNS

**Проблема:** Самая слабая глава. Риск «открытая рекурсия» описан одной строкой. Нет ни одной команды для проверки.

**Полностью заменить раздел 6.2 «Типовые слабые места»** на расширенный:

**Открытая рекурсия** — твой DNS отвечает на запросы от посторонних:

```bash
# Проверить с внешней машины (не из своей сети)
dig @YOUR_SERVER_IP any google.com

# Если пришёл ответ — рекурсия открыта, это проблема
# Если REFUSED или timeout — правильно
```

Почему это плохо: используется для DNS Amplification DDoS — маленький запрос вызывает большой ответ от твоего сервера к жертве.

**Zone transfer** — утечка всех DNS-записей зоны:

```bash
# Попробовать zone transfer своего домена
dig @YOUR_NS_IP AXFR yourdomain.com

# Ожидаемо: Transfer failed — правильно, transfer разрешён только для secondary NS
# Если пришёл список записей — zone transfer открыт
```

**Устаревшие записи** — как найти:

```bash
# Посмотреть все A-записи домена
dig +short A yourdomain.com
dig +short A subdomain.yourdomain.com

# Проверить, отвечает ли реальный сервер
curl -so /dev/null -w "%{http_code}" https://subdomain.yourdomain.com
# 000 = сервера нет, запись устарела — удалить
```

**Нет журналов DNS** — как включить в systemd-resolved:

```bash
# Проверить текущий уровень логирования
resolvectl log-level
# Включить DNS-запросы в лог
sudo resolvectl log-level debug
journalctl -u systemd-resolved -f
```

**Добавить** в раздел 6.5 «Lab-only проверка» конкретные команды:

```bash
# На своём lab DNS сервере (bind9/unbound) проверить рекурсию
# С другой машины в том же стенде:
dig @LAB_DNS_IP any google.com

# Проверить логи запросов
sudo journalctl -u named -n 30 --no-pager
# или
sudo journalctl -u unbound -n 30 --no-pager
```
