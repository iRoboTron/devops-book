Каждый практический блок должен отвечать на три вопроса:

Если любой из трёх пунктов отсутствует — раздел нужно дописать.

**Проблема:** раздел 1.5 «Практика» перечисляет команды без примера вывода. Непонятно, на что смотреть в результате.

**Что добавить:**

В раздел 1.5 после каждой команды добавить пример вывода и объяснение что именно искать:

```bash
ss -tulpn
```

Пример вывода (сокращённый):
```
Netid  State  Local Address:Port  Process
tcp    LISTEN 0.0.0.0:22          sshd
tcp    LISTEN 0.0.0.0:5432        postgres
tcp    LISTEN 127.0.0.1:8000      python3
```

Добавить пояснение: строка `0.0.0.0:5432` — PostgreSQL слушает всех. Строка `127.0.0.1:8000` — backend виден только локально. Задача — чтобы публичных строк было ровно столько, сколько нужно.

Аналогично для `nmap`: добавить пример вывода до и после ufw, чтобы читатель видел разницу:
```
# До ufw
PORT     STATE  SERVICE
22/tcp   open   ssh
80/tcp   open   http
5432/tcp open   postgresql   ← это плохо

# После ufw deny + allow 22,80,443
PORT     STATE    SERVICE
22/tcp   open     ssh
80/tcp   open     http
5432/tcp filtered postgresql  ← уже лучше
```

### Глава 3 (`chapter-03.md`) — SSH hardening

**Проблема:** глава в целом хорошая, но раздел 3.3 показывает конфиг без примера того, как выглядит лог при атаке. Читатель настроил защиту, но не знает как убедиться что она работает.

**Что добавить:**

1. Новый раздел **3.5а «Как выглядит атака в логах»** (между 3.5 и 3.6):

Показать реальный пример из `auth.log` при brute-force:
```
Apr 15 03:12:01 sshd[12345]: Failed password for invalid user admin from 185.234.X.X port 44321 ssh2
Apr 15 03:12:03 sshd[12346]: Failed password for invalid user root from 185.234.X.X port 44322 ssh2
Apr 15 03:12:05 sshd[12347]: Failed password for invalid user ubuntu from 185.234.X.X port 44323 ssh2
```

Объяснить: именно такие строки читает fail2ban. После `MaxAuthTries 3` каждая сессия даёт максимум 3 попытки — это видно в логе как 3 строки `Failed` с одного IP за секунды.

2. Добавить команду быстрого подсчёта атак:
```bash
sudo grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn | head -10
```

Пример вывода: топ-10 IP по количеству неудачных попыток. Это даёт понимание масштаба до включения fail2ban.

### Глава 5 (`chapter-05.md`) — Reverse proxy, rate limiting и WAF basics

**Проблема:** раздел 5.2 описывает rate limiting концептуально, но не даёт реальный конфиг nginx. Раздел 5.3 перечисляет security headers без конфигурации.

**Что добавить:**

1. В раздел 5.2 — конкретный рабочий конфиг nginx с rate limiting:

```nginx
# /etc/nginx/nginx.conf или в http {}
limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;

# В server {} блоке
location /login {
    limit_req zone=login burst=3 nodelay;
    proxy_pass http://127.0.0.1:8000;
}
```

Объяснить параметры:
- `5r/m` — максимум 5 запросов в минуту с одного IP
- `burst=3` — допускает кратковременный всплеск до 3 запросов
- `nodelay` — лишние сразу получают 503, а не ждут в очереди

Показать как выглядит ответ при превышении лимита:
```bash
curl -I https://HOST/login
# HTTP/2 503 при превышении
```

2. В раздел 5.3 — добавить готовый nginx-блок с security headers:

```nginx
# В server {} или location {}
add_header X-Frame-Options "SAMEORIGIN" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

И как проверить что headers реально отдаются:
```bash
curl -sI https://HOST | grep -E 'x-frame|x-content|strict-transport|referrer'
```

### Глава 6 (`chapter-06.md`) — Fail2ban и CrowdSec

**Проблема:** самая слабая глава. Раздел 6.2 (CrowdSec) — всего 3 буллета без конфигурации. Нет реального `jail.local`. Нет критериев выбора между fail2ban и CrowdSec. Нет примера что видно в логах после бана.

**Что добавить — полностью переписать разделы 6.2–6.4:**

**6.2 — Рабочая конфигурация fail2ban:**

```ini
# /etc/fail2ban/jail.local
[DEFAULT]
bantime  = 1h
findtime = 10m
maxretry = 5
backend  = systemd

[sshd]
enabled = true
port    = ssh
logpath = %(sshd_log)s

[nginx-http-auth]
enabled  = true
port     = http,https
logpath  = /var/log/nginx/error.log
maxretry = 3
```

Объяснить каждый параметр:
- `bantime = 1h` — IP заблокирован на 1 час
- `findtime = 10m` — `maxretry` считается в этом окне
- `maxretry = 5` — 5 ошибок за 10 минут = бан
- `backend = systemd` — читать journald, а не файл (актуально для Ubuntu 22+)

**6.3 — Как выглядит бан в логах:**

```bash
sudo journalctl -u fail2ban -n 20 --no-pager
```

Пример что ищем:
```
fail2ban.actions: NOTICE  [sshd] Ban 185.234.X.X
fail2ban.actions: NOTICE  [sshd] Unban 185.234.X.X
```

И как посмотреть текущие активные баны:
```bash
sudo fail2ban-client status sshd
```

Пример вывода:
```
Status for the jail: sshd
|- Filter
|  |- Currently failed: 2
|  |- Total failed:     47
`- Actions
   |- Currently banned: 1
   |- Total banned:     8
   `- Banned IP list: 185.234.X.X
```

**6.4 — Когда выбирать fail2ban, а когда CrowdSec:**

Добавить таблицу критериев:

| Критерий | fail2ban | CrowdSec |
|---|---|---|
| Один VPS, один человек | ✅ хватит | избыточно |
| Несколько серверов | ❌ каждый настраивать отдельно | ✅ централизованно |
| Только SSH | ✅ просто | ✅ тоже просто |
| HTTP + API + SSH | возможно, но конфиги сложнее | ✅ удобнее |
| Community threat intelligence | ❌ нет | ✅ есть |
| Сложность старта | низкая | средняя |

**Вывод:** для первого VPS fail2ban — правильный выбор. CrowdSec рассмотреть когда серверов больше одного или нужна защита HTTP-endpoint'ов с гибкой логикой.

### Глава 7 (`chapter-07.md`) — IDS/IPS и сетевые сигналы

**Проблема:** раздел 7.2 просто перечисляет инструменты (Suricata, Snort, Zeek) без объяснения в чём разница. Читатель не знает что выбрать.

**Что добавить:**

1. В раздел 7.2 — добавить краткое сравнение:

| Инструмент | Что делает | Когда нужен |
|---|---|---|
| Suricata | Сигнатурный IDS/IPS, глубокий анализ пакетов | Small business+, если есть кто разбирать алерты |
| Snort | Классический IDS, много готовых правил | Учебные цели, legacy среды |
| Zeek | Сетевая телеметрия: не «бан», а «покажи что происходит» | Аналитика, forensics, когда важен контекст |

Пояснение: Suricata умеет работать как IPS (блокировать), Zeek только наблюдает. На одном VPS без команды аналитиков Zeek или Suricata создадут больше шума, чем пользы.

2. В раздел 7.4 — расширить tcpdump-пример: показать как выглядит scan в выводе:

```bash
sudo tcpdump -ni eth0 'tcp[tcpflags] & (tcp-syn) != 0' -c 20
```

Пример вывода при nmap-скане:
```
03:15:01 IP 10.0.0.5.44321 > 10.0.0.10.22: Flags [S]
03:15:01 IP 10.0.0.5.44322 > 10.0.0.10.80: Flags [S]
03:15:01 IP 10.0.0.5.44323 > 10.0.0.10.443: Flags [S]
03:15:01 IP 10.0.0.5.44324 > 10.0.0.10.3306: Flags [S]
```

Пояснить: много SYN-пакетов за долю секунды с одного IP на разные порты — это port scan. Именно это пытается поймать IDS-движок.

Running 10s test @ http://HOST/
