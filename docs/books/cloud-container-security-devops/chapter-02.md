# Глава 2: IAM и минимальные права

> **Запомни:** в облаке компрометация часто начинается не с root на сервере, а с лишних прав у роли, токена или CI-пользователя.

---

## 2.1 Контекст и границы

Облачный контроль доступа позволяет очень точно ограничивать действия, но в реальности многие проекты живут на широких ролях вроде admin, потому что так быстрее стартовать.

Минимальные права — это рабочая практика: разделять human access, CI access, runtime access и break-glass сценарии.

Эта глава особенно полезна для всех, кто использует cloud IAM, service accounts, API tokens и automation.

---

## 2.2 Как выглядит риск

Типовые слабые места:
- одна роль дает доступ ко всем сервисам;
- CI/CD использует long-lived static keys;
- runtime сервис может читать больше секретов и ресурсов, чем нужно;
- нет разделения на чтение, запись и администрирование;
- нет журнала использования привилегированных ролей.

### Где особенно важно
- VPS API
- object storage
- managed databases
- CI/CD
- контейнерные runtime identities

---

## 2.3 Что строит защитник

- разделение ролей по функциям;
- короткоживущие токены и federation там, где возможно;
- отдельные роли для runtime, CI и людей;
- review effective permissions;
- журналирование и алерты на привилегированные действия.

### Практический результат главы
- ты можешь показать, кто чем управляет в облаке;
- понимаешь, почему static access keys — долгосрочный риск;
- умеешь проектировать роль под конкретный сервис.

```text
Allow: s3:GetObject on arn:aws:s3:::myapp-assets/*
Deny:  s3:* on *
```

---

## 2.4 Практика

### Шаг 1: Инвентаризируй identities
- перечисли людей, сервисы, CI jobs и роли, которые имеют доступ к облаку;
- для каждой сущности опиши назначение и срок жизни.

```bash
printf 'human-admin
ci-deploy
app-runtime
'
```

### Шаг 2: Раздели broad roles
- поищи роли или ключи с административными правами, которые используются в рантайме;
- выдели из них минимум, нужный сервису.

```bash
rg -n "admin|owner|fullaccess|\*" infra/ terraform/ .github/ || true
```

### Шаг 3: Проверь секреты доступа
- найди static keys в CI и .env;
- зафиксируй, какие из них нужно заменить на короткоживущие механизмы.

```bash
rg -n "AWS_|SECRET_ACCESS_KEY|TOKEN|KEY=" .github/ . env* || true
```

### Что нужно явно показать
- список ролей и их назначение;
- какие права есть у runtime роли;
- какие секреты доступа еще живут статически;
- как логируются привилегированные действия.

---

## 2.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- на своем стенде сократи одну роль до реально нужных прав и проверь, что сервис продолжает работать;
- сделай controlled проверку: задача без лишних прав должна завершиться отказом;
- зафиксируй, что CI не использует широчайшие credentials без необходимости.

---

## 2.6 Типовые ошибки

- использовать одну admin-role везде;
- хранить static keys годами;
- не разделять runtime и deployment access;
- не проверять effective permissions на практике.
