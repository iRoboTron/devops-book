# Глава 1: Cloud perimeter и публичные сервисы

> **Запомни:** в облаке пограничная защита размазана по нескольким слоям: security groups, публичные IP, load balancer, WAF, reverse proxy и host firewall.

---

## 1.1 Контекст и границы

Облако дает удобство, но усложняет модель ответственности. Публичный IP может принадлежать виртуальной машине, балансировщику или NAT gateway, а security group легко принять за единственный источник правды.

Инженеру нужно уметь проследить путь входящего трафика и понять, где именно происходит фильтрация, TLS termination и журналирование.

Эта глава особенно полезна для VPS, облачных VM и небольших кластеров, где сервисы выставляются в интернет.

---

## 1.2 Как выглядит риск

Типовые слабые места:
- на VM висит публичный IP без необходимости;
- security group разрешает слишком широкие CIDR;
- health-check и admin endpoints доступны из интернета;
- cloud firewall и host firewall конфликтуют или дублируют хаотично;
- непонятно, кто завершает TLS: LB, proxy или приложение.

### Где особенно важно
- один VPS
- облачный reverse proxy
- публичный API
- приложение за load balancer

---

## 1.3 Что строит защитник

- минимум публичных IP;
- точные security groups и network ACL;
- разделение public and private subnets;
- reverse proxy или WAF перед приложением;
- host firewall как дополнительный слой.

### Практический результат главы
- ты умеешь описать путь интернет-трафика в облаке;
- знаешь, какие cloud controls должны дублироваться на хосте;
- можешь проверить, какие порты и endpoints реально доступны извне.

```text
ss -tulpn
sudo ufw status verbose
curl -I https://HOST
```

---

## 1.4 Практика

### Шаг 1: Собери карту public exposure
- перечисли публичные IP, DNS-имена, балансировщики и security groups;
- для каждого ресурса ответь, зачем он публичный.

```bash
ip a
ss -tulpn
```

### Шаг 2: Сравни cloud и host policy
- проверь, что открыто на уровне облака и что слушает VM;
- убедись, что внутренние сервисы не получают публичный вход из-за широких правил.

```bash
sudo ufw status verbose
```

### Шаг 3: Проверь surface извне
- со своей внешней машины проверь только собственные адреса и домены;
- зафиксируй, что admin и health endpoints не торчат наружу.

```bash
nmap -Pn SERVER_IP
curl -I https://HOST/health || true
```

### Что нужно явно показать
- какие компоненты имеют публичный адрес;
- где стоит reverse proxy или load balancer;
- какие правила действуют на уровне облака и хоста;
- как выглядит минимально необходимая экспозиция.

---

## 1.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- убери один лишний публичный порт и проверь, что пользовательский трафик не ломается;
- сравни результаты внешней проверки до и после tightening;
- зафиксируй, что админские endpoints закрыты от мира.

---

## 1.6 Типовые ошибки

- раздавать публичные IP всем VM по умолчанию;
- полагаться только на security group без host firewall;
- держать health, debug и admin endpoints на том же public surface;
- не знать, где завершается TLS.
