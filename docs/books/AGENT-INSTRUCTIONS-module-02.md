### Часть 1: Как работает веб (Главы 1-2)

#### Глава 1: HTTP/HTTPS изнутри

**Цель:** читатель перестаёт воспринимать веб как магию.

- Что происходит когда ты открываешь сайт (от нажатия Enter до страницы)
- HTTP-запрос: метод, путь, заголовки, тело
- HTTP-ответ: статус-код, заголовки, тело
- Коды статусов которые нужно знать: 200, 301, 302, 400, 403, 404, 500, 502
- Разница HTTP vs HTTPS: что шифруется, что нет, зачем нужен сертификат
- `curl` как инструмент исследования: `-v`, `-I`, `-H`, `-X`, `-d`
- `curl` для отладки: смотреть заголовки, редиректы, время ответа
- **Упражнения:** разобрать curl-запрос вручную, найти 404 и 302

> **Улучшение по сравнению с Модулем 1:** В Модуле 1 `curl` был инструментом "скачать файл". Здесь он становится **диагностическим прибором**. Объясни этот переход явно.

#### Глава 2: DNS — от домена до IP

**Цель:** читатель понимает как domain.ru превращается в 203.0.113.50.

- DNS как телефонная книга интернета
- Что происходит при DNS-запросе (шаги: браузер → /etc/hosts → резолвер → root → TLD → авторитативный)
- Типы записей: A, AAAA, CNAME, MX, TXT, NS — для чего каждый
- `dig` — диагностика DNS (не `nslookup`)
  - `dig domain.ru`
  - `dig domain.ru A`
  - `dig domain.ru MX`
  - `dig @8.8.8.8 domain.ru` — спросить конкретный резолвер
- `/etc/hosts` — локальный DNS (удобно для разработки)
- TTL — почему DNS-изменения применяются не сразу
- Где купить домен и как настроить A-запись (практически, без воды)
- **Упражнения:** разобрать DNS-цепочку для реального домена, настроить `/etc/hosts` для локальной разработки

### Часть 2: Nginx (Главы 3-4)

#### Глава 3: Nginx — первые шаги

**Цель:** читатель запускает Nginx и понимает как он работает.

- Что такое веб-сервер и зачем он нужен (Python-приложение не умеет само обрабатывать 10к соединений)
- Установка и запуск Nginx
- Структура конфигурации:
  - `/etc/nginx/nginx.conf` — главный файл, трогаем только `http {}`
  - `/etc/nginx/sites-available/` — конфиги сайтов
  - `/etc/nginx/sites-enabled/` — включённые сайты (symlinks)
- Минимальный конфиг для статического сайта: `server`, `listen`, `root`, `index`
- `nginx -t` — проверка конфига (запускать **всегда** перед reload)
- `systemctl reload nginx` vs `restart` — в чём разница и когда что
- Где смотреть логи: `access.log` и `error.log`
- **Упражнения:** поднять статический сайт, сломать конфиг и починить по логам

> **Опасно:** объясни что `nginx -t` — это not optional. Один неверный символ может уронить все сайты. Формируй привычку.

#### Глава 4: Nginx как обратный прокси

**Цель:** читатель соединяет Nginx с Python-приложением.

- Что такое reverse proxy и зачем он нужен (схема: клиент → Nginx → приложение)
- `proxy_pass` — базовый проброс
- Директива `location`: точное совпадение `=`, префикс `/api/`, регулярки `~`
- Важные заголовки: `proxy_set_header Host`, `X-Real-IP`, `X-Forwarded-For`, `X-Forwarded-Proto`
  - Почему без них приложение не видит реальный IP клиента
- `upstream` — группа бэкендов (базово, для понимания)
- Таймауты: `proxy_connect_timeout`, `proxy_read_timeout` — что делать при 504
- `client_max_body_size` — загрузка файлов
- Логи: как добавить в лог upstream-время для отладки производительности
- **Полный пример конфига** для Python-приложения на порту 8000
- **Упражнения:** поднять FastAPI/Flask, подключить через Nginx, проверить что IP передаётся корректно

### Часть 3: Безопасность (Главы 5-6)

#### Глава 5: HTTPS с Let's Encrypt

**Цель:** читатель добавляет HTTPS без покупки сертификата.

- Что такое TLS/SSL и зачем (не "безопасность", а конкретно: шифрование данных + подтверждение личности сервера)
- Как работает Let's Encrypt: ACME-протокол, challenge, CA
- `certbot` — установка и получение сертификата
  - `certbot --nginx -d domain.ru` — автоматический режим
  - `certbot certonly` — только сертификат, без правки Nginx
- Что certbot меняет в конфиге Nginx (показать diff до/после)
- Автопродление: `systemctl status certbot.timer`, как проверить
- `certbot renew --dry-run` — тест без реального продления
- Редирект HTTP → HTTPS: как работает, как настроить вручную
- Что делать если certbot не может получить сертификат (порт 80 закрыт, DNS не настроен)
- **Для разработки без домена:** self-signed сертификат через `openssl` — быстро и без регистрации
- **Упражнения:** получить сертификат, проверить через браузер, симулировать продление

> **Запомни:** Let's Encrypt требует открытый порт 80 во время получения сертификата. Закроешь его в ufw — certbot не сработает.

#### Глава 6: Фаервол с ufw

**Цель:** читатель закрывает сервер от внешнего мира, оставив только нужное.

- Зачем нужен фаервол (реальный сценарий: PostgreSQL открыт наружу = катастрофа)
- `ufw` как удобная обёртка над `iptables`
- Базовые команды:
  - `ufw status verbose`
  - `ufw default deny incoming`
  - `ufw default allow outgoing`
  - `ufw allow 22` / `ufw allow ssh`
  - `ufw allow 80` / `ufw allow 443`
  - `ufw allow from 10.0.0.0/8 to any port 5432` — открыть только для своей сети
  - `ufw delete` — удалить правило
  - `ufw enable` / `ufw disable`
- Порядок правил — почему важен
- Профили приложений: `ufw allow 'Nginx Full'`
- Логирование: `ufw logging on`

> **Опасно:** Не запускай `ufw enable` пока не добавил `ufw allow 22`. Закроешь себе SSH — придётся идти к серверу физически (или через VNC).

- **Упражнения:** настроить фаервол для типичного веб-сервера, проверить что лишние порты закрыты, симулировать попытку подключения к закрытому порту

### Часть 4: Диагностика (Глава 7)

#### Глава 7: Диагностика сетевых проблем

**Цель:** читатель умеет самостоятельно найти причину когда "не работает".

- Алгоритм диагностики: от простого к сложному
  ```
  1. Сервер вообще доступен? → ping
  2. Порт открыт? → nc / ss / ufw status
  3. Сервис запущен? → systemctl status
  4. Конфиг правильный? → nginx -t
  5. Что отвечает? → curl -v
  6. Что в логах? → journalctl / nginx error.log
  ```
- `tcpdump` — смотреть пакеты (минимум: увидеть SYN, RST, ESTABLISHED)
- `ss -tlnp` — кто слушает какой порт
- `nc -zv host port` — проверка открытости порта
- Типичные ошибки и их причины:
  - `502 Bad Gateway` — приложение не запущено или неправильный `proxy_pass`
  - `504 Gateway Timeout` — приложение зависло или таймаут слишком короткий
  - `ERR_SSL_PROTOCOL_ERROR` — слушаешь HTTP на порту 443
  - `connection refused` — сервис не запущен
  - `connection timeout` — фаервол блокирует
  - `no route to host` — сетевая проблема или неправильный IP
- **Упражнения:** диагностировать намеренно сломанный стек (5 сценариев)

### Часть 5: Caddy — твой реальный сервер (Глава 8)

#### Глава 8: Caddy — когда Nginx делает слишком много

**Цель:** читатель, понимая Nginx, легко разбирается в Caddy и умеет с ним работать.

> **Важно для агента:** Эта глава строится на контрасте. Читатель только что вручную настроил всё что Caddy делает автоматически — это лучший момент чтобы показать Caddy. Не раньше.

- Почему Caddy существует: боль от ручной настройки SSL, редиректов и заголовков
- Что Caddy делает автоматически (и почему ты теперь это понимаешь):
  ```
  Nginx (вручную):                    Caddy (автоматически):
  ─────────────────────────────────   ──────────────────────────────
  certbot --nginx -d domain.ru        (получает сертификат сам)
  server { listen 80; return 301 }    (редирект HTTP→HTTPS сам)
  add_header Strict-Transport...      (заголовки безопасности сам)
  ssl_certificate /etc/letsencrypt/   (обновляет сертификат сам)
  ```
- **Caddyfile — синтаксис:**
  ```caddy
  domain.ru {
      reverse_proxy localhost:8000
  }
  ```
  Это всё. SSL, HTTPS, редирект — уже работают.
- Разбор реального Caddyfile по строкам (на примере `nc.npcbasto.ru`):
  ```caddy
  {
      servers {
          protocols h1 h2 h3        # HTTP/1.1, HTTP/2, HTTP/3
          trusted_proxies static 192.168.13.1   # доверять заголовкам от роутера
      }
  }

  domain.ru {
      reverse_proxy 127.0.0.1:8000  # проброс на приложение
  }

  static.domain.ru {
      root * /srv/static
      file_server                   # раздача статики
  }
  ```
- Запуск Caddy в Docker (как на реальном сервере):
  ```yaml
  services:
    caddy:
      image: caddy:2.10-alpine
      restart: always
      network_mode: host
      volumes:
        - ./Caddyfile:/etc/caddy/Caddyfile:ro
        - ./data:/data       # сертификаты хранятся здесь
        - ./config:/config
  ```
- Управление Caddy:
  - `docker compose exec caddy caddy reload --config /etc/caddy/Caddyfile` — применить изменения без рестарта
  - `docker compose logs caddy` — логи (включая получение сертификата)
  - `docker compose exec caddy caddy validate --config /etc/caddy/Caddyfile` — проверка конфига (аналог `nginx -t`)
- Добавить новый сайт/сервис: просто добавить блок в Caddyfile и сделать reload
- Где хранятся сертификаты: `./data/caddy/certificates/`
- `network_mode: host` vs проброс портов — почему на этом сервере используется host
- Диагностика Caddy:
  - Сертификат не получается → DNS не настроен или порт 80 закрыт (то же что certbot)
  - `caddy reload` упал → ошибка синтаксиса в Caddyfile (читать логи)
  - 502 → приложение не запущено или неправильный порт в `reverse_proxy`

- **Сравнительная таблица Nginx vs Caddy:**

  | | Nginx | Caddy |
  |---|---|---|
  | SSL | certbot отдельно | автоматически |
  | Конфиг | 30+ строк | 3-5 строк |
  | Синтаксис | директивы + блоки | domain { } |
  | Проверка | `nginx -t` | `caddy validate` |
  | Применить | `systemctl reload` | `caddy reload` |
  | Логи | `/var/log/nginx/` | `docker logs` / journalctl |
  | Когда выбрать | сложная логика, тонкая настройка | типичный reverse proxy, быстрый старт |

- **Упражнения:**
  1. Поднять Caddy в Docker, добавить reverse_proxy на локальное приложение
  2. Добавить второй домен в Caddyfile, сделать reload
  3. Намеренно сломать Caddyfile, прочитать ошибку в логах, починить

> **Запомни:** Caddy не магия — это Nginx-знания, упакованные в умолчания. Ты понимаешь что происходит под капотом, потому что сделал это руками в предыдущих главах.

## Принципы написания

Все принципы из Модуля 1 плюс специфичные для этой книги:

### 1. Одна команда — одно объяснение (как в Модуле 1)
Каждую команду разбирай:
- **Что делает** (простыми словами)
- **Синтаксис** (шаблон)
- **Пример** (из реальной DevOps-практики)
- **Что может пойти не так**

### 2. Контекст важнее синтаксиса (как в Модуле 1)
Не просто `proxy_pass http://127.0.0.1:8000`, а:
> "Твоё Python-приложение запущено на порту 8000, но снаружи его не видно — оно слушает только локально.
> Nginx выступает посредником: принимает запрос из интернета и пересылает его приложению.
> `proxy_pass` говорит Nginx: 'всё что пришло ко мне — отдай вот туда'."

### 3. Показывай схемы в ASCII
Сеть — это отношения между компонентами. Текст не передаёт это так, как схема.
После каждой ключевой концепции добавляй ASCII-диаграмму потока данных:
```
Браузер → [DNS] → IP → [ufw] → [Nginx :443] → [Python :8000]
```

### 4. Безопасный порядок действий
Есть операции, которые могут отрезать тебя от сервера:
- Сначала добавь правило `ufw allow 22` — **потом** `ufw enable`
- Сначала `nginx -t` — **потом** `systemctl reload nginx`
- Сначала проверь certbot на dry-run — **потом** запускай на реальном домене

Формируй эти привычки явно. Каждый раз, когда порядок важен — пиши об этом.

### 5. Для разработки без реального домена
Не у всех есть домен и публичный IP. Показывай как практиковаться локально:
- `/etc/hosts` для имитации домена
- `openssl` для self-signed сертификата
- VirtualBox NAT с проброшенными портами

### 6. Визуальные элементы (как в Модуле 1)
```
> **Совет:** лучшие практики
> **Опасно:** можно всё сломать или потерять доступ
> **Запомни:** ключевые концепции
> **Практика:** упражнения
> **Порядок важен:** шаги нельзя менять местами
```

Новый элемент `> **Порядок важен:**` — специфичен для этой книги, где неправильная последовательность может отрезать от сервера.

### 7. Caddy — только в главе 8
Не упоминать Caddy раньше главы 8. Когда читатель настроил всё вручную через Nginx — именно тогда Caddy производит нужный эффект: "о, так это можно было сделать в 3 строки?"

### 8. Никакой воды
- Без истории "как развивался HTTP"
- Без теории TLS 1.2 vs 1.3
- Без сравнения Nginx с Apache и HAProxy
- Без Docker, CI/CD, Kubernetes — это другие книги

### Форматирование
- Markdown
- `#` — название главы, `##` — раздел, `###` — подраздел
- Код в блоках с языком: ` ```bash `, ` ```nginx `, ` ```text `
- ASCII-схемы в блоках ` ```text ` или ` ```
- Таблицы для сравнений и справочной информации

### Объём
- Каждая глава: 15-20 страниц
- Примеры: минимум 3 на команду
- Упражнения: 3-5 на главу

## Проверка качества

Перед сдачей каждой главы проверь:

1. **Понятность:** поймёт ли человек который никогда не настраивал Nginx?
2. **Практичность:** можно ли сразу применить на сервере?
3. **Безопасность:** есть ли предупреждения про опасные операции (ufw enable, потеря SSH)?
4. **Схемы:** есть ли ASCII-диаграмма потока данных?
5. **Упражнения:** есть ли после главы? Реальные задачи?
6. **Без воды:** нет ли теории ради теории?
7. **Порядок:** если порядок шагов важен — выделено ли это явно?

## Что НЕ надо делать

- ❌ Не объяснять Docker, CI/CD, Kubernetes — это следующие книги
- ❌ Не упоминать Caddy раньше главы 8 — эффект "вот зачем ты всё это делал вручную" работает только в конце
- ❌ Не сравнивать Nginx с Apache, HAProxy, Traefik — только Nginx и Caddy
- ❌ Не углубляться в TLS-математику (достаточно: "шифрует данные, подтверждает личность")
- ❌ Не предполагать что читатель знает Nginx или Caddy
- ❌ Не давать конфиги без объяснения каждой строки
- ❌ Не пропускать `nginx -t` / `caddy validate` перед reload в примерах — это формирует привычку
- ❌ Не писать команды без контекста — зачем, когда, что будет если не сделать
- ❌ Не предполагать что у читателя есть реальный домен (показывай локальную практику)

## Связь с другими модулями

**Что нужно из Модуля 1:**
- `systemctl` — для управления Nginx и приложением
- `journalctl` — для отладки
- Shell-скрипты — для мониторинг-скрипта в итоговом проекте
- SSH + пользователи — для безопасной работы с сервером

**Что даёт Модулю 3 (Docker):**
- Понимание портов и proxy_pass / reverse_proxy — Docker тоже работает через порты
- Nginx и Caddy как reverse proxy — в Docker Compose будет то же самое
- ufw — фаервол останется тем же, Docker просто добавит свои сети
- Caddy в Docker (`network_mode: host`) — уже знакомая схема из главы 8

**Что даёт Модулю 4 (CI/CD):**
- Умение настраивать веб-стек — деплой будет на этот стек
- Понимание HTTPS и домена — GitHub Actions будет деплоить на реальный сервер
- `caddy reload` в скрипте деплоя — применять новый Caddyfile без даунтайма
