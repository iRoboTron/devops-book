# Глава 2: NAT, port forwarding и DMZ

> **Запомни:** публикация сервиса в интернет почти всегда проходит через NAT, и именно здесь удобно случайно открыть лишнее.

---

## 2.1 Контекст и границы

NAT скрывает внутренние адреса, но сам по себе не является политикой безопасности. Как только появляется port forwarding, вопрос уже не в переводе адреса, а в том, какие сервисы ты публикуешь и зачем.

DMZ в бытовом роутере и DMZ как сетевая зона — это разные вещи. Нужно описывать потоки трафика, а не пользоваться названием как ярлыком.

Эта глава особенно полезна для дома, VPS с NAT/forwarding у провайдера и малых сетей с одним пограничным устройством.

---

## 2.2 Как выглядит риск

Типовые слабые места:
- проброшены старые или забытые порты;
- используется полный DMZ на один хост;
- админка торчит на том же IP и портах, что и пользовательский трафик;
- непонятно, какие сервисы должны быть доступны только изнутри;
- hairpin NAT и внутренний DNS настроены хаотично.

### Где особенно важно
- домашний сервер
- малый офис
- лаборатория с несколькими VM

---

## 2.3 Что строит защитник

- порт-форварды только по списку сервисов;
- внутренние базы, кэши и панели не публикуются;
- реальная DMZ как отдельная подсеть или отдельный сегмент;
- админ-доступ уводится в VPN или management network;
- описание входящих и исходящих потоков.

### Практический результат главы
- ты различаешь бытовой DMZ toggle и архитектурную DMZ;
- умеешь минимизировать port forwarding;
- можешь описать, какие сервисы должны быть публичными, а какие нет.

```text
table ip nat {
  chain prerouting {
    type nat hook prerouting priority dstnat;
    tcp dport 443 dnat to 192.168.10.20:443
  }
}
```

---

## 2.4 Практика

### Шаг 1: Инвентаризация port forwarding
- запиши все проброшенные порты и их целевые адреса;
- для каждого порта ответь, это действительно публичный сервис или просто привычка.

```bash
sudo nft list ruleset
```

### Шаг 2: Раздели публичное и внутреннее
- перенеси БД, admin panels и SSH с публичного входа в VPN или management path;
- если сервис один, оставь снаружи только reverse proxy.

```bash
ss -tulpn
```

### Шаг 3: Проверь внешний вид для клиента
- после cleanup просканируй внешний IP и сравни с инвентаризацией;
- проверь, что внутренние сервисы доступны только из нужной подсети.

```bash
nmap -Pn EXTERNAL_IP
nmap -Pn 192.168.10.20
```

### Что нужно явно показать
- таблицу всех проброшенных портов;
- какие из них относятся к публичному трафику;
- где живут внутренние сервисы;
- какой механизм используется для админ-доступа.

---

## 2.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- убери один лишний форвард и проверь, что сервис продолжает работать по задуманной схеме;
- сравни внешний и внутренний scan-результат;
- зафиксируй, что публикация через DMZ toggle не используется без осознанной причины.

---

## 2.6 Типовые ошибки

- держать весь хост в DMZ ради удобства;
- открывать SSH и БД одновременно наружу;
- не вести инвентаризацию проброшенных портов;
- строить политику безопасности только на NAT.
