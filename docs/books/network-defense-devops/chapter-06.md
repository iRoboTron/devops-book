# Глава 6: DNS и сетевые злоупотребления

> **Запомни:** DNS выглядит как скучный сервис, пока через него не начинают происходить утечки, redirection, amplification и обходы политик.

---

## 6.1 Контекст и границы

DNS — это и контрольная плоскость, и источник телеметрии. Через него удобно замечать нетипичные исходящие запросы, ошибки конфигурации и попытки использовать инфраструктуру не по назначению.

Для небольших систем полезно как минимум понимать рекурсивный и авторитативный DNS, split-horizon, кэширование и какие записи реально публикуются наружу.

Эта глава особенно полезна для всех инфраструктур, где есть собственный resolver, публичные записи и критичная зависимость от DNS.

---

## 6.2 Как выглядит риск

Типовые слабые места:
- открытая рекурсия;
- лишние публичные записи и утечка внутренней структуры;
- устаревшие записи после миграции сервиса;
- нет журналов DNS-запросов;
- серверы и клиенты используют разные, неуправляемые резолверы.

### Где особенно важно
- домашняя лаборатория
- офисный resolver
- публичный домен приложения
- cloud DNS

---

## 6.3 Что строит защитник

- разделение авторитативной и рекурсивной роли;
- инвентаризация DNS-записей и TTL;
- контроль того, какие резолверы используют серверы и контейнеры;
- ограничение рекурсии и source networks;
- журналирование и review нетипичных доменных запросов.

### Практический результат главы
- ты понимаешь, какие DNS-компоненты есть у твоей инфраструктуры;
- умеешь проверять опубликованные записи и резолверы на хостах;
- можешь объяснить, почему open resolver — плохая идея.

```text
dig +short A example.com
dig +trace example.com
resolvectl status
```

---

## 6.4 Практика

### Шаг 1: Собери карту DNS-зависимостей
- какие домены и поддомены у тебя есть;
- кто их обслуживает: cloud DNS, registrar, локальный resolver.

```bash
dig +short A HOST
dig +short MX HOST
```

### Шаг 2: Проверь серверные резолверы
- на серверах и контейнерах посмотри, к каким DNS они обращаются;
- убедись, что это управляемые и ожидаемые резолверы.

```bash
resolvectl status
cat /etc/resolv.conf
```

### Шаг 3: Проверь лишнюю экспозицию
- сними список записей, которые реально нужны наружу;
- удали устаревшие A/CNAME/TXT записи, особенно служебные.

```bash
dig +short TXT HOST
```

### Что нужно явно показать
- какие публичные DNS-записи обслуживают сервис;
- какой resolver используют серверы;
- есть ли split-horizon или внутренние зоны;
- как собираются логи DNS-запросов.

---

## 6.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- проверь, что твой resolver не является открытой рекурсией для мира;
- сравни запросы к внутренним и внешним зонам;
- зафиксируй неожиданные обращения к внешним доменам из своей lab.

---

## 6.6 Типовые ошибки

- публиковать внутренние хостнеймы в публичной зоне;
- забывать старые записи после миграций;
- не контролировать, какой DNS используют контейнеры и VM;
- смешивать авторитативный и рекурсивный сервис без необходимости.
