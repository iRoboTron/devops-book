# Глава 4: VPN и безопасный административный доступ

> **Запомни:** административный доступ стоит строить как отдельный доверенный канал, а не как открытый SSH-порт “потому что удобно”.

---

## 4.1 Контекст и границы

Почти любой маленький проект начинает с SSH из интернета. Но чем дольше система живет, тем заметнее становится, что управление и пользовательский трафик должны быть разделены.

VPN дает контролируемую management surface: отдельная подсеть, предсказуемые адреса, меньше лишней экспозиции и проще журналирование.

Эта глава особенно полезна для всех, кто администрирует серверы из дома, офиса или через интернет.

---

## 4.2 Как выглядит риск

Типовые слабые места:
- SSH открыт всему интернету;
- одинаковые ключи и доступы используются на всех серверах;
- root login и password auth включены;
- нет журнала, кто и когда получал доступ к management plane;
- внутренние панели доступны по публичному адресу.

### Где особенно важно
- домашний сервер
- VPS
- небольшой офисный bastion

---

## 4.3 Что строит защитник

- VPN для админ-доступа;
- SSH только по ключам, PermitRootLogin no, PasswordAuthentication no;
- отдельные аккаунты и sudo;
- ограничение AllowUsers/AllowGroups и IP allowlist там, где возможно;
- bastion или jump host при росте числа серверов.

### Практический результат главы
- ты умеешь минимизировать exposed admin surface;
- знаешь, какие настройки SSH обязательны в baseline;
- можешь объяснить, когда нужен bastion, а когда достаточно VPN.

```text
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers adminuser

sudo ufw allow from 10.10.0.0/24 to any port 22 proto tcp
```

---

## 4.4 Практика

### Шаг 1: Ужесточи SSH baseline
- открой sshd_config и убери root login и password auth;
- оставь доступ только нужным пользователям.

```bash
sudo nano /etc/ssh/sshd_config
sudo sshd -t
sudo systemctl reload ssh
```

### Шаг 2: Спрячь админ-порты за VPN или allowlist
- если есть VPN, ограничь SSH только management subnet;
- если VPN пока нет, хотя бы сузь доступ по IP.

```bash
sudo ufw status numbered
```

### Шаг 3: Проверь access logs
- посмотри, какие попытки входа реально прилетают на сервер;
- зафиксируй baseline по успешным и неуспешным входам.

```bash
sudo journalctl -u ssh -n 100 --no-pager
```

### Что нужно явно показать
- свой sshd_config baseline;
- какой IP или подсеть может ходить к SSH;
- есть ли отдельный VPN или bastion;
- как выглядит журнал входов.

---

## 4.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- проверь со своей машины, что SSH доступен только по management path;
- убедись, что попытка пароля не проходит, если password auth отключен;
- зафиксируй, что root login запрещен.

---

## 4.6 Типовые ошибки

- оставлять парольный SSH в интернет;
- использовать один shared key на всю команду;
- администрировать production от root;
- не тестировать sshd -t перед reload.
