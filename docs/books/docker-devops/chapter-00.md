# Глава 0: Зачем Docker и чем он отличается от виртуалки

> **Запомни:** Контейнер — это не мини-виртуалка. Это изолированный процесс на хосте с собственной файловой системой. Пойми это сейчас — не будет проблем потом.

---

## 0.1 Проблема: "у меня работает, у тебя не работает"

Знакомая ситуация?

```
Разработчик: "У меня на ноутбуке всё работает!"
DevOps:    "А на сервере — нет"

Разработчик: "Ну... у меня Python 3.12"
DevOps:    "А на сервере 3.10"

Разработчик: "А, у меня ещё flask 3.0 стоит"
DevOps:    "А тут 2.4"

Разработчик: "Ну и поставь такую же версию"
DevOps:    "А тут ещё одно приложение которое требует 2.4"

Разработчик: "..."
DevOps:    "..."
```

**Проблема:** приложение зависит от:
- Версии Python
- Версий библиотек (flask, sqlalchemy, ...)
- Системных пакетов (libpq, openssl, ...)
- Переменных окружения
- Конфигурационных файлов

На твоём ноутбуке — одна комбинация.
На сервере — другая.

### Старое решение: виртуалка

Можно поднять точно такую же Ubuntu как на сервере:
- Виртуальная машина
- Тот же Python, те же библиотеки
- Те же конфиги

**Проблема:** виртуалка тяжёлая:
- 2-4 ГБ RAM каждая
- Минуты на запуск
- Гигабайты на диске
- Сложно обновлять

### Решение Docker

**Контейнер** — та же идея, но легче:

```
Не нужно: целую операционную систему для каждого приложения
Нужно:    изолированный процесс с нужными файлами
```

Docker даёт:
- ✅ Те же зависимости что у разработчика
- ✅ Запуск за секунды
- ✅ МБ на диске (не ГБ)
- ✅ Одинаковое поведение везде

> **Запомни:** Docker — не про "запустить Linux в Linux".
> Docker — про "запустить приложение с нужными зависимостями, одинаково везде".

## 0.2 Виртуальная машина vs контейнер

### Виртуальная машина

```
┌─────────────────────────────────┐
│  Приложение  │  Приложение     │
├──────────────┼─────────────────┤
│  Guest OS    │  Guest OS       │  ← Полная ОС для каждого!
│  (2 ГБ RAM)  │  (2 ГБ RAM)     │
├──────────────┴─────────────────┤
│  Hypervisor (VirtualBox)       │
├────────────────────────────────┤
│  Host OS (твоя Ubuntu)         │
├────────────────────────────────┤
│  Железо                        │
└────────────────────────────────┘
```

Каждая ВМ = полная копия операционной системы.

### Контейнер (Docker)

```
┌─────────────────────────────────┐
│  Приложение  │  Приложение     │
├──────────────┼─────────────────┤
│  Контейнер   │  Контейнер      │  ← Изолированные процессы
├──────────────┴─────────────────┤
│  Container runtime (Docker)    │  ← Одна прослойка для всех
├────────────────────────────────┤
│  Host OS (твоя Ubuntu)         │  ← Одна ОС на всех
├────────────────────────────────┤
│  Железо                        │
└────────────────────────────────┘
```

Контейнеры делят ядро хоста. Нет дублирования ОС.

### Сравнение

| Параметр | Виртуалка | Контейнер |
|----------|-----------|-----------|
| RAM | 2-4 ГБ каждая | 50-200 МБ каждый |
| Запуск | 1-5 минут | 1-5 секунд |
| Размер | 5-20 ГБ | 50-500 МБ |
| Изоляция | Полная (своя ОС) | Процессная (общее ядро) |
| Безопасность | Выше | Ниже (но достаточно для DevOps) |
| Когда использовать | Разные ОС, строгая безопасность | Один хост, много сервисов |

> **Запомни:** Контейнеры не заменяют виртуалки полностью.
> Но для 95% задач DevOps — контейнеры то что нужно.

## 0.3 Ключевые понятия

### Образ (Image)

**Образ** — готовая "упаковка" приложения.

Как `.iso` файл для виртуалки. Только легче.

```
Образ python-app:1.0
├── Базовая система (python:3.12-slim)
├── Твои библиотеки (flask, sqlalchemy)
├── Твой код (app.py)
└── Конфигурация (переменные, порты)
```

Образ **не меняется** когда запущен. Он read-only.

### Контейнер (Container)

**Контейнер** — запущенный экземпляр образа.

Как запущенная виртуалка. Только легче.

```
Образ: python-app:1.0  (рецепт, неизменный)
    ↓ запущен
Контейнер: myapp-1     (живой процесс)
Контейнер: myapp-2     (ещё одна копия)
```

Из одного образа можно запустить много контейнеров.

### Dockerfile

**Dockerfile** — рецепт сборки образа.

```dockerfile
FROM python:3.12-slim     ← базовый образ
WORKDIR /app              ← рабочая папка
COPY requirements.txt .   ← зависимости
RUN pip install ...       ← установка
COPY . .                  ← твой код
CMD ["python", "app.py"]  ← команда запуска
```

Dockerfile → `docker build` → Образ → `docker run` → Контейнер

### Registry (Реестр)

**Registry** — хранилище образов.

Docker Hub — публичный реестр (hub.docker.com).
Там лежат официальные образы: `python`, `postgres`, `nginx`.

```
Docker Hub:  python:3.12-slim  ← скачал
                           ↓
Твой ноут: docker build → свой образ  ← собрал
                           ↓
Docker Hub: твой-username/myapp:1.0   ← загрузил
                           ↓
Сервер:    docker pull → твой-username/myapp:1.0  ← скачал
```

### Volume (Том)

**Volume** — хранилище данных которое переживает контейнер.

Контейнер удалил → данные в volume остались.

```
Контейнер PostgreSQL
    ↓ пишет данные
Volume: pgdata  ← данные здесь
    ↓
Контейнер удалил и создал новый
    ↓ читает данные
Volume: pgdata  ← данные на месте
```

### Network (Сеть)

**Network** — изолированная сеть для контейнеров.

Контейнеры в одной сети видят друг друга **по имени**.

```
Сеть: myapp-net
├── app  (доступен как http://app:8000)
├── db   (доступен как http://db:5432)
└── redis (доступен как http://redis:6379)
```

## 0.4 Архитектура Docker

```
Ты (терминал)
    │
    │ docker run, docker build, ...
    ▼
┌──────────────────────────────┐
│  Docker CLI                  │  ← То что ты набираешь
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│  Docker Daemon (dockerd)     │  ← Фоновый процесс
│                              │
│  Управляет:                  │
│  - Контейнерами              │
│  - Образами                  │
│  - Сетями                    │
│  - Томами                    │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│  Container runtime           │  ← Запускает контейнеры
│  (containerd + runc)         │
└──────────────┬───────────────┘
               │
               ▼
         Ядро Linux
```

> **Запомни:** Docker CLI — просто клиент.
> Реальную работу делает Docker Daemon (dockerd).
> Ты посылаешь команды — демон выполняет.

## 0.5 Установка Docker на Ubuntu

> **Важно:** Не используй `apt install docker.io` — это старая версия из репозитория Ubuntu.
> Используй официальный репозиторий Docker.

### Шаг 1: Удали старые версии

```bash
sudo apt remove -y docker docker-engine docker.io containerd runc
```

### Шаг 2: Установи зависимости

```bash
sudo apt update
sudo apt install -y ca-certificates curl gnupg
```

### Шаг 3: Добавь ключ репозитория Docker

```bash
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg
```

### Шаг 4: Добавь репозиторий

```bash
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
```

### Шаг 5: Установи Docker

```bash
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

### Шаг 6: Запусти Docker

```bash
sudo systemctl enable --now docker
```

### Шаг 7: Проверь

```bash
docker --version
Docker version 27.0.0, build abcdef0

sudo docker run hello-world
```

Если увидел сообщение "Hello from Docker!" — работает!

## 0.6 Добавь себя в группу docker

По умолчанию Docker требует `sudo`:

```bash
docker ps
# permission denied

sudo docker ps
# работает
```

Чтобы не писать `sudo` каждый раз:

```bash
sudo usermod -aG docker $USER
```

**Перелогинься** (выйди и зайди снова) чтобы изменение применилось.

Проверь:
```bash
docker ps
# работает без sudo!
```

> **Запомни:** Группа `docker` = root-эквивалент.
> Кто в группе docker — может делать что угодно с системой.
> Добавляй только доверенных пользователей.
