## Контекст проекта

Ученик прошёл все предыдущие модули. У него есть работающий сервер:
- Python-приложение в Docker за Caddy/Nginx
- PostgreSQL с бэкапами по расписанию
- CI/CD пайплайн — push в main деплоит автоматически

**Чего пока нет:**
Сервер работает, но он уязвим и нем. Можно зайти с любым паролем сколько угодно раз — никто не заблокирует. Пакеты не обновлялись с установки. Когда что-то падает — узнаёт через час, когда пользователи пишут. Диск переполнился — узнал когда приложение упало.

**Что он хочет после этой книги:**
Сервер который сам сигнализирует о проблемах до того как они стали инцидентами. Злоумышленник не сможет зайти брутфорсом. Уязвимости закрываются до того как их начнут эксплуатировать. Он знает о состоянии сервера не открывая терминал.

## Главная идея книги

Два принципа:

**Безопасность — это слои.** Нет одной настройки которая защищает всё. Каждый слой усложняет жизнь злоумышленнику: SSH-ключи, fail2ban, закрытые порты, обновления, минимальные права.

**Мониторинг — это уши и глаза.** Ты не можешь смотреть в терминал 24/7. Нужна система которая смотрит за тебя и говорит "здесь проблема" раньше чем она стала катастрофой.

```
Без мониторинга:          С мониторингом:
────────────────          ──────────────────────────────
Диск переполнился         Алерт: "Диск 85%, осталось 15GB"
Приложение упало          Алерт: "Healthcheck провалился"
CPU 100% час              Алерт: "CPU > 90% уже 10 минут"
Взлом обнаружен           Алерт: "100 неудачных SSH-попыток"
  через неделю              → fail2ban заблокировал IP
```

## Структура книги

### Глава 0: Модель угроз — от чего защищаемся

**Цель:** читатель понимает реальные угрозы, а не абстрактную "безопасность".

- Кто реально атакует маленькие серверы:
  - Боты-сканеры (пробуют стандартные пароли на SSH, 24/7)
  - Криптомайнеры (ищут слабые места, устанавливают майнер)
  - Шантажисты (шифруют данные если нет бэкапов)
- Что они ищут конкретно:
  - SSH с паролем `root:root` или `admin:admin`
  - Открытые порты баз данных (5432, 3306, 27017)
  - Устаревшие версии nginx/apache с известными уязвимостями
- Чего обычно НЕ бывает на маленьких серверах:
  - Целенаправленных атак с большим ресурсом
  - APT-группировок (если ты не крупная компания)
- Модель защиты для этого курса:
  ```
  1. Закрыть очевидные дыры (SSH с паролем)
  2. Замедлить автоматические атаки (fail2ban)
  3. Сократить поверхность атаки (минимум открытых портов)
  4. Узнавать о попытках (логи + алерты)
  5. Быстро восстанавливаться (бэкапы — Модуль 5)
  ```
- **Нет упражнений** — обзорная глава

> **Запомни:** Цель не "стать неуязвимым" (это невозможно), а "стать неинтересной целью". Боты ищут лёгкое — дай им отпор, пойдут дальше.

### Часть 1: Безопасность (Главы 1–4)

#### Глава 1: SSH — только ключи, никаких паролей

**Цель:** закрыть самый распространённый вектор атаки.

> **Важно:** Читатель знает SSH-ключи из Модуля 1. Здесь — hardening SSH-конфига на сервере, а не введение в тему.

- Почему пароли по SSH — плохая идея:
  - Любой бот пробует `root:123456` и тысячи вариантов 24/7
  - SSH-ключ в 2048 бит = практически невозможно перебрать
- Правка `/etc/ssh/sshd_config` — ключевые параметры:
  ```
  PermitRootLogin no           # запретить вход под root
  PasswordAuthentication no    # только ключи
  PubkeyAuthentication yes     # ключи разрешены
  Port 22                      # (опционально) сменить порт
  MaxAuthTries 3               # максимум 3 попытки
  LoginGraceTime 20            # 20 секунд на аутентификацию
  AllowUsers deploy            # только конкретные пользователи
  ```
- **Порядок важен:** сначала убедись что ключ работает, потом отключай пароли
  ```
  1. Добавить ключ в authorized_keys
  2. Проверить вход по ключу (в новой сессии!)
  3. Отключить PasswordAuthentication
  4. Перезапустить sshd
  ```
- `sshd -t` — проверка конфига (аналог `nginx -t`)
- `systemctl reload sshd` — применить без обрыва текущих соединений
- SSH-конфиг на клиенте `~/.ssh/config` — алиасы для удобства
- Two-factor: `libpam-google-authenticator` — упомянуть, не углубляться

> **Опасно:** Отключи пароли только ПОСЛЕ того как убедился что вход по ключу работает — в другой терминальной сессии. Иначе закроешь себя снаружи.

- **Упражнения:** настроить sshd_config, проверить что вход по паролю заблокирован, создать алиас в ~/.ssh/config

#### Глава 2: fail2ban — автоматическая блокировка атак

**Цель:** автоматически блокировать IP после нескольких неудачных попыток.

- Что такое fail2ban: читает логи, ищет паттерны, добавляет правила iptables
- Как работает:
  ```
  Лог: sshd: Failed password for root from 1.2.3.4
  Лог: sshd: Failed password for root from 1.2.3.4
  Лог: sshd: Failed password for root from 1.2.3.4
  fail2ban: ban 1.2.3.4 на 1 час
  iptables: DROP все пакеты от 1.2.3.4
  ```
- Установка и базовая конфигурация
- Файл `/etc/fail2ban/jail.local` (не трогать `jail.conf`):
  ```ini
  [DEFAULT]
  bantime  = 1h
  findtime = 10m
  maxretry = 5

  [sshd]
  enabled = true
  port    = ssh
  logpath = %(sshd_log)s

  [nginx-http-auth]
  enabled = true

  [nginx-req-limit]
  enabled  = true
  filter   = nginx-req-limit
  logpath  = /var/log/nginx/error.log
  maxretry = 10
  ```
- Управление:
  - `fail2ban-client status` — общий статус
  - `fail2ban-client status sshd` — статус jail
  - `fail2ban-client set sshd unbanip 1.2.3.4` — разбанить IP
  - `fail2ban-client banned` — список заблокированных
- Логи: `journalctl -u fail2ban -f`
- Защита Caddy/Nginx от брутфорса веб-форм — кастомный фильтр
- Свой IP в whitelist: `ignoreip = 127.0.0.1/8 ::1 твой.ip`
- **Упражнения:** настроить fail2ban, симулировать 5 неудачных SSH-попыток с другого IP, убедиться что IP заблокирован, разбанить

> **Опасно:** Добавь свой IP в `ignoreip` до тестирования. Иначе заблокируешь сам себя.

#### Глава 3: Обновления и патчи

**Цель:** уязвимости закрываются автоматически, без ручного `apt upgrade` каждую неделю.

- Почему обновления критичны: большинство взломов — через известные уязвимости в устаревшем ПО
- `apt update && apt upgrade` — вручную, знает из Модуля 1
- `unattended-upgrades` — автоматические обновления безопасности:
  ```bash
  sudo apt install unattended-upgrades
  sudo dpkg-reconfigure --priority=low unattended-upgrades
  ```
- Конфиг `/etc/apt/apt.conf.d/50unattended-upgrades`:
  - Какие пакеты обновлять (только security vs все)
  - Автоперезагрузка при обновлении ядра: когда да, когда нет
  - Уведомление на email при обновлении
- `unattended-upgrade --dry-run` — что было бы обновлено
- `unattended-upgrade -d` — запустить вручную с отладкой
- `needrestart` — показывает сервисы которые нужно перезапустить после обновления
- Docker-образы: они не обновляются автоматически
  - Фиксировать теги (не `latest`) — знает из Модуля 3
  - Периодически пересобирать образы: `docker compose build --pull`
  - `docker scout` или `trivy` — сканирование образов на уязвимости (упомянуть)
- **Упражнения:** включить unattended-upgrades, проверить что работает, посмотреть логи

> **Запомни:** Автообновления для пакетов системы. Docker-образы — вручную раз в месяц или через CI/CD.

#### Глава 4: Минимизация поверхности атаки

**Цель:** убрать всё лишнее что может быть использовано для атаки.

- Принцип: меньше запущено → меньше уязвимостей → меньше что ломать
- Аудит запущенных сервисов:
  ```bash
  systemctl list-units --type=service --state=running
  ss -tlnp   # что слушает порты
  ```
- Отключить ненужные сервисы:
  ```bash
  sudo systemctl disable --now bluetooth  # если нет Bluetooth
  sudo systemctl disable --now cups       # принтеры на сервере не нужны
  ```
- Аудит открытых портов:
  ```bash
  ss -tlnp                        # локально
  nmap -sV localhost              # полное сканирование
  ```
- Проверить что наружу открыто только нужное:
  ```bash
  # С внешней машины или через сервис
  nmap -p 1-65535 IP_сервера
  ```
- Пользователи: удалить неиспользуемых, проверить у кого есть sudo:
  ```bash
  getent passwd | grep -v nologin | grep -v false
  getent group sudo
  ```
- Файлы с SUID/SGID — потенциальный вектор:
  ```bash
  find / -perm /4000 -type f 2>/dev/null   # SUID файлы
  ```
- `lynis` — автоматический аудит безопасности системы:
  ```bash
  sudo lynis audit system
  ```
  Показывает список рекомендаций с приоритетами.
- **Упражнения:** запустить `lynis`, изучить топ-5 рекомендаций, применить те что понятны

### Часть 2: Мониторинг (Главы 5–7)

#### Глава 5: Метрики в реальном времени — Netdata

**Цель:** видеть состояние сервера в браузере, без настройки сложного стека.

- Зачем Netdata а не Prometheus+Grafana:
  - Устанавливается за 1 команду
  - Не требует конфигурации
  - Работает сразу после установки
  - Для одного сервера — более чем достаточно
- Установка:
  ```bash
  wget -O /tmp/netdata-kickstart.sh https://get.netdata.cloud/kickstart.sh
  sh /tmp/netdata-kickstart.sh --stable-channel --disable-telemetry
  ```
- Что показывает из коробки:
  - CPU (общий + по ядрам)
  - RAM (используется / кеш / свободно)
  - Диск (I/O, занятое место)
  - Сеть (входящий/исходящий трафик)
  - Процессы (топ по CPU и памяти)
  - Docker-контейнеры (CPU/RAM каждого)
  - Nginx/Caddy: запросы, коды ответов, таймауты
  - PostgreSQL: коннекты, запросы, блокировки
- Доступ: `http://server-ip:19999`
- Закрыть Netdata от интернета (только localhost или VPN):
  ```bash
  # /etc/netdata/netdata.conf
  [web]
    bind to = 127.0.0.1
  ```
  Потом пробросить через SSH-туннель:
  ```bash
  ssh -L 19999:localhost:19999 user@server
  # Открыть http://localhost:19999
  ```
- Алерты Netdata: встроенные (CPU > 85%, диск > 85%, RAM < 10%) — показать где смотреть
- **Упражнения:** установить Netdata, найти контейнер с максимальным потреблением RAM, создать нагрузку и посмотреть как реагируют графики

#### Глава 6: Алерты в Telegram

**Цель:** получать сообщение в телефон когда что-то пошло не так.

> **Почему Telegram, не email:** email часто не читают сразу. Telegram — мгновенно. Для алертов с одного сервера — идеальный вариант.

- Создать Telegram-бота: `@BotFather` → `/newbot` → получить токен
- Узнать свой `chat_id`:
  ```bash
  curl "https://api.telegram.org/bot<TOKEN>/getUpdates"
  ```
- Функция отправки алерта (использовать везде):
  ```bash
  send_alert() {
      local message="$1"
      curl -s -X POST \
        "https://api.telegram.org/bot${TG_TOKEN}/sendMessage" \
        -d "chat_id=${TG_CHAT_ID}" \
        -d "text=${message}" \
        -d "parse_mode=HTML" > /dev/null
  }
  ```
- Хранить токены в `/etc/alerting.env` с правами `600`
- **Скрипт мониторинга** `health-monitor.sh` — всё в одном:
  ```bash
  #!/bin/bash
  set -euo pipefail
  source /etc/alerting.env

  HOSTNAME=$(hostname)
  ALERTS=()

  # Диск
  DISK_USAGE=$(df / | awk 'NR==2 {print $5}' | tr -d '%')
  [ "$DISK_USAGE" -gt 80 ] && ALERTS+=("Диск: ${DISK_USAGE}% (порог 80%)")

  # RAM
  RAM_FREE=$(free | awk '/^Mem:/ {printf "%.0f", $4/$2*100}')
  [ "$RAM_FREE" -lt 10 ] && ALERTS+=("RAM: свободно ${RAM_FREE}%")

  # Приложение
  HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:8000/health)
  [ "$HTTP_CODE" != "200" ] && ALERTS+=("Приложение: healthcheck вернул ${HTTP_CODE}")

  # Docker-контейнеры
  STOPPED=$(docker ps -a --filter "status=exited" --format "{{.Names}}" | tr '\n' ' ')
  [ -n "$STOPPED" ] && ALERTS+=("Остановлены контейнеры: ${STOPPED}")

  # Отправить если есть алерты
  if [ ${#ALERTS[@]} -gt 0 ]; then
      MESSAGE="<b>Сервер: ${HOSTNAME}</b>%0A"
      for alert in "${ALERTS[@]}"; do
          MESSAGE+="• ${alert}%0A"
      done
      send_alert "$MESSAGE"
  fi
  ```
- Cron: запускать каждые 5 минут
- **Алерт при падении сервиса** через systemd:
  ```ini
  # В .service файле
  [Service]
  ExecStopPost=/usr/local/bin/notify-failure.sh %n %i
  ```
- **Алерт при успешном деплое** из CI/CD пайплайна
- **Ежедневный отчёт** — краткая сводка раз в сутки:
  ```
  Сервер myapp — ежедневный отчёт
  • Аптайм: 15 дней
  • Диск: 42% (8.4GB/20GB)
  • RAM: использовано 1.2GB/4GB
  • Бэкап вчера: OK (84MB)
  • fail2ban: заблокировано 47 IP за сутки
  ```
- **Упражнения:** создать бота, настроить алерт на диск, искусственно заполнить диск, получить алерт

#### Глава 7: Логи и logwatch

**Цель:** понимать что происходит на сервере без чтения сырых логов.

> **Важно:** Читатель знает `journalctl` и `tail -f` из Модуля 1. Здесь — агрегация и анализ, а не чтение по одному.

- Проблема сырых логов: `/var/log/auth.log` за неделю = тысячи строк
- `logwatch` — ежедневный дайджест системных логов:
  ```bash
  sudo apt install logwatch
  logwatch --output stdout --range today --detail low
  ```
  Показывает: попытки входа, sudo-действия, ошибки сервисов, Nginx статистику.
- Настройка logwatch для отправки в Telegram (через скрипт-обёртку вместо email)
- Ключевые логи и что в них искать:
  ```
  /var/log/auth.log       — попытки входа, sudo
  /var/log/syslog         — системные события
  /var/log/nginx/*.log    — запросы, ошибки, коды
  /var/log/fail2ban.log   — блокировки
  journalctl -u myapp     — твоё приложение
  ```
- `goaccess` — анализ Nginx access.log в реальном времени:
  ```bash
  sudo apt install goaccess
  goaccess /var/log/nginx/access.log --log-format=COMBINED
  ```
  Показывает: топ-IP, топ-URL, коды ответов, трафик.
- Что искать в логах как признаки атаки:
  - Много `Failed password` из одного IP → брутфорс
  - Запросы к `/wp-admin`, `/.env`, `/phpMyAdmin` → сканирование
  - 400/404 выше нормы → исследование сайта ботами
- Структурированные логи: JSON-формат в Nginx — почему удобнее для анализа
- **Упражнения:** запустить logwatch, найти в выводе IP с наибольшим числом попыток, проверить заблокирован ли он fail2ban

### Часть 3: Финал (Главы 8–9)

#### Глава 8: Аварийное восстановление — план действий

**Цель:** читатель знает что делать когда сервер упал в 3 ночи.

> **Важно:** Бэкапы и восстановление данных — в Модуле 5. Здесь — восстановление **сервиса** (когда данные есть, но что-то сломалось).

- Почему нужен план до аварии: в панике думать сложно
- **Runbook** — пошаговая инструкция для типичных сценариев:

  **Сценарий 1: Приложение не отвечает**
  ```
  1. curl https://domain.ru → что отвечает?
  2. docker compose ps → все контейнеры запущены?
  3. docker compose logs --tail=50 app → что в логах?
  4. df -h → не заполнен ли диск?
  5. free -h → хватает ли памяти?
  6. docker compose restart app → помогло?
  ```

  **Сценарий 2: Сервер не отвечает на SSH**
  ```
  1. ping IP → доступен?
  2. Консоль провайдера (VNC) → войти
  3. journalctl -b → что случилось при последней загрузке?
  4. df -h → не переполнен ли диск (частая причина)
  5. dmesg | tail → ошибки ядра?
  ```

  **Сценарий 3: База данных не стартует**
  ```
  1. docker compose logs db → ошибка в логах?
  2. Типично: "could not write to file" → диск полный
  3. Типично: "address already in use" → порт занят
  4. Восстановление из бэкапа → процедура из Модуля 5
  ```

- Как написать свой runbook: шаблон
- Контакты и доступы в одном месте: что должно быть под рукой в аварии
- **Упражнения:** написать runbook для своего проекта, проиграть сценарий "диск переполнен"

## Принципы написания

Все принципы предыдущих модулей, плюс специфичные:

### 1. Угрозы — конкретные, не абстрактные
Не "безопасность важна", а:
> "Прямо сейчас боты сканируют SSH-порт твоего сервера.
> `grep 'Failed password' /var/log/auth.log | wc -l` — сколько попыток было сегодня?"

### 2. Каждый инструмент — решение реальной проблемы
Сначала показывай проблему, потом инструмент:
- fail2ban появляется после `grep 'Failed password' /var/log/auth.log | sort | uniq -c | sort -rn`
- Netdata появляется после "приложение упало ночью, как узнать почему"

### 3. Алерты — практика, не теория
Для каждого алерта:
- Написать скрипт
- Запустить вручную
- Убедиться что сообщение пришло в Telegram
- Добавить в cron

### 4. Runbook — обязательно
Это финальный модуль. Читатель должен уйти с документом который поможет ему в 3 ночи.

### 5. Никакой воды
- Без WAF, IDS/IPS, SELinux, AppArmor в деталях
- Без Prometheus + Grafana + Alertmanager (слишком большой стек)
- Без аудита кода приложения
- Без penetration testing
