# Глава 2: Один VPS / один публичный сервис

> **Запомни:** один VPS — это уже архитектура, и в нем нужно явно разделить интернет-поверхность, управление, данные и восстановление.

---

## 2.1 Контекст и границы

Многие проекты застревают на стадии один сервер, все на нем. Это нормально, если архитектурно понятно, где входной периметр, где proxy, где приложение, где база, как устроены бэкапы и что будет при сбое.

Хорошая односерверная архитектура не обязана быть сложной, но обязана быть описанной и проверяемой.

Эта глава особенно полезна для личных проектов и небольших production-сервисов на одном сервере.

---

## 2.2 Как выглядит риск

Типовые слабые места:
- веб, база, админка и backup живут без границ;
- SSH торчит наружу без отдельного management path;
- нет reverse proxy и TLS discipline;
- секреты и данные лежат в непредсказуемых местах;
- нет плана миграции при росте нагрузки или риска.

### Где особенно важно
- личный сайт
- API на VPS
- небольшой internal service

---

## 2.3 Что строит защитник

- reverse proxy перед приложением;
- минимальный набор открытых портов;
- локальная БД и внутренние сервисы без внешней экспозиции;
- отдельный backup и restore процесс;
- документированная точка роста: когда и что выносить на отдельный узел.

### Практический результат главы
- ты умеешь описать secure baseline одного VPS;
- можешь показать, что на одном сервере уже разделено логически;
- понимаешь, когда пора выносить компоненты в отдельные слои.

```text
Интернет -> 443 -> nginx -> app

localhost only: db, redis, admin sockets
```

---

## 2.4 Практика

### Шаг 1: Собери схему одного VPS
- нарисуй входящий трафик, процессы, секреты, данные и backup path;
- отдельно опиши admin-доступ.

```bash
ss -tulpn
```

### Шаг 2: Проверь границы на хосте
- убедись, что БД и служебные сервисы слушают localhost или внутренний сокет;
- оставь наружу только reverse proxy и осознанный admin path.

```bash
ss -tulpn | rg '5432|3306|6379|22|80|443'
```

### Шаг 3: Опиши точку роста
- реши заранее, что первым выносится при росте: БД, storage, CI runner или bastion;
- это и будет дорожной картой эволюции.

```bash
printf 'document first split point
'
```

### Что нужно явно показать
- односерверную схему сервиса;
- какие компоненты доступны извне;
- где живут секреты, данные и бэкапы;
- какая первая точка архитектурного роста.

---

## 2.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- на своем VPS или lab-VM проверь, что наружу открыты только нужные сервисы;
- убедись, что БД и внутренние компоненты не публикуются в интернет;
- зафиксируй, как будет выглядеть вынос первого компонента на отдельный слой.

---

## 2.6 Типовые ошибки

- держать весь стек в одной открытой trust zone;
- не иметь reverse proxy и TLS discipline;
- не знать, где именно живут данные и backup path;
- не думать о первой точке роста заранее.
