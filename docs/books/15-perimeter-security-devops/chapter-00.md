# Глава 0: Threat model и поверхность атаки

> **Запомни:** хороший защитник не начинает с пакетов и правил. Он начинает с вопроса: что у меня вообще торчит наружу и кому это может быть интересно.

---

## 0.1 Кто реально атакует публичные серверы

Большинство атак на маленькие серверы не целевые. Это:

- интернет-боты, которые круглосуточно сканируют диапазоны IP;
- автоматические попытки входа по SSH;
- поиски открытых БД и панелей управления;
- сканирование типовых уязвимостей в веб-сервисах;
- всплески мусорного трафика и брутфорс форм логина.

Для маленького публичного сервера самая частая реальность такая:

```text
бот увидел порт 22
-> попробовал стандартные логины
-> не вошел
-> пошёл дальше
```

Твоя задача не "сделать сервер невидимым", а сделать его плохой целью для автоматического злоупотребления.

---

## 0.2 Что такое поверхность атаки

Поверхность атаки — это всё, что доступно атакующему снаружи или через доверенный путь.

Для одного VPS это обычно:

- открытые TCP/UDP-порты;
- баннеры сервисов;
- SSH;
- веб-вход через `80/443`;
- панели администрирования;
- лишние заголовки, публичные файлы и debug-endpoints;
- уязвимые зависимости и конфигурации.

### Простая карта поверхности атаки

```text
Интернет
  ├── SSH :22
  ├── HTTP :80
  ├── HTTPS :443
  ├── лишние порты? :5432 :3306 :8080 :9000
  └── DNS / домен / баннеры / сертификаты
```

Если хотя бы один лишний сервис смотрит наружу, ты уже увеличил площадь атаки.

## 0.3 Дом, VPS, small business, enterprise

Одна и та же идея выглядит по-разному по масштабу.

### Домашняя лаборатория

Главная цель — научиться не строить опасные привычки.

Что важно:
- не держать всё в одной плоской сети без понимания;
- не кидать сервер в "DMZ" домашнего роутера без разбора;
- делать snapshots перед тестами.

### Один VPS

Главная цель — не выставить наружу лишнее.

Что важно:
- только `22/80/443` или вообще `80/443` + VPN;
- БД не должна торчать наружу;
- SSH — только по ключам;
- firewall и логи обязательны.

### Small business

Главная цель — отделить пользователей, админ-доступ и публичные сервисы.

Что важно:
- gateway/firewall;
- сегментация;
- централизованные правила;
- понятная модель "кто куда может ходить".

### Enterprise pattern

Главная цель — та же, но больше слоёв и автоматизации:
- UTM/NGFW;
- WAF;
- IDS/IPS;
- SIEM/EDR;
- централизованная идентичность и процессы.

## 0.4 Что мы строим в этой книге

К концу книги у тебя должен быть не список теорий, а понятный defensive baseline.

```text
Интернет
  -> edge device / router / firewall
  -> reverse proxy / WAF basics
  -> host firewall
  -> SSH hardening
  -> fail2ban / CrowdSec
  -> логи и сигналы
```

И не менее важно: ты должен уметь безопасно проверить, что это всё реально работает.

---

## 0.5 Lab-only правила

Все проверки в этой книге проводятся только на своих системах.

Минимальная безопасная лаборатория:

- 1 VM: самый простой вариант;
- 2 VM: `client` + `server`;
- 3 VM: `client` + `gateway` + `server`.

Перед опасными шагами:

```bash
# 1. Сохрани состояние VM
# 2. Запиши текущие открытые порты
ss -tulpn

# 3. Запиши firewall baseline
sudo ufw status verbose

# 4. Запиши SSH baseline
sudo sshd -T | grep -E 'permitrootlogin|passwordauthentication|maxauthtries'
```
