# Глава 1: Что ставят на входе перед сервером

> **Запомни:** устройство перед сервером полезно, но оно не заменяет hardening самого сервера.

---

## 1.1 Картина целиком

Когда говорят "поставить что-то на входе перед сервером", обычно имеют в виду один из классов решений:

- router;
- firewall/security gateway;
- UTM/NGFW;
- reverse proxy;
- WAF;
- anti-DDoS/CDN слой.

Они похожи внешне, но решают разные задачи.

```text
Интернет
  -> router / gateway
  -> firewall policy
  -> reverse proxy / WAF
  -> Linux host
  -> app
```

---

## 1.2 Router, firewall, UTM, WAF — что делает что

### Router

Роутер:
- соединяет сети;
- делает NAT;
- умеет port forwarding;
- часто имеет очень базовые firewall-функции.

Домашний Linksys — это чаще всего именно роутер с ограниченным security stack.

### Firewall / security gateway

Firewall:
- разрешает и запрещает трафик по правилам;
- фильтрует входящие и исходящие соединения;
- может быть отдельным устройством или функцией на Linux.

### UTM / NGFW

Более тяжёлый класс:
- DPI;
- IDS/IPS;
- category filtering;
- application awareness;
- VPN;
- централизованные security-функции.

Это уже уровень small business и enterprise.

### Reverse proxy

Reverse proxy стоит перед приложением:
- принимает HTTP/HTTPS;
- завершает TLS;
- прячет backend;
- даёт rate limit и централизованные правила.

### WAF

WAF смотрит именно на HTTP/HTTPS:
- подозрительные запросы;
- SQLi/XSS-like patterns;
- аномалии в URI, headers и body.

WAF не заменяет firewall. Он решает другой уровень проблемы.

---

## 1.3 Что разумно для разных сценариев

### Дом

Минимум:
- нормальный роутер;
- отключенный UPnP;
- удалённое администрирование роутера выключено;
- только нужные порты наружу;
- лучше VPN для админки.

### Один VPS

Обычно хватает:
- cloud firewall/security group;
- host firewall на Linux;
- reverse proxy;
- fail2ban/CrowdSec.

### Small business

Обычно нужны:
- gateway/firewall;
- сегментация;
- VPN;
- IDS/IPS или хотя бы сетевое наблюдение;
- WAF/CDN перед веб-сервисом.

### Enterprise

Чаще всего есть:
- NGFW/UTM;
- отдельный WAF;
- managed anti-DDoS;
- централизованные логи;
- политика по зонам доверия.

---

## 1.4 Почему одного устройства мало

Ошибка мышления:

```text
"Я купил firewall -> теперь сервер защищён"
```

Реальность:
- firewall не исправит слабый `sshd_config`;
- WAF не исправит дырявый backend;
- роутер не закроет плохие права на Linux;
- Linux firewall не спасёт от большого volumetric DDoS.

Защита всегда многослойная.

## 1.5 Практика: быстро инвентаризировать точку входа

### На домашнем роутере / gateway

Проверь:
- отключено ли удалённое администрирование с интернета;
- включён ли UPnP;
- какие порты проброшены;
- есть ли режим `DMZ` и не включён ли он случайно.

### На Linux-сервере

```bash
ss -tulpn
sudo ufw status verbose
sudo lsof -i -P -n | grep LISTEN
```

Пример вывода `ss -tulpn`:

```
Netid State  Local Address:Port  Process
tcp   LISTEN 0.0.0.0:22          users:(("sshd",pid=812,fd=3))
tcp   LISTEN 0.0.0.0:5432        users:(("postgres",pid=1044,fd=5))
tcp   LISTEN 127.0.0.1:8000      users:(("python3",pid=1550,fd=7))
```

Что здесь искать:
- `0.0.0.0:22` — SSH слушает все интерфейсы, это нормально только если порт реально должен быть публичным;
- `0.0.0.0:5432` — PostgreSQL торчит наружу, для обычного VPS это плохой признак;
- `127.0.0.1:8000` — backend доступен только локально, это правильный вариант для сервиса за reverse proxy.

Задача проверки простая: публичных строк должно быть ровно столько, сколько нужно для твоего сценария.

### С другой своей машины

```bash
nmap -Pn SERVER_IP
nmap -Pn -p 22,80,443,5432,3306,6379 SERVER_IP
```

Пример разницы до и после firewall:

```
# До ufw
PORT     STATE  SERVICE
22/tcp   open   ssh
80/tcp   open   http
5432/tcp open   postgresql
```

```
# После ufw deny incoming + allow 22,80,443
PORT     STATE     SERVICE
22/tcp   open      ssh
80/tcp   open      http
5432/tcp filtered  postgresql
```

Что это значит:
- `open` — сервис отвечает и его реально видно снаружи;
- `filtered` — пакет режется firewall-ом или gateway, сервис уже не торчит в интернет;
- если `5432` по-прежнему `open`, значит слой периметра настроен не так, как ты ожидал.

Смысл не в самом `nmap`, а в вопросе: **видно ли снаружи что-то лишнее**.

## 1.6 Lab-only проверка

Сравни два сценария на своей lab:

1. сервер с открытыми лишними портами;
2. сервер после cleanup и firewall.

Проверь и зафиксируй разницу:

```bash
# До
nmap -Pn SERVER_IP

# После настройки gateway/firewall
nmap -Pn SERVER_IP
```

В логах и заметках важно зафиксировать:
- какие порты были открыты;
- какие остались;
- кто именно фильтрует трафик: gateway, cloud firewall или host firewall.

---

## 1.7 Типовые ошибки

- путать router и firewall;
- считать, что домашний `DMZ` — это настоящая DMZ-архитектура;
- открывать SSH всему интернету без hardening;
- публиковать БД наружу "на минутку" и забывать;
- ждать от WAF защиты от всех сетевых атак.
