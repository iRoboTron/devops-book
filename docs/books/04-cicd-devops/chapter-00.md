# Глава 0: Что такое CI/CD и зачем это нужно

> **Запомни:** CI/CD — это не инструмент. Это способ работы. Ты один раз настраиваешь конвейер — и он работает каждый раз одинаково. Без забывчивости, без "забыл шаг".

---

## 0.1 Жизнь без CI/CD

Ты настроил Docker в Модуле 3. Приложение работает. Но деплой выглядит так:

```bash
# 1. Зайти на сервер по SSH
ssh deploy@server

# 2. Перейти в директорию
cd /opt/myapp

# 3. Забрать новый код
git pull

# 4. Пересобрать образ
docker compose build

# 5. Перезапустить
docker compose up -d

# 6. Проверить что работает
curl http://localhost/health
```

**Знакомо?** А теперь представь:

### Сценарий 1: Забыл шаг

```bash
# Забыл docker compose build
git pull
docker compose up -d    ← старый образ, новый код не применён
```

Приложение "не обновилось". Ты час ищешь проблему.

### Сценарий 2: Сломал код

```bash
# Запушил баг
git push

# На сервере
git pull
docker compose build    ← собралось с багом
docker compose up -d    ← сломанное приложение в продакшне
```

Пользователи видят ошибку. Ты узнаёшь когда позвонят.

### Сценарий 3: Ручной деплой в 2 часа ночи

```
"Нужно пофиксить баг"
→ git push
→ SSH на сервер
→ git pull, build, up
→ "Работает"
→ Сон в 3:00
```

Каждый раз вручную. Каждый раз можно ошибиться.

## 0.2 Решение: конвейер

**CI/CD** — это конвейер который делает всё автоматически:

```
Ты: git push
         │
         ▼
┌─────────────────────────────────┐
│        CI/CD конвейер           │
│                                 │
│  1. Забирает код               │
│  2. Запускает тесты            │
│  3. Собирает Docker-образ      │
│  4. Публикует в реестр         │
│  5. Деплоит на сервер          │
│  6. Проверяет что работает     │
└─────────────────────────────────┘
         │
         ▼
   Сервер обновлён
```

**Каждый раз одинаково.** Без забывчивости. Без ошибок "забыл шаг".

## 0.3 CI и CD — в чём разница

### CI — Continuous Integration

```
git push → автоматически запускаются тесты → результат
```

**CI говорит:** "Этот коммит не сломал ничего."

### CD — Continuous Delivery

```
CI прошёл → образ собран → готов к деплою → нажми кнопку
```

**CD говорит:** "Код готов. Нажми кнопку чтобы задеплоить."

### CD — Continuous Deployment

```
CI прошёл → образ собран → автоматически на сервер
```

**CD говорит:** "Код прошёл все проверки — деплою сам."

### Визуально

```
Push → [CI: тесты] → [CD: сборка] → [CD: деплой]
                  │              │
        Delivery: кнопка    Deployment: автоматически
```

В этой книге мы строим **Continuous Deployment** — полный автопилот.

## 0.4 Инструменты: почему GitHub Actions

| Инструмент | Бесплатно | Сложность | Когда |
|-----------|-----------|-----------|-------|
| **GitHub Actions** | ✅ 2000 мин/мес | Низкая | Для начала |
| GitLab CI | ✅ 400 мин/мес | Средняя | Если уже GitLab |
| Jenkins | ✅ свой сервер | Высокая | Для больших команд |
| CircleCI | ✅ 6000 мин/мес | Средняя | Альтернатива |

**Почему GitHub Actions:**
- Бесплатно для публичных репозиториев
- Встроено в GitHub — не нужно ничего ставить
- Простой YAML-синтаксис
- Огромный маркетплейс готовых действий
- Достаточно для 95% проектов

> **Запомни:** GitHub Actions — не лучший инструмент вообще.
> Он лучший **для обучения**. Принципы CI/CD одинаковы везде.
> Научишься на Actions — разберёшься с любым другим.

## 0.5 Как GitHub Actions вписывается в рабочий процесс

```
Разработчик                GitHub                    Твой сервер
    │                      Actions                      │
    │                                                  │
    │  git push main                                   │
    ├────────────────────────►│                         │
    │                      │  Событие: push в main      │
    │                      │                            │
    │                      │  Job 1: Тесты              │
    │                      │  ✅ pytest                 │
    │                      │                            │
    │                      │  Job 2: Сборка Docker      │
    │                      │  ✅ docker build + push     │
    │                      │                            │
    │                      │  Job 3: Деплой             │
    │                      ├──────────────────────────►│
    │                      │                           │  docker pull
    │                      │                           │  docker compose up
    │                      │                           │
    │◄─────────────────────────────────────────────────│
    │  Сайт обновлён!                                  │
```

### Ключевые понятия

| Термин | Значение |
|--------|----------|
| **Workflow** | Файл `.github/workflows/ci.yml` — описание всего пайплайна |
| **Event** | Триггер: push, pull_request, schedule |
| **Job** | Задача в пайплайне: тесты, сборка, деплой |
| **Step** | Шаг внутри job: `run: pytest`, `uses: actions/checkout` |
| **Action** | Готовый шаг из маркетплейса: `actions/checkout@v4` |
| **Runner** | Виртуальная машина где выполняется job |
