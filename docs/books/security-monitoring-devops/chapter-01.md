# Глава 1: SSH — только ключи, никаких паролей

> **Запомни:** SSH с паролем = открытая дверь для ботов. SSH с ключом = дверь с замком который невозможно подобрать.

---

## 1.1 Почему пароли по SSH — плохая идея

Ты думаешь: "У меня сложный пароль `MyS3cur3P@ss!`".

Боты думают: "Попробуем 10 000 паролей в секунду".

### Реальность

```bash
# Посмотри сколько попыток было на твоём сервере:
grep "Failed password" /var/log/auth.log | wc -l
# 500? 2000? 10000?
```

Каждая попытка — бот пробует пароль. Рано или поздно угадает.

### SSH-ключ vs пароль

| | Пароль | SSH-ключ (2048 бит) |
|--|--------|---------------------|
| Перебор | Часы/дни | Миллиарды лет |
| Перехват | Можно (MITM) | Практически нет |
| Ввод | Каждый раз | Автоматически |
| Отзыв | Поменять везде | Удалить один ключ |

> **Запомни:** RSA 2048 бит = 2²⁰⁴⁸ комбинаций.
> Все компьютеры мира не переберут за время жизни вселенной.

## 1.2 Ключевые параметры sshd_config

Файл: `/etc/ssh/sshd_config`

### PermitRootLogin no

```
PermitRootLogin no
```

**Что:** Запрещает вход под root.

**Зачем:** root = полный доступ. Если подберут пароль root — конец.

**Как зайти:** Зайди как `deploy`, потом `sudo`.

---

### PasswordAuthentication no

```
PasswordAuthentication no
```

**Что:** Отключает вход по паролю. Только ключи.

**Зачем:** Боты не могут перебрать ключи.

**Порядок важен:** Сначала убедись что ключ работает — потом отключай пароли!

---

### PubkeyAuthentication yes

```
PubkeyAuthentication yes
```

**Что:** Разрешает вход по ключам.

---

### MaxAuthTries 3

```
MaxAuthTries 3
```

**Что:** Максимум 3 попытки аутентификации.

**Зачем:** Усложняет брутфорс.

---

### LoginGraceTime 20

```
LoginGraceTime 20
```

**Что:** 20 секунд на вход.

**Зачем:** Боты подключаются и "висят" — это занимает ресурсы.

---

### AllowUsers

```
AllowUsers deploy
```

**Что:** Только пользователь `deploy` может войти по SSH.

**Зачем:** Даже если пароль подобран — другой пользователь не зайдёт.

---

## 1.3 Порядок действий: НЕ закрой себя снаружи

> **Опасно:** Неправильный порядок = потеря доступа к серверу.

```
ШАГ 1: Добавить публичный ключ в authorized_keys
ШАГ 2: Открыть НОВУЮ сессию SSH и проверить что вход по ключу работает
ШАГ 3: Отключить PasswordAuthentication
ШАГ 4: Перезапустить sshd
ШАГ 5: НЕ закрывать текущую сессию пока новая работает
```

Если что-то пошло не так — у тебя ещё есть открытая сессия для исправления.

---

## 1.4 Практика: настраиваем sshd

### Шаг 1: Проверь что ключ работает

```bash
# На локальной машине
ssh deploy@server-ip
# Вошёл без пароля? ✅
```

Если нет — сначала настрой SSH-ключ (Модуль 1, Глава 9).

### Шаг 2: Открой вторую сессию

Не закрывай текущую! Открой новый терминал:

```bash
ssh deploy@server-ip
# Работает? Отлично.
```

### Шаг 3: Редактируем sshd_config

```bash
sudo nano /etc/ssh/sshd_config
```

Найди и измени:

```
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
MaxAuthTries 3
LoginGraceTime 20
AllowUsers deploy
```

### Шаг 4: Проверь конфиг

```bash
sudo sshd -t
```

Если ошибки нет — конфиг валидный.

> **Запомни:** `sshd -t` = аналог `nginx -t` для SSH.
> Всегда проверяй перед перезапуском.

### Шаг 5: Перезапусти

```bash
sudo systemctl reload sshd
```

`reload` = применить конфиг без обрыва текущих соединений.
НЕ `restart` — он оборвёт все сессии.

### Шаг 6: Проверь в новой сессии

Открой ещё один терминал:

```bash
ssh deploy@server-ip
# Работает по ключу? ✅
```

Попробуй с паролем (с другой машины или с `PasswordAuthentication yes`):

```bash
ssh -o PreferredAuthentications=password deploy@server-ip
# Permission denied (publickey) ← правильно!
```

### Шаг 7: Не закрывай сессии пока не уверен

У тебя должно быть 3 терминала:
1. Старая сессия (была до изменений)
2. Новая сессия (после reload)
3. Тестовая (проверка что пароль не работает)

Когда все 3 подтверждают что всё ок — можно работать.

## 1.5 ~/.ssh/config — алиасы на клиенте

На своей машине (не на сервере):

```bash
nano ~/.ssh/config
```

```
Host prod
    HostName 203.0.113.50
    User deploy
    IdentityFile ~/.ssh/id_ed25519
    Port 22

Host staging
    HostName 192.168.1.100
    User deploy
    IdentityFile ~/.ssh/id_ed25519
```

Теперь вместо:

```bash
ssh -i ~/.ssh/id_ed25519 deploy@203.0.113.50
```

Просто:

```bash
ssh prod
```

## 1.6 Двухфакторная аутентификация (упоминаю)

`libpam-google-authenticator` — требует код из приложения на телефоне + SSH-ключ.

**Когда нужно:**
- Сервер с очень ценными данными
- Соответствие требованиям безопасности

**Когда избыточно:**
- Один сервер, один разработчик
- Уже есть SSH-ключ + fail2ban

> **Совет:** Для большинства проектов ключ + fail2ban = достаточно.
> 2FA — если требования или клиенты настаивают.

## 📝 Упражнения

### Упражнение 1.1: Настроить sshd_config
**Задача:**
1. Открой 2 терминала на сервер (не закрывай!)
2. В одном редактируй `/etc/ssh/sshd_config`:
   - `PermitRootLogin no`
   - `MaxAuthTries 3`
   - `LoginGraceTime 20`
3. Проверь конфиг: `sudo sshd -t`
4. Перезапусти: `sudo systemctl reload sshd`
5. Проверь в новом терминале: `ssh deploy@server`

### Упражнение 1.2: Отключить пароли
**Задача:**
1. Убедись что ключ работает: `ssh deploy@server` (без пароля)
2. Добавь `PasswordAuthentication no` в sshd_config
3. Проверь: `sudo sshd -t`
4. Перезапусти: `sudo systemctl reload sshd`
5. Проверь в новом терминале — ключ работает?
6. Попробуй войти по паролю — отказано?

### Упражнение 1.3: SSH алиасы
**Задача:**
1. Создай `~/.ssh/config` на своей машине
2. Добавь алиас для сервера
3. Подключись: `ssh prod` (короткая команда)

### Упражнение 1.4: DevOps Think
**Задача:** «Ты отключил пароли и закрыл терминал. Потом понял что ключ не работает. Что делать?»

Ответ:
1. Если есть другая открытая сессия — исправь через неё
2. Если нет — через VNC/консоль провайдера
3. Если нет VNC — перезагрузи сервер в single-user mode
4. Урок: ВСЕГДА проверяй вход по ключу в НОВОЙ сессии перед отключением паролей
5. Держи 2-3 терминала открытыми пока не уверен что всё работает
