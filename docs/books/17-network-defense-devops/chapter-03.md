# Глава 3: VLAN и сегментация

> **Запомни:** когда все узлы живут в одной подсети, компрометация одного хоста быстро превращается в проблему всей сети.

---

## 3.1 Контекст и границы

Сегментация — это способ ограничить blast radius. Она нужна не только enterprise-сетям. Даже дома полезно отделить серверы, рабочие станции, IoT и guest-сеть.

VLAN не является полной защитой сама по себе. Она задает границы уровня 2/3, которые нужно подкрепить routing/firewall политиками и понятной схемой потоков.

Эта глава особенно полезна для лабораторий, офисов и небольших production-сетей с несколькими группами устройств.

---

## 3.2 Как выглядит риск

Типовые слабые места:
- одна flat network для серверов, ноутбуков и IoT — компрометация любого узла даёт широкую lateral movement поверхность.
  Проверить: `ip a`, `ip r` и список подсетей; если всё живёт в одной сети, сегментации фактически нет.
- нет правил east-west трафика — рабочие станции и IoT могут ходить к серверным портам напрямую.
  Проверить: `nc -vz DB_IP 5432` или `nc -vz SERVER_IP 22` с разных зон.
- management plane живет в той же подсети, что и пользователи — административные интерфейсы доступны оттуда же, где обычный клиентский шум.
  Проверить: попробовать доступ к management IP из user-сегмента.
- доступ между сегментами разрешен по умолчанию — firewall/VLAN формально существуют, но матрица потоков ничего не ограничивает.
  Проверить: пройтись по критичным направлениям из матрицы и отметить, что неожиданно оказывается доступным.
- нет карты, какие VLAN существуют и зачем — через месяц никто не помнит, почему правило вообще было открыто.
  Проверить: есть ли документ с зонами, подсетями и допустимыми потоками.

### Где особенно важно
- домашняя lab
- офис 10-50 устройств
- small business с Wi‑Fi и серверами

---

## 3.3 Что строит защитник

- выделение VLAN/подсетей по ролям: users, servers, management, guest, IoT;
- маршрутизация между сегментами только через контролируемый gateway;
- default deny между зонами и точечные исключения;
- отдельный management segment;
- описание потоков: кто к кому и по каким портам может ходить.

### Практический результат главы
- ты можешь объяснить, зачем сегментация нужна даже без enterprise-коммутатора;
- умеешь проектировать зоны по функциям;
- понимаешь, где заканчивается VLAN и начинается firewall policy.

```
users -> app 443/tcp allow
app -> db 5432/tcp allow
management -> all admin ports allow
guest -> servers deny
iot -> management deny
```

---

## 3.4 Практика

### Шаг 1: Опиши зоны своей сети
- выпиши все типы устройств и сервисов;
- разбей их на 3-5 зон с разным уровнем доверия.

```bash
ip a
ip r
```

### Шаг 2: Построй матрицу доступов
- для каждой зоны ответь, к кому она должна иметь доступ и по каким портам;
- отдельно выдели management-plane.

```bash
cat > /tmp/zone-matrix.txt <<'EOF'
users->app 443
app->db 5432
management->all admin
EOF
cat /tmp/zone-matrix.txt
```

### Шаг 3: Проверь фактическую связность
- с разных узлов проверь reachability до сервисов;
- сравни реальность с матрицей доступов.

```bash
ping -c 2 SERVER_IP
nc -vz DB_IP 5432
```

Дополнительная проверка изоляции между зонами:

```bash
nc -vz MANAGEMENT_IP 22
nc -vz DB_IP 5432
nc -vz SERVER_IP 443
```

Как читать результат:
- `Connection refused` — хост ответил, но сервис не слушает порт; трафик до цели дошёл;
- `timeout` — пакет режется по пути, значит межзонная изоляция реально работает;
- из `app` к `db` ожидаем `Connected`, если это разрешённый поток по матрице.

### Что нужно явно показать
- список зон и их назначение;
- какие сервисы доступны межзонно;
- где выделен management segment;
- какая политика действует по умолчанию.

---

## 3.5 Захват трафика для проверки

Иногда проще доказать сегментацию не словом, а пакетом:

```bash
sudo tcpdump -ni eth0 src MANAGEMENT_IP -c 10
ping SERVER_IP
```

Если пакеты от `MANAGEMENT_IP` видны в `tcpdump`, трафик management -> servers проходит. Если пакетов нет, значит путь реально изолирован на уровне сети или firewall.

## 3.6 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- проверь, что рабочая станция не видит лишние административные порты сервера;
- убедись, что guest или IoT зона не может ходить к management подсети;
- зафиксируй blast radius при компрометации одного хоста и как его режет сегментация.

---

## 3.7 Типовые ошибки

- создать VLAN и забыть про ACL/firewall;
- держать management интерфейсы в пользовательской сети;
- не вести карту зон и связей;
- считать flat network нормой просто потому, что так проще.
