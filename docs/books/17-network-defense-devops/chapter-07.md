# Глава 7: Tcpdump, Wireshark и pcap basics

> **Запомни:** сетевой дамп нужен не для красивой картинки, а чтобы инженер мог доказать, что на проводе происходило на самом деле.

---

## 7.1 Контекст и границы

Логи приложений и firewall могут терять детали. pcap позволяет увидеть фактический трафик: handshake, DNS-resolve, retries, reset, направление соединения и временную последовательность.

Важно уметь снимать минимум нужного трафика, не утонуть в гигабайтах и не собирать больше данных, чем нужно для задачи.

Эта глава особенно полезна для всех, кто расследует сетевые проблемы, аномалии или хочет проверить гипотезу руками.

---

## 7.2 Как выглядит риск

Типовые слабые места:
- снимок делается не на том интерфейсе — нужный трафик просто не попадает в capture.
  Проверить: `ip -br a` и сверить, по какому интерфейсу реально идёт нужный маршрут.
- собирается слишком широкий дамп без фильтрации — вместо одной гипотезы получается гигабайтный мусор.
  Проверить: есть ли в команде `tcpdump` host/port/BPF-фильтр.
- pcap содержит чувствительные данные и хранится как попало — артефакт расследования сам становится утечкой.
  Проверить: где сохраняется файл и как он удаляется после разбора.
- инженер не умеет связать сетевой дамп с логами приложений — пакет виден, а смысла в нём нет.
  Проверить: сопоставить timestamp `tcpdump` с access log или journald.
- делается вывод по одному пакету без контекста — ошибка интерпретации превращается в ложный вывод об инциденте.
  Проверить: смотреть не один SYN, а последовательность DNS -> TCP -> TLS -> HTTP.

### Где особенно важно
- debug на сервере
- анализ VPN
- сетевые аномалии
- проверка DNS, HTTP и TLS поведения

## 7.3 Что строит защитник

- точный выбор интерфейса и BPF-фильтра;
- короткие controlled captures;
- связка временной шкалы pcap и логов;
- хранение и удаление pcap как чувствительного артефакта;
- документирование цели съемки.

### Практический результат главы
- ты умеешь снять минимальный полезный pcap;
- понимаешь, что смотреть в Wireshark и что проще увидеть через tcpdump;
- можешь доказать путь запроса от клиента к сервису.

```bash
sudo tcpdump -i eth0 -nn host 203.0.113.10 and \(port 53 or port 80 or port 443\)
sudo tcpdump -i eth0 -nn -w /tmp/capture.pcap host 203.0.113.10
```

## 7.4 Практика

### Шаг 1: Выбери интерфейс и фильтр
- сначала определи интерфейс, по которому идет нужный трафик;
- затем сузь фильтр до хоста, порта или протокола.

```bash
ip -br a
```

### Шаг 2: Сними controlled pcap
- запусти короткий capture и параллельно создай один проверочный запрос;
- не собирай весь трафик сети без причины.

```bash
sudo tcpdump -i eth0 -nn -c 50 host SERVER_IP
curl -I https://HOST
```

### Шаг 3: Свяжи дамп и логи
- сопоставь время пакетов с временем access log и системного лога;
- проверь реальный IP, порты и статус ответа.

```bash
sudo tail -n 20 /var/log/nginx/access.log
```

### Что нужно явно показать
- какой интерфейс использовался для capture;
- какой BPF-фильтр был применен;
- как сетевое событие бьется с логом приложения;
- где хранится pcap и как он удаляется после работы.

---

## 7.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- сними pcap одного запроса к своему сервису и покажи DNS + TCP/TLS + HTTP цепочку;
- проверь, как выглядит неуспешное соединение по сравнению с успешным;
- не используй чужой трафик и не собирай больше данных, чем нужно для своей lab.

---

## 7.6 Типовые ошибки

- слушать не тот интерфейс;
- забывать про -nn и утонуть в reverse DNS;
- хранить pcap с чувствительными данными без контроля;
- делать выводы без сопоставления с логами приложений.
