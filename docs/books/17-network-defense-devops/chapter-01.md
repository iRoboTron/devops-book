# Глава 1: Router, firewall, UTM и NGFW

> **Запомни:** сетевую защиту удобно ломать понятийной путаницей: роутер делает не то же самое, что firewall, а UTM и WAF закрывают другие уровни.

---

## 1.1 Контекст и границы

Почти все проблемы на границе сети начинаются с неверной модели. Если админ считает домашний роутер полноценным perimeter firewall, он открывает слишком много и слишком поздно узнает о последствиях.

Инженеру важно понимать, какие задачи решают NAT, stateful firewall, IDS/IPS, VPN termination и reverse proxy.

Эта глава особенно полезна для дома, VPS и small business, где нужно понять, что именно ставить на входе и за что отвечает каждый слой.

---

## 1.2 Как выглядит риск

Типовые слабые места:
- на границе сети нет инвентаризации открытых сервисов — наружу остаются забытые порты, и команда узнаёт о них только после инцидента.
  Проверить: `nmap -Pn -p 22,80,443,5432,3306,6379,8080,9090 SERVER_IP`.
- админ верит, что режим DMZ на SOHO-роутере эквивалентен реальной DMZ — весь трафик уходит на один хост, но изоляции между зонами так и не появляется.
  Проверить: посмотреть настройки роутера и убедиться, что не включён полный DMZ на один адрес.
- один и тот же узел одновременно принимает интернет и хранит внутренние сервисы — компрометация веб-сервиса сразу открывает путь к БД и админ-панелям.
  Проверить: `ss -tulpn` и отметить, какие внутренние сервисы слушают вместе с публичными.
- журналы граничного устройства не собираются — потом невозможно доказать, какой слой пропустил соединение.
  Проверить: открыть логи gateway/firewall и убедиться, что события allow/deny реально пишутся.
- ожидается, что WAF решает задачи сетевой фильтрации — HTTP-уровень подменяет собой firewall-модель, и лишние TCP-порты остаются открытыми.
  Проверить: сравнить список открытых портов через `nmap` и список правил WAF/proxy.

### Где особенно важно
- домашняя лаборатория
- один офисный интернет-канал
- малый бизнес с 1-2 публичными сервисами

---

## 1.3 Что строит защитник

- четкая схема: router/gateway -> firewall policy -> reverse proxy/WAF -> hosts;
- минимальный набор проброшенных портов;
- отдельный VPN для администрирования;
- логирование событий граничного устройства;
- инвентаризация внешней экспозиции с регулярной проверкой.

### Практический результат главы
- ты отличаешь маршрутизацию, фильтрацию и прикладную защиту;
- можешь обосновать, нужен ли отдельный firewall appliance или достаточно cloud firewall + host firewall;
- понимаешь границы SOHO-устройств.

```bash
sudo ufw default deny incoming
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw allow from 203.0.113.10 to any port 22 proto tcp
sudo ufw enable
```

## 1.4 Практика

### Шаг 1: Нарисуй реальную схему входа
- запиши, где заканчивается интернет и что стоит перед сервером;
- отметь, кто делает NAT, кто фильтрует трафик, кто завершает TLS.

```bash
ip r
ss -tulpn
```

### Шаг 2: Проверь внешнюю экспозицию
- со своей второй машины просканируй только свой сервер или свой внешний IP;
- сопоставь результат со списком port-forwarding и firewall rules.

```bash
nmap -Pn SERVER_IP
nmap -Pn -p 22,80,443,5432,3306,6379,8080,9090 SERVER_IP
```

Пример вывода:

```
PORT     STATE    SERVICE
22/tcp   open     ssh
80/tcp   open     http
443/tcp  open     https
5432/tcp filtered postgresql
3306/tcp filtered mysql
8080/tcp open     http-proxy
```

Как это читать:
- `filtered` для БД и кэша — хороший результат, периметр режет доступ;
- `open` на `8080` или `9090` часто означает случайно опубликованную админку;
- список должен совпадать с тем, что ты сознательно публикуешь.

### Шаг 3: Собери минимальную двухслойную схему
- закрой лишнее на gateway и дублируй критичные ограничения на Linux host;
- оставь наружу только нужные сервисы.

```bash
sudo ufw status verbose
```

## 1.5 Двухслойная проверка

Проверь, что cloud firewall и host firewall работают независимо:

```bash
sudo ufw disable
nmap -Pn -p 5432 SERVER_IP
sudo ufw enable
```

Если после временного отключения `ufw` порт `5432` всё ещё остаётся `filtered`, значит внешний слой тоже держит защиту. Если порт внезапно становится `open`, ты зависишь только от одного firewall.

### Что нужно явно показать
- какие порты доступны извне;
- какой слой их разрешает;
- есть ли отдельный путь для администрирования;
- как собираются журналы пограничного устройства.

---

## 1.6 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- сравни результаты nmap до и после ужесточения правил;
- убедись, что SSH не открыт всему интернету без необходимости;
- зафиксируй, где именно режется соединение: gateway или host firewall.

---

## 1.7 Типовые ошибки

- путать роутер и полноценный firewall;
- использовать SOHO DMZ как замену сегментации;
- открывать базу данных наружу для удобства;
- полагаться на один слой фильтрации.
