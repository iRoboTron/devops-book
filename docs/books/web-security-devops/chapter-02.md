# Глава 2: SQLi и безопасная работа с базой

> **Запомни:** SQL injection обычно начинается не с базы как таковой, а с решения разработчика строить запросы строками и не ограничивать поведение пользователя.

---

## 2.1 Контекст и границы

Реальная проблема в том, что приложение смешивает пользовательский ввод с синтаксисом SQL и доверяет динамической логике больше, чем структуре запроса.

Даже если ORM закрывает базовый класс инъекций, остаются опасные места: raw SQL, динамический ORDER BY, поиск по колонкам, генерация отчетов и административные фильтры.

Эта глава особенно полезна для всех приложений, где есть поиск, фильтрация, сортировка, отчеты и административные запросы.

---

## 2.2 Как выглядит риск

Типовые слабые места:
- ручная сборка SQL через f-string или конкатенацию;
- динамический список колонок без allowlist;
- единый DB user с правами на все схемы;
- вывод подробных SQL-ошибок в ответ приложения;
- отсутствие запроса к базе в audit trail.

### Где особенно важно
- поиск по каталогу
- админские таблицы
- экспорт CSV/Excel
- фильтрация по множеству параметров

---

## 2.3 Что строит защитник

- параметризованные запросы и placeholders вместо сборки строк;
- allowlist для колонок и направлений сортировки;
- разделение DB users: app readonly, app readwrite, migration;
- ограничение ошибок наружу и трассировка внутрь логов;
- минимальные привилегии на схему и таблицы.

### Практический результат главы
- ты различаешь data values и SQL structure;
- умеешь безопасно реализовать фильтры и сортировку;
- понимаешь, почему DBA-права приложения опасны даже без явной SQLi.

```text
# Плохо
query = f"SELECT * FROM users WHERE email = '{email}'"

# Лучше
cur.execute('SELECT * FROM users WHERE email = %s', (email,))

allowed_sort = {'created_at', 'email'}
sort = sort if sort in allowed_sort else 'created_at'
query = f'SELECT * FROM users ORDER BY {sort} DESC LIMIT %s'
cur.execute(query, (50,))
```

## 2.4 Практика

### Шаг 1: Найди сырой SQL в кодовой базе
- ищи execute(, raw(, text( и похожие паттерны;
- проверь, какие запросы собираются динамически и чем это обосновано.

```bash
rg -n "execute\(|raw\(|text\(|SELECT .*\{" app/ src/
```

### Шаг 2: Раздели структуру и данные
- вынеси допустимые поля сортировки в allowlist;
- значения фильтров передавай параметрами драйвера;
- ошибки БД отправляй только во внутренний лог.

```bash
psql "$DATABASE_URL" -c '\du'
psql "$DATABASE_URL" -c '\dn+'
```

### Шаг 3: Проверь DB-права приложения
- посмотри, может ли app user менять системные объекты или читать лишние схемы;
- раздели пользователя миграций и пользователя рантайма.

```bash
psql "$DATABASE_URL" -c '\z'
```

### Что нужно явно показать
- пример параметризованного запроса;
- пример allowlist для сортировки;
- снимок прав DB user;
- лог SQL-ошибки внутри сервера без утечки в HTTP-ответ.

---

## 2.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- на своем стенде отправь в параметры фильтра значения с кавычками и спецсимволами и убедись, что приложение корректно обрабатывает их как данные;
- проверь, что сортировка по недопустимому полю отклоняется или откатывается к безопасному значению;
- убедись, что пользователь приложения не может делать schema-changing операции.

---

## 2.6 Типовые ошибки

- считать ORM полной заменой review запросов;
- разрешать произвольный order by из query string;
- держать общий суперпользовательский DSN в .env приложения;
- показывать stack trace и текст SQL-ошибки клиенту.
