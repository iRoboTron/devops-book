# Глава 6: Security roadmap

> **Запомни:** дорожная карта нужна, чтобы не расползаться по модным задачам и строить безопасность по приоритетам и зрелости.

---

## 6.1 Контекст и границы

Security roadmap — это способ выбрать правильную последовательность: сначала внешний периметр, auth, backup и logging; потом сегментация, supply chain, detection; потом более сложные платформенные решения.

Без roadmap легко тратить время на красивый инструмент при отсутствии базовых вещей вроде backups, secrets hygiene и четкого admin path.

Эта глава особенно полезна для одиночного инженера, команды и растущей компании.

---

## 6.2 Как выглядит риск

Типовые слабые места:
- хаотичные улучшения без приоритетов — команда делает то, что интересно, а не то, что реально снижает риск.
  Проверить: есть ли backlog и причина, почему задача делается сейчас.
- нет связи между рисками и задачами — нельзя объяснить, какой риск закрывает конкретная мера.
  Проверить: у каждой задачи есть risk statement и expected outcome.
- базовые проблемы не решены, но ресурсы уходят на advanced tooling — SIEM появляется раньше backup restore.
  Проверить: закрыты ли SSH, firewall, backup и secrets hygiene.
- roadmap не привязан к владельцам и срокам — задачи зависают между командами и кварталами.
  Проверить: owner и due date по каждому пункту.
- нет повторной оценки после внедрения — задача закрыта формально, но риск мог остаться.
  Проверить: есть ли критерий done и дата ревью.

### Где особенно важно
- личный проект
- малая команда
- growth stage
- платформенная группа

---

## 6.3 Что строит защитник

- приоритизация по риску и blast radius;
- группировка по уровням зрелости: must have, next, later;
- owners, сроки и критерии завершения;
- измеримые результаты: что именно станет лучше;
- регулярный пересмотр roadmap.

### Практический результат главы
- ты умеешь строить roadmap безопасности без псевдо-стратегии;
- знаешь, как связать задачи с риском и зрелостью;
- можешь показать, что делать сначала на доме, VPS, small business и дальше.

| Критерий | P1 (сейчас) | P2 (в ближайшие 2-3 мес) | P3 (потом) |
|----------|-------------|--------------------------|------------|
| Impact если реализуется | Потеря данных или недоступность > 4ч | Утечка данных или деградация 1-4ч | Частичный ущерб или неудобство |
| Вероятность | Уже происходит или активно эксплуатируется | Реальный вектор для твоей среды | Пока теоретический риск |
| Стоимость исправления | Часы | Дни | Недели |

Примеры P1:
- backup не существует или ни разу не проверялся;
- SSH принимает парольную аутентификацию из интернета;
- root login разрешён по SSH;
- нет firewall или БД открыта наружу;
- `.env` с реальными секретами лежит в Git.

Примеры P2:
- секреты не ротируются;
- нет rate limit на auth endpoints;
- нет централизованных логов;
- есть устаревшие пакеты с известными CVE.

Примеры P3:
- IDS/IPS;
- SIEM интеграция;
- WAF перед приложением;
- advanced threat hunting.

## 6.4 Практика

### Шаг 1: Собери backlog безопасности
- выпиши все найденные риски и архитектурные пробелы;
- не сортируй по интересности, сортируй по impact и вероятности.

```bash
ss -tulpn
systemctl list-units --type=service --state=running --no-pager
find /opt /home -name ".env" 2>/dev/null -exec ls -la {} \;
sudo ufw status | head -5
```

### Шаг 2: Разбей на приоритеты
- сделай 3 корзины: must have, next, later;
- для каждой задачи добавь owner и критерий завершения.

```bash
cat > /tmp/security-roadmap.md <<'EOF'
P1: backup restore, ssh hardening, firewall, secrets out of git
P2: segmentation, dependency scanning, central logs
P3: IDS/IPS, SIEM, WAF
EOF
cat /tmp/security-roadmap.md
```

### Шаг 3: Привяжи к масштабу
- отметь, что актуально уже дома, что на VPS, что в small business, а что имеет смысл позже.

```bash
awk '{print "scope:", $0}' /tmp/security-roadmap.md
```

## 6.5 Как провести security audit за 1 час

```bash
#!/bin/bash
echo "=== [1] SSH ==="
sshd -T | grep -E 'permitrootlogin|passwordauthentication'

echo "=== [2] Firewall ==="
sudo ufw status | head -5

echo "=== [3] Open ports ==="
ss -tulpn | grep LISTEN

echo "=== [4] .env permissions ==="
find /opt /home -name ".env" 2>/dev/null -exec ls -la {} \;

echo "=== [5] sudoers changes ==="
stat /etc/sudoers | grep Modify

echo "=== [6] New users ==="
awk -F: '{print $1, $3}' /etc/passwd | sort -t' ' -k2 -n | tail -5

echo "=== [7] Security updates ==="
apt list --upgradable 2>/dev/null | grep -i security

echo "=== [8] Fresh backups ==="
find /var/backups -name "*.sql" -o -name "*.tar.gz" 2>/dev/null | \
  xargs -I{} stat {} | grep Modify | sort -r | head -3
```

Такой быстрый аудит даёт сырое состояние инфраструктуры, из которого потом строится roadmap.

### Что нужно явно показать
- roadmap с приоритетами;
- owners и критерии завершения;
- какие задачи относятся к дому, VPS и small business;
- что сознательно отложено и почему.

---

## 6.6 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- построй roadmap для своей lab или реальной инфраструктуры на ближайшие 3 этапа;
- не добавляй больше задач, чем реально сможешь поддерживать;
- проверь, что базовые меры стоят выше advanced tooling.

---

## 6.7 Типовые ошибки

- строить roadmap по моде;
- не назначать owners и критерии закрытия;
- ставить advanced tooling выше базового hardening;
- не пересматривать приоритеты.
