# Глава 6: Lateral movement theory

> **Запомни:** если один узел скомпрометирован, следующая защитная задача — не дать этому превратиться в перемещение по всей инфраструктуре.

---

## 6.1 Контекст и границы

Lateral movement в защитной оптике — это вопрос связности, доверия и повторного использования учетных данных. Его останавливают сегментация, separate admin plane, минимальные права и хорошая инвентаризация доступов.

Важно заранее понимать, какие связи между узлами реально нужны, а какие сложились исторически и только увеличивают риск.

Эта глава особенно полезна для тех, кто строит несколько VM, зоны, VPN и хочет ограничить blast radius.

---

## 6.2 Как выглядит риск

Типовые слабые места:
- flat network;
- shared credentials между хостами;
- одни и те же SSH keys и admin users везде;
- management plane совпадает с пользовательской сетью;
- от одного узла доступны БД, хранилища и другие серверы без явной причины.

### Где особенно важно
- lab с несколькими VM
- small business
- группа VPS

---

## 6.3 Что строит защитник

- сегментация сети и management plane;
- раздельные доступы и keys per host and role;
- jump host или VPN для администрирования;
- ограничение east-west traffic;
- visibility на межхостовые соединения.

### Практический результат главы
- ты умеешь описывать lateral movement risk как карту доверия;
- можешь показать, что сокращает blast radius в твоей сети;
- понимаешь, почему одна компрометация не должна автоматически открывать всю инфраструктуру.

```text
ssh host-b
nc -vz db.internal 5432
ip r
```

---

## 6.4 Практика

### Шаг 1: Построй карту доверия
- для каждого хоста и зоны запиши, к кому он может ходить по сети и с какими credentials;
- отдельно выдели admin paths.

```bash
printf 'host->host/port and credential map
'
```

### Шаг 2: Сократи лишнюю связность
- убери один ненужный east-west доступ или shared credential;
- проверь, что бизнес-функция не пострадала.

```bash
ss -tulpn
```

### Шаг 3: Проверь видимость межхостовых соединений
- зафиксируй, где логируются соединения между зонами и админские входы;
- добавь недостающий logging.

```bash
sudo journalctl -u ssh -n 50 --no-pager
```

### Что нужно явно показать
- какие узлы связаны между собой;
- какие credentials повторно используются;
- где проходит management plane;
- какие связи уже сокращены.

---

## 6.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- на своей lab сравни схему до и после сегментации или VPN для администрирования;
- убедись, что одна рабочая станция не дает прямой доступ ко всем серверам;
- не выполняй offensive-перемещения, цель — defensive mapping и hardening.

---

## 6.6 Типовые ошибки

- держать flat network;
- использовать shared admin credentials на всех узлах;
- не отделять management plane;
- не знать межхостовую связность.

---

## 6.7 Чеклист главы

- [ ] У меня есть карта межхостового доверия
- [ ] Shared credentials и лишняя связность сокращены
- [ ] Management plane выделен и наблюдаем
- [ ] Blast radius одной компрометации ограничен
