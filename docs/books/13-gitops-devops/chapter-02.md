# Глава 2: GitOps идея

---

## 2.1 Push vs Pull

```
Push:                        Pull (GitOps):
CI → SSH → кластер           Git repo ← ArgoCD → кластер
CI нужны credentials         CI не нужен доступ к кластеру
                              ArgoCD синхронизирует сам
```

---

## 2.2 Два репозитория

```
app-code/          → код, Dockerfile, тесты
app-infra/         → Helm values, K8s manifests
```

CI обновляет app-infra → ArgoCD видит → синхронизирует.

---

## 2.3 Что значит Self-Heal

Кто-то вручную изменил реплики в кластере:

```bash
kubectl scale deployment myapp --replicas=1
```

Git говорит: `replicas = 2`
Кластер показывает: `replicas = 1`

ArgoCD видит расхождение и при `selfHeal: true` возвращает состояние к тому что лежит в Git.

Без `selfHeal` ручные изменения накапливаются, и Git перестаёт быть источником истины.

---

## 2.4 Два репозитория — почему

```
app-code/    → разработчики коммитят код
app-infra/   → CI обновляет image tag автоматически
```

Если всё держать в одном репозитории, CI начнёт коммитить туда же куда пушат разработчики. Это создаёт лишний шум в истории и может запускать ненужные циклы pipeline.

## 📝 Упражнения

### Упражнение 2.1: Понять auto-heal
1. Создай `Application` с `autoSync + selfHeal`
2. Выполни `kubectl scale deployment myapp --replicas=0`
3. Подожди около 30 секунд
4. Проверяй `kubectl get deploy myapp`
5. Убедись что ArgoCD восстановил нужное число реплик

### Упражнение 2.2: Два репозитория
1. Раздели проект на `app-code` и `app-infra`
2. В одном держи код, в другом манифесты
3. Объясни себе какой из них меняет разработчик, а какой меняет CI
4. Убедись что ArgoCD читает только infra-репозиторий
