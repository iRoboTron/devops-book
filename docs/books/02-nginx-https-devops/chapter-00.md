## 0.1 Что мы построим за эту книгу

В Модуле 1 ты научился работать с Linux изнутри.
В этом модуле ты научишься **выставлять сервисы наружу** — в интернет.

К концу книги у тебя будет:

```
Интернет (браузер пользователя)
    │  HTTPS (порт 443)
    ▼
┌──────────────────────────────┐
│         Nginx                │
│  Reverse Proxy + SSL         │
└──────────────┬───────────────┘
               │  HTTP (порт 8000, только локально)
               ▼
┌──────────────────────────────┐
│   Python-приложение          │
│   (Flask / FastAPI / Django) │
│   systemd сервис             │
└──────────────┬───────────────┘
               │
               ▼
        /var/log/myapp/ ← логи

[ ufw ] — фаервол: открыты только 22, 80, 443
[ certbot ] — SSL-сертификат Let's Encrypt, автопродление
[ DNS ] — domain.ru → IP твоего сервера
```

**Каждая глава — один кирпич этой архитектуры:**

| Глава | Что добавляешь |

| 1 | Понимаешь HTTP/HTTPS |

| 3 | Ставишь Nginx, поднимаешь статику |
| 4 | Подключаешь Nginx к Python-приложению |

## 0.2 Как это работает: модель мышления

### Ты уже знаешь это из Модуля 1

```
Ты → терминал → команда → Linux выполняет
```

Это **одностороннее** общение. Ты просишь — Linux делает.

### Теперь добавляем внешнюю сторону

```
Браузер (у пользователя) → запрос → Твой сервер → ответ → Браузер
```

Это **двустороннее** общение. Сервер **ждёт** запросы и **отвечает** на них.

### Поток запроса: шаг за шагом

```
Пользователь вводит: https://myapp.ru

1. Браузер спрашивает DNS: "Какой IP у myapp.ru?"
   → DNS отвечает: 203.0.113.50

2. Браузер подключается к 203.0.113.50:443
   → Проходит через ufw (фаервол)
   → Попадает в Nginx

3. Nginx расшифровывает HTTPS (TLS)
   → Видит что это HTTP-запрос GET /
   → Пересылает Python-приложению на 127.0.0.1:8000

4. Python-приложение обрабатывает
   → Возвращает HTML-страницу
   → Nginx получает ответ

5. Nginx шифрует ответ (TLS)
   → Отправляет браузеру

6. Браузер расшифровывает
   → Показывает страницу пользователю
```

> **Запомни:** Каждый шаг — отдельный компонент.
> Когда что-то не работает — ты проверяешь каждый шаг по очереди.
> Именно этому научит Глава 7.

## 0.3 Требования к окружению

### Что тебе нужно

| Требование | Минимум | Комфорт |
|------------|---------|---------|
| Ubuntu Server | 22.04 LTS | 24.04 LTS |
| RAM | 2 ГБ | 4 ГБ |
| Диск | 25 ГБ | 50 ГБ |
| Интернет | NAT или мост | Публичный IP (для реального домена) |
| Права | sudo | sudo |

### Где практиковаться

**Вариант 1: Виртуалка дома** (для обучения)
- VirtualBox + Ubuntu Server
- Доступ только из твоей сети
- Домен через `/etc/hosts`
- ✅ Достаточно для 90% упражнений

**Вариант 2: VPS в облаке** (для реального сервиса)
- DigitalOcean, Hetzner, Timeweb, Aeza
- Публичный IP
- Настоящий домен
- ✅ Для полноценного развёртывания

> **Совет:** Для обучения хватит виртуалки.
> Когда всё заработает — перенесёшь на реальный сервер.

## 0.4 Практика без реального домена

У тебя может не быть домена и публичного IP. Это нормально.

### `/etc/hosts` — локальный DNS

```bash
sudo nano /etc/hosts
```

Добавь:
```
127.0.0.1   myapp.local
127.0.0.1   api.myapp.local
```

Теперь `myapp.local` будет указывать на твой сервер.

```bash
curl http://myapp.local
```

> **Запомни:** `/etc/hosts` работает только на этой машине.
> Другие компьютеры не знают про `myapp.local`.
> Для практики — идеально.

### Self-signed сертификат для HTTPS

Для практики без certbot — создашь самоподписанный сертификат:

```bash
sudo openssl req -x509 -nodes -days 365 \
  -newkey rsa:2048 \
  -keyout /etc/ssl/private/myapp.key \
  -out /etc/ssl/certs/myapp.crt \
  -subj "/CN=myapp.local"
```

Браузер будет ругаться что сертификат "недоверенный". Это нормально — для тестов сойдёт.

> **Запомни:** Self-signed ≠ для продакшена.
> Никогда не используй его на реальном сервере с пользователями.

### Проверка готовности

```bash
# Всё ли установлено
which curl && echo "✅ curl" || echo "❌ curl"
which dig && echo "✅ dig" || echo "❌ dig"
which openssl && echo "✅ openssl" || echo "❌ openssl"

# Если чего-то нет
sudo apt update
sudo apt install -y curl dnsutils openssl
```

## 0.5 Как смотреть на сеть: пакеты, запросы, ответы

### Клиент-серверная модель

```
Клиент (браузер)          Сервер (твой)
     │                         │
     │─── HTTP GET / ──────────│→
     │                         │ (обрабатывает)
     │←── HTTP 200 + HTML ─────│
     │                         │
```

Одна строка запроса — один ответ. Это **HTTP**.

### Что такое порт

```
203.0.113.50:443
└──────────┘ └┘
     IP      Порт
```

| Порт | Что слушает |
|------|------------|
| 22 | SSH |
| 80 | HTTP (Nginx) |
| 443 | HTTPS (Nginx) |
| 8000 | Python-приложение |

> **Запомни:** IP = какой компьютер. Порт = какая программа.
> Один сервер может обслужив сотни сайтов — каждый на своём "виртуальном" порту.

### Что такое пакет

Данные не летят целиком. Они разбиваются на кусочки — **пакеты**.

```
"Hello World!" → [пакет 1] [пакет 2] [пакет 3] → собираются обратно
```

Тебе не нужно думать о пакетах каждый день. Но когда что-то ломается — ты будешь смотреть на них через `tcpdump` (Глава 7).

## 0.6 Чем эта книга отличается от Модуля 1

В Модуле 1 ты работал **локально**:

```
Ты → команда → результат
```

Здесь ты работаешь **с сетью**:

```
Клиент → запрос → сервер → ответ → клиент
```

**Ключевые отличия:**

| Модуль 1 | Модуль 2 |
|----------|----------|
| Одна машина | Клиент + сервер |
| Ты контролируешь всё | Клиент вне твоего контроля |
| Ошибка = сообщение в терминале | Ошибка = пользователь видит "сайт не работает" |
| `curl` = скачать файл | `curl` = диагностический прибор |
| Права = chmod | Права = фаервол + SSL + proxy |

> **Запомни:** В сетевом мире ты не видишь ошибку клиента напрямую.
> Ты смотришь логи и делаешь выводы.

## 0.7 Безопасный порядок действий

В этой книге есть операции которые могут **отрезать тебя от сервера**.

### Золотые правила

> **Порядок важен:**
> 1. Сначала `ufw allow 22` — **потом** `ufw enable`
> 2. Сначала `nginx -t` — **потом** `systemctl reload nginx`
> 3. Сначала `certbot renew --dry-run` — **потом** реальное продление

### Если что-то пошло не так

| Проблема | Решение |
|----------|---------|
| Закрыл SSH через ufw | Перезагрузи сервер (через VNC/консоль провайдера) |
| Сломал конфиг Nginx | `sudo nginx -t` покажет ошибку |
| certbot не работает | Проверь что порт 80 открыт и DNS настроен |
| Сайт не открывается | Иди по алгоритму из Главы 7 |
