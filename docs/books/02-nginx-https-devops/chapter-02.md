# Глава 2: DNS — от домена до IP

> **Запомни:** DNS — это телефонная книга интернета. Без неё тебе пришлось бы запоминать IP-адреса. Когда "сайт не открывается" — часто виноват DNS.

---

## 2.1 DNS как телефонная книга

У сервера есть IP: `203.0.113.50`
У тебя есть домен: `myapp.ru`

**DNS** (Domain Name System) соединяет их:

```
myapp.ru  ──DNS──►  203.0.113.50
```

Браузер не понимает `myapp.ru`. Он понимает только IP.
DNS — переводчик между человеческими именами и компьютерными адресами.

---

## 2.2 Что происходит при DNS-запросе: полная цепочка

Ты вводишь `myapp.ru` в браузере. Вот что происходит:

```
Браузер
  │
  ▼
1. Кэш браузера: "Я недавно запрашивал myapp.ru?"
   ├── Да → использую кеш (мгновенно)
   └── Нет → дальше
  │
  ▼
2. /etc/hosts: "Есть запись для myapp.ru?"
   ├── Да → использую её
   └── Нет → дальше
  │
  ▼
3. Системный резолвер (ISP): "192.168.1.1, какой IP у myapp.ru?"
   ├── Есть в кеше → отвечает
   └── Нет → дальше
  │
  ▼
4. Root сервер (13 штук по миру): "Я не знаю IP myapp.ru,
   но знаю где .ru — иди туда: 193.0.0.1"
  │
  ▼
5. TLD сервер (.ru): "Я не знаю myapp.ru,
   но знаю авторитативный сервер — иди туда: ns1.registrar.ru"
  │
  ▼
6. Авторитативный сервер: "myapp.ru = 203.0.113.50"
  │
  ▼
7. Ответ идёт обратно: браузер → кэш → IP найден!
```

> **Запомни:** Первый запрос медленный (все 7 шагов).
> Последующие — мгновенные (кэш).
> Когда меняешь DNS — нужно ждать пока кеш обновится (TTL).

### Визуально

```
Ты: "myapp.ru?"
  │
  ├── Браузер кэш? ❌
  ├── /etc/hosts? ❌
  ├── ISP резолвер? ❌
  │
  ├── Root: ".ru? → 193.0.0.1"
  │     └── TLD .ru: "myapp.ru? → ns1.registrar.ru"
  │           └── Авторитативный: "myapp.ru = 203.0.113.50"
  │
  └── Ответ: 203.0.113.50
```

## 2.3 Типы DNS-записей

Один домен может иметь **несколько** записей разных типов.

| Тип | Что хранит | Пример |
|-----|-----------|--------|
| **A** | IPv4 адрес | `myapp.ru → 203.0.113.50` |
| **AAAA** | IPv6 адрес | `myapp.ru → 2001:db8::1` |
| **CNAME** | Алиас (другое имя) | `www.myapp.ru → myapp.ru` |
| **MX** | Почтовый сервер | `myapp.ru → mail.myapp.ru` |
| **TXT** | Произвольный текст | SPF, DKIM, верификация |
| **NS** | Авторитативный сервер | `myapp.ru → ns1.registrar.ru` |

### A-запись (самая важная для тебя)

```
myapp.ru  →  A  →  203.0.113.50
```

Это то что связывает твой домен с сервером.

### CNAME — псевдоним

```
www.myapp.ru  →  CNAME  →  myapp.ru
```

`www` ведёт туда же куда основной домен.

### MX — почта

```
myapp.ru  →  MX  →  mail.yandex.ru  (приоритет 10)
```

Когда кто-то шлёт письмо на `user@myapp.ru` — почтовый сервер спрашивает MX-запись.

### TXT — проверки и политики

```
myapp.ru  →  TXT  →  "v=spf1 include:_spf.google.com ~all"
```

Используется для:
- SPF (кто может слать почту от твоего домена)
- DKIM (подпись писем)
- Верификация (подтвердить что домен твой)

> **Запомни:** Для веб-сервера тебе нужна только **A-запись**.
> Остальные — для почты и специальных задач.

## 2.4 `dig` — диагностика DNS

В Модуле 1 ты знал `nslookup`. Забудь его.
`dig` — профессиональный инструмент.

### Базовый запрос

```bash
dig google.com
```

Вывод большой. Вот что важно:

```
;; ANSWER SECTION:
google.com.    300    IN    A    142.250.189.14
```

| Часть | Значение |
|-------|----------|
| `google.com.` | Домен |
| `300` | TTL (секунд) — кеш живёт 5 минут |
| `IN` | Internet |
| `A` | Тип записи |
| `142.250.189.14` | IP-адрес |

### Запрос конкретного типа

```bash
dig google.com A       # IPv4 (по умолчанию)
dig google.com AAAA    # IPv6
dig google.com MX      # Почтовые серверы
dig google.com TXT     # Текстовые записи
dig google.com NS      # Авторитативные серверы
```

### Спросить конкретный DNS-сервер

```bash
dig @8.8.8.8 google.com     # Google DNS
dig @1.1.1.1 google.com     # Cloudflare DNS
dig @ns1.registrar.ru myapp.ru  # Авторитативный сервер
```

> **Зачем:** Если DNS у пользователя не работает — проверь через другой резолвер.
> Так ты поймёшь: проблема у провайдера или глобальная.

### Краткий вывод (+short)

```bash
dig google.com +short
142.250.189.14
```

Только IP, без мусора. Удобно для скриптов.

### Полный трейс (+trace)

```bash
dig myapp.ru +trace
```

Покажет **всю цепочку** от root до авторитативного сервера.
Как в разделе 2.2 — шаг за шагом.

> **Совет:** `+trace` — когда DNS не работает и ты не понимаешь почему.
> Покажет на каком шаге цепочка ломается.

## 2.5 `/etc/hosts` — локальный DNS

В Главе 0 ты уже трогал `/etc/hosts`. Теперь разберём подробнее.

### Приоритет

```
/etc/hosts  >  DNS
```

Если запись есть в `/etc/hosts` — система **не спрашивает** DNS.

### Зачем это DevOps'у

| Ситуация | Зачем |
|----------|-------|
| Локальная разработка | `127.0.0.1 myapp.local` |
| Тестирование домена | `203.0.113.50 myapp.ru` (пока DNS не настроен) |
| Блокировка сайтов | `0.0.0.0 ads.example.com` |
| Миграция сервера | Временно направить домен на новый IP |

### Пример: тестирование домена

```bash
sudo nano /etc/hosts
```

Добавь:
```
203.0.113.50   myapp.ru   www.myapp.ru
```

Теперь `myapp.ru` идёт на твой сервер, даже если DNS ещё не настроен.

> **Запомни:** `/etc/hosts` — только на этой машине.
> Другие пользователи `myapp.ru` попадут на старый IP.
> Для продакшена нужен настоящий DNS.

## 2.6 TTL — почему DNS-изменения применяются не сразу

**TTL** (Time To Live) — сколько секунд кеш живёт.

```
myapp.ru  →  A  →  203.0.113.50   TTL: 3600
                                             ↑
                                    Кеш живёт 1 час
```

### Сценарий

```
10:00  Ты поменял A-запись: 203.0.113.50 → 203.0.113.99
10:01  Пользователь заходит на сайт
       Его резолвер: "У меня в кеше 203.0.113.50, TTL не истёк"
       Пользователь идёт на СТАРЫЙ сервер

11:01  TTL истёк (3600 секунд)
       Резолвер спрашивает снова
       Получает новый IP: 203.0.113.99
```

> **Запомни:** DNS-изменения — не мгновенные.
> TTL 3600 = до 1 часа пока все увидят новый IP.
> TTL 300 = 5 минут (быстрее, но больше нагрузки на DNS).

### Практическое правило

| TTL | Когда использовать |
|-----|-------------------|
| 86400 (24ч) | Стабильный сайт, редкие изменения |
| 3600 (1ч) | Стандартное значение |
| 300 (5м) | Перед миграцией (чтобы быстрее обновилось) |

> **Совет:** Перед миграцией сервера:
> 1. За день до миграции поставь TTL 300
> 2. Сделай миграцию
> 3. Через день верни TTL 3600

## 2.7 Покупка домена и настройка A-записи

### Где купить домен

| Регистратор | Для .ru | Для .com |
|-------------|---------|----------|
| Reg.ru | ✅ | ✅ |
| Beget | ✅ | ✅ |
| Namecheap | ❌ | ✅ |
| Cloudflare | ❌ | ✅ (по себестоимости) |

### Процесс покупки

1. Выбери доменное имя
2. Проверь что свободно
3. Оплати (обычно 200-2000₽/год)
4. Получи доступ к панели управления DNS

### Настройка A-записи

В панели управления доменом:

```
Тип: A
Имя: @ (или пусто — это основной домен)
Значение: 203.0.113.50 (IP твоего сервера)
TTL: 3600
```

Для `www`:

```
Тип: CNAME
Имя: www
Значение: myapp.ru
TTL: 3600
```

### Проверка

```bash
# Подожди 5-15 минут
dig myapp.ru +short
203.0.113.50
```

Если видишь свой IP — работает!

> **Запомни:** DNS не работает мгновенно.
> Подожди минимум 5 минут, иногда до часа.
> Проверяй через `dig` — не через браузер.

### Упражнение 2.5: DevOps Think
**Задача:** «Ты сменил IP сервера. Обновил A-запись. Пользователи всё ещё попадают на старый сервер. Почему?»

Подсказки:
1. Что такое TTL?
2. Сколько TTL стоит по умолчанию?
3. Где кешируется DNS? (браузер, ОС, ISP резолвер)
4. Как ускорить обновление для всех?
5. Как проверить что DNS обновился?

Ответ:
- DNS кешируется на всех уровнях
- Если TTL = 3600 — до часа ждать
- Решение: заранее уменьшить TTL до миграции
- Проверка: `dig myapp.ru +short` через разные резолверы
