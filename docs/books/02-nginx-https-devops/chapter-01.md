# Глава 1: HTTP/HTTPS изнутри

> **Запомни:** В Модуле 1 `curl` был инструментом "скачать файл". Здесь он становится **диагностическим прибором**. Ты научишься видеть что происходит между браузером и сервером.

---

## 1.1 Что происходит когда ты открываешь сайт

Ты вводишь `https://google.com` и нажимаешь Enter.

За доли секунды происходит:

```
1. Браузер → DNS: "Какой IP у google.com?"
             DNS → "142.250.189.14"

2. Браузер → 142.250.189.14:443
             "Привет, я хочу GET /"

3. Сервер → браузер
             "Вот HTML-страница (200 OK)"

4. Браузер рисует страницу
```

> **Запомни:** Вся веба — это запросы и ответы.
> Ничего магического. Один компьютер просит, другой отвечает.

---

## 1.2 HTTP-запрос: что внутри

HTTP-запрос — это текст. Обычный текст который отправляет браузер.

```http
GET /about HTTP/1.1
Host: myapp.ru
User-Agent: Mozilla/5.0
Accept: text/html
```

### Разбор по частям

```
GET /about HTTP/1.1
└┬─┘ └──┬──┘ └───┬────┘
 │       │        │
 │       │        Версия HTTP
 │       Путь (URL после домена)
 Метод (что хотим сделать)
```

### Методы HTTP

| Метод | Что значит | Пример |
|-------|-----------|--------|
| `GET` | Получить данные | Открыть страницу |
| `POST` | Отправить данные | Форма логина |
| `PUT` | Обновить данные | Изменить профиль |
| `DELETE` | Удалить данные | Удалить аккаунт |
| `PATCH` | Частично обновить | Изменить email |

> **Запомни:** 95% времени ты работаешь с `GET` и `POST`.
> Остальные — для API.

### Заголовки (Headers)

После первой строки идут **заголовки** — дополнительная информация:

```http
GET /api/users HTTP/1.1
Host: myapp.ru
User-Agent: Mozilla/5.0
Accept: application/json
Authorization: Bearer abc123...
```

| Заголовок | Зачем |
|-----------|-------|
| `Host` | Какой сайт хочешь (на одном IP их может быть много) |
| `User-Agent` | Кто ты (браузер, бот, скрипт) |
| `Accept` | В каком формате хочешь ответ |
| `Authorization` | Кто ты (логин/токен) |
| `Content-Type` | В каком формате отправляешь данные |
| `Cookie` | Твои куки (сессия, предпочтения) |

> **Запомни:** Заголовки — это метаданные. Не сами данные, а информация о запросе.
> Как конверт: не письмо, а информация для почты.

### Тело запроса (Body)

Для `POST` и `PUT` после заголовков идёт **тело** — данные которые отправляешь:

```http
POST /api/users HTTP/1.1
Host: myapp.ru
Content-Type: application/json

{"name": "Ivan", "email": "ivan@example.com"}
```

Пустая строка между заголовками и телом — **обязательна**.

## 1.3 HTTP-ответ: что приходит обратно

Сервер отвечает тоже текстом:

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1234
Server: nginx/1.24.0

<!DOCTYPE html>
<html>
<head><title>My App</title></head>
...
```

### Первая строка ответа

```
HTTP/1.1 200 OK
         └─┬─┘ └┬┘
           │    │
           │    Текстовое описание
           Статус-код (число)
```

### Статус-коды которые нужно знать

| Код | Значение | Что значит |
|-----|----------|-----------|
| **200** | OK | Всё хорошо, вот данные |
| **201** | Created | Ресурс создан |
| **301** | Moved Permanently | Страница переехала навсегда |
| **302** | Found | Временный редирект |
| **400** | Bad Request | Ты отправил чушь |
| **401** | Unauthorized | Не залогинен |
| **403** | Forbidden | Залогинен, но нет прав |
| **404** | Not Found | Страницы не существует |
| **500** | Internal Server Error | Сервер сломался |
| **502** | Bad Gateway | Прокси не получил ответ от приложения |
| **503** | Service Unavailable | Сервис временно недоступен |
| **504** | Gateway Timeout | Прокси ждал, приложение не ответил |

### Группы статус-кодов

| Первая цифра | Группа | Значение |
|-------------|--------|----------|
| `2xx` | Успех | Запрос выполнен |
| `3xx` | Редирект | Иди по другому адресу |
| `4xx` | Ошибка клиента | Ты сделал что-то не так |
| `5xx` | Ошибка сервера | Сервер сломался |

> **Запомни:**
> - 4xx = проблема в запросе (ты виноват)
> - 5xx = проблема на сервере (сервер виноват)

## 1.4 HTTP vs HTTPS: в чём разница

### HTTP (без шифрования)

```
Браузер ───→ "GET /login" ───→ Сервер
             "password=12345"
```

**Видно всем** кто стоит посередине:
- Твой провайдер
- Админ Wi-Fi в кафе
- Любой кто перехватит пакеты

### HTTPS (с шифрованием)

```
Браузер ───→ 🔒 зашифровано 🔒 ───→ Сервер
             (никто посередине не видит)
```

**Видно только:**
- Браузер (отправитель)
- Сервер (получатель)

> **Запомни:** HTTPS шифрует **данные** (тело, заголовки, URL).
> Не шифрует: IP-адрес сервера, факт что ты подключился.

### Что такое TLS/SSL

**TLS** (Transport Layer Security) — протокол шифрования.
**SSL** — старое название. Сейчас все говорят SSL, но имеют в виду TLS.

Как работает:

```
1. Браузер: "Привет, давай общаться безопасно"
2. Сервер:  "Вот мой сертификат (публичный ключ)"
3. Браузер: "Ок, сертификат валидный. Шифруем"
4. Оба:     обмениваются зашифрованными данными
```

### Что даёт сертификат

Сертификат подтверждает **личность сервера**:

```
Браузер: "Ты действительно myapp.ru? А не хакер?"
Сертификат: "Да, вот подтверждение от Let's Encrypt (CA)"
```

> **Запомни:** TLS делает две вещи:
> 1. **Шифрует** данные (никто посередине не видит)
> 2. **Подтверждает** личность сервера (ты общаешься с тем с кем думаешь)

## 1.5 `curl` как диагностический прибор

В Модуле 1 ты использовал `curl` чтобы скачать файл.
Теперь `curl` — твой **главный инструмент отладки**.

### `-I` — только заголовки

```bash
curl -I https://google.com
HTTP/2 200
content-type: text/html; charset=UTF-8
server: gws
```

`-I` = HEAD запрос. Не скачивает тело, только заголовки.

> **Зачем:** Быстро проверить статус-код и сервер.
> Не качая всю страницу.

### `-v` — verbose (подробно)

```bash
curl -v https://google.com
* Connected to google.com (142.250.189.14) port 443
> GET / HTTP/2
> Host: google.com
> User-Agent: curl/8.5.0
> Accept: */*
>
< HTTP/2 200
< content-type: text/html
< ...
```

`-v` показывает:
- `>` — что ты отправляешь (запрос)
- `<` — что получаешь (ответ)
- `*` — техническая информация (соединение, TLS)

> **Запомни:** `-v` — первая команда когда "сайт не работает".
> Ты видишь весь разговор между клиентом и сервером.

### `-X` — метод запроса

```bash
curl -X POST https://api.myapp.ru/users
curl -X DELETE https://api.myapp.ru/users/42
curl -X PUT https://api.myapp.ru/users/42
```

### `-d` — тело запроса

```bash
curl -X POST https://api.myapp.ru/users \
  -d '{"name":"Ivan","email":"ivan@example.com"}'
```

### `-H` — добавить заголовок

```bash
curl -H "Authorization: Bearer abc123" \
  https://api.myapp.ru/users
```

### Комбо: POST + JSON + заголовки

```bash
curl -X POST https://api.myapp.ru/users \
  -H "Content-Type: application/json" \
  -d '{"name":"Ivan","email":"ivan@example.com"}'
```

### `-o` — сохранить в файл

```bash
curl -o page.html https://myapp.ru
```

### `--silent` — без прогресс-бара

```bash
curl -s https://api.myapp.ru/health
{"status":"ok"}
```

## 1.6 Диагностика через curl: реальные сценарии

### Сценарий 1: Проверить что сервер отвечает

```bash
curl -I http://127.0.0.1:8000
HTTP/1.1 200 OK
Server: gunicorn
```

✅ Сервис запущен и работает.

### Сценарий 2: Проверить редирект

```bash
curl -I http://myapp.ru
HTTP/1.1 301 Moved Permanently
Location: https://myapp.ru
```

✅ HTTP редиректит на HTTPS — правильно.

### Сценарий 3: Проверить время ответа

```bash
curl -w "DNS: %{time_namelookup}s\nConnect: %{time_connect}s\nTTFB: %{time_starttransfer}s\nTotal: %{time_total}s\n" \
  -o /dev/null -s https://myapp.ru

DNS: 0.023s
Connect: 0.045s
TTFB: 0.120s
Total: 0.156s
```

| Метрика | Значение |
|---------|----------|
| DNS | Сколько искал IP |
| Connect | Сколько подключался |
| TTFB | Time To First Byte — сколько сервер думал |
| Total | Общее время |

> **Совет:** TTFB > 1 секунды = сервер тормозит.
> TTFB < 0.2 = нормально.

### Сценарий 4: Посмотреть цепочку редиректов

```bash
curl -L -v http://myapp.ru 2>&1 | grep "< HTTP"
< HTTP/1.1 301 Moved Permanently
< HTTP/1.1 301 Moved Permanently
< HTTP/1.1 200 OK
```

Три редиректа прежде чем получил ответ. Может быть проблема.

### Сценарий 5: Проверить SSL-сертификат

```bash
curl -vI https://myapp.ru 2>&1 | grep -E "SSL|subject|expire"
*  SSL certificate verify ok.
*  subject: CN=myapp.ru
*  expire date: Jul 15 12:00:00 2026 GMT
```

## 1.7 Разница между curl и браузером

| curl | Браузер |
|------|---------|
| Один запрос | Множество запросов (CSS, JS, картинки) |
| Сырой текст | Красивая страница |
| Ты контролируешь всё | Браузер добавляет заголовки сам |
| Без JavaScript | Выполняет JavaScript |
| Без кеша | Кеширует ресурсы |

> **Запомни:** curl показывает **голый** HTTP.
> Браузер делает сотни запросов за кулисами.
> Когда отлаживаешь проблему — curl даёт чистую картину.

**Задача:** «Пользователь жалуется что сайт долго грузится. Диагностируй»

2. Какой TTFB? (больше 1 секунды = проблема)
3. Какой статус-код? (5xx = сервер, 4xx = клиент)
4. Проверь заголовок `Server:` — что отвечает?
5. Если TTFB большой — проблема в приложении, не в сети
