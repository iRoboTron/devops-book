# Глава 7: Диагностика сетевых проблем

> **Запомни:** Когда "сайт не работает" — паника не помогает. Нужен алгоритм. Эта глава — набор алгоритмов для каждого типа проблем.

---

## 7.1 Алгоритм диагностики: от простого к сложному

Когда что-то не работает — не тыкай случайные команды.
Иди по шагам: от общих вопросов к конкретным.

### Базовый алгоритм

```
1. Сервер вообще доступен?
   → ping server-ip

2. Порт открыт?
   → nc -zv server-ip 80
   → ss -tlnp | grep :80
   → ufw status

3. Сервис запущен?
   → systemctl status nginx
   → systemctl status myapp

4. Конфиг правильный?
   → nginx -t

5. Что отвечает сервис?
   → curl -v http://localhost
   → curl -v https://myapp.ru

6. Что в логах?
   → journalctl -u nginx -n 50
   → tail /var/log/nginx/error.log
   → journalctl -u myapp -n 50
```

### Визуально

```
ping? ──❌── Сетевая проблема
 │
 ✅
nc порт? ──❌── Порт закрыт (ufw / сервис не запущен)
 │
 ✅
systemctl? ──❌── Сервис не запущен / упал
 │
 ✅
nginx -t? ──❌── Сломан конфиг
 │
 ✅
curl? ──❌── 5xx → смотри логи приложения
 │         ──❌── 4xx → проблема в запросе
 │
 ✅
Всё работает!
```

> **Запомни:** 80% проблем решаются на шагах 2-3.
> Порт закрыт или сервис не запущен.
> Не лезь в логи пока не проверил basics.

## 7.2 `ss` — кто слушает порт

```bash
ss -tlnp
State   Recv-Q  Send-Q   Local Address:Port    Peer Address:Port  Process
LISTEN  0       128            0.0.0.0:22           0.0.0.0:*      users:(("sshd",pid=234,fd=3))
LISTEN  0       511            0.0.0.0:80           0.0.0.0:*      users:(("nginx",pid=567,fd=6))
LISTEN  0       511          127.0.0.1:8000         0.0.0.0:*      users:(("gunicorn",pid=890,fd=5))
```

| Опция | Значение |
|-------|----------|
| `-t` | TCP |
| `-l` | Listening (слушают) |
| `-n` | Числа (не имена сервисов) |
| `-p` | Процесс (кто слушает) |

### Разбор

| Адрес | Значение |
|-------|----------|
| `0.0.0.0:22` | SSH доступен отовсюду |
| `0.0.0.0:80` | Nginx доступен отовсюду |
| `127.0.0.1:8000` | Python только с localhost (правильно!) |

### Быстрая проверка

```bash
ss -tlnp | grep :8000
LISTEN  0  511  127.0.0.1:8000  0.0.0.0:*  users:(("gunicorn",pid=890))
```

Если пусто — ничего не слушает порт 8000. Приложение не запущено.

## 7.3 `nc` — проверка открытости порта

```bash
nc -zv server-ip 80
Connection to server-ip 80 port [tcp/http] succeeded!
```

| Опция | Значение |
|-------|----------|
| `-z` | Scan mode (не отправлять данные) |
| `-v` | Verbose (показать результат) |

### Проверить несколько портов

```bash
nc -zv server-ip 22 80 443 8000
```

### Проверить диапазон

```bash
nc -zv server-ip 1-1024
```

> **Совет:** `nc -zv` быстрее чем `curl` когда нужно просто проверить порт.
> curl делает HTTP-запрос. nc просто проверяет что порт открыт.

## 7.4 `tcpdump` — смотреть пакеты

`tcpdump` показывает **сырые пакеты** на сетевом интерфейсе.

### Базовое использование

```bash
sudo tcpdump -i eth0
```

Показывает все пакеты. Много текста. Неудобно.

### Фильтр по порту

```bash
sudo tcpdump -i eth0 port 80
```

Только HTTP-трафик.

### Фильтр по IP

```bash
sudo tcpdump -i eth0 host 203.0.113.50
```

Только трафик с/на этот IP.

### Комбинация

```bash
sudo tcpdump -i eth0 port 443 and host 203.0.113.50
```

### Смотреть SYN/RST (подключения)

```bash
sudo tcpdump -i eth0 'tcp[tcpflags] & (tcp-syn|tcp-rst) != 0'
```

Покажет только попытки подключения (SYN) и отказы (RST).

> **Запомни:** `tcpdump` — инструмент последнего средства.
> Используй когда `ss`, `nc`, `curl` не дали ответа.
> Для выхода — `Ctrl+C`.

### Что видишь

```
SYN → серверу: кто-то пытается подключиться
SYN-ACK ← от сервера: сервер принял
ACK → серверу: соединение установлено

RST → серверу: соединение разорвано (сервис не запущен)
```

## 7.5 Типичные ошибки и их причины

### 502 Bad Gateway

```
curl -I http://myapp.ru
HTTP/1.1 502 Bad Gateway
```

**Что значит:** Nginx получил запрос, передал Python'у, но Python вернул что-то непонятное или не ответил.

**Причины:**

| Причина | Как проверить | Решение |
|---------|--------------|---------|
| Python не запущен | `systemctl status myapp` | `sudo systemctl start myapp` |
| Python на другом порту | `ss -tlnp \| grep 8000` | Проверь `proxy_pass` |
| Python упал с ошибкой | `journalctl -u myapp -n 50` | Чини приложение |
| proxy_pass неправильный | `grep proxy_pass /etc/nginx/sites-enabled/` | Исправь конфиг |

### 504 Gateway Timeout

```
curl -I http://myapp.ru
HTTP/1.1 504 Gateway Timeout
```

**Что значит:** Nginx ждал ответ от Python, но не дождался за отведённое время.

**Причины:**

| Причина | Как проверить | Решение |
|---------|--------------|---------|
| Приложение зависло | `ps aux \| grep python` (CPU 100%?) | Рестарт, оптимизация |
| Долгий запрос к БД | Логи приложения | Оптимизируй запрос |
| Таймаут маленький | `grep proxy_read_timeout /etc/nginx/...` | Увеличь таймаут |

### ERR_SSL_PROTOCOL_ERROR (в браузере)

**Что значит:** Браузер пытается установить HTTPS соединение, но сервер не понимает TLS.

**Причины:**

| Причина | Решение |
|---------|---------|
| Слушаешь HTTP на порту 443 | Настрой SSL в Nginx |
| Сертификат не настроен | `sudo certbot --nginx` |
| Порт 443 закрыт | `sudo ufw allow 443` |

### Connection refused

```
curl http://myapp.ru
curl: (7) Failed to connect to myapp.ru port 80: Connection refused
```

**Что значит:** Никто не слушает этот порт.

**Причины:**

| Причина | Решение |
|---------|---------|
| Nginx не запущен | `sudo systemctl start nginx` |
| Nginx слушает другой порт | `ss -tlnp \| grep nginx` |
| server_name неправильный | Проверь конфиг Nginx |

### Connection timeout

```
curl http://myapp.ru
curl: (28) Connection timed out
```

**Что значит:** Сервер не отвечает вообще. Пакеты теряются.

**Причины:**

| Причина | Решение |
|---------|---------|
| Фаервол блокирует | `sudo ufw status` |
| Сервер выключен | Проверь доступность |
| Неправильный IP | `dig myapp.ru +short` |

### No route to host

**Что значит:** Сетевая проблема. Маршрут до сервера неизвестен.

**Причины:**

| Причина | Решение |
|---------|---------|
| Неправильный IP | Проверь DNS |
| Сервер в другой сети | Проверь маршрутизацию |

## 7.6 Диагностический чеклист

Распечатай и держи под рукой:

```
□ 1. ping server-ip          — сервер доступен?
□ 2. nc -zv server-ip 80     — порт 80 открыт?
□ 3. nc -zv server-ip 443    — порт 443 открыт?
□ 4. ss -tlnp | grep :80     — кто слушает 80?
□ 5. systemctl status nginx  — Nginx запущен?
□ 6. systemctl status myapp  — приложение запущено?
□ 7. nginx -t                — конфиг валидный?
□ 8. curl -v http://localhost — Nginx отвечает?
□ 9. curl -v http://127.0.0.1:8000 — приложение отвечает?
□ 10. tail /var/log/nginx/error.log — ошибки Nginx?
□ 11. journalctl -u myapp -n 50 — ошибки приложения?
□ 12. ufw status              — фаервол не блокирует?
□ 13. dig myapp.ru +short     — DNS правильный?
```

> **Совет:** Когда всё сломалось — иди по списку сверху вниз.
> 90% проблем находятся в первых 5 пунктах.

## 📝 Упражнения

### Упражнение 7.1: Диагностика по шагам
**Задача:**
1. Проверь доступность сервера: `ping 127.0.0.1`
2. Проверь порт 80: `nc -zv 127.0.0.1 80`
3. Кто слушает: `ss -tlnp | grep :80`
4. Статус Nginx: `systemctl status nginx`
5. Проверь конфиг: `sudo nginx -t`
6. curl: `curl -v http://localhost`
7. Пройди все 7 шагов — всё ли в порядке?

### Упражнение 7.2: Сломанный стек
**Задача:** Намеренно сломай и диагностируй.
1. Останови Nginx: `sudo systemctl stop nginx`
2. Попробуй: `curl -I http://localhost` — что?
3. Иди по алгоритму:
   - ping? ✅
   - nc порт 80? ❌
   - systemctl status nginx? ❌ inactive
4. Почини: `sudo systemctl start nginx`
5. Проверь: `curl -I http://localhost` — ✅

### Упражнение 7.3: 502 Bad Gateway
**Задача:**
1. Останови Python-приложение (если запущено)
2. Убедись что Nginx работает
3. Попробуй: `curl http://myapp.local` — 502?
4. Посмотри error.log: `tail /var/log/nginx/myapp-error.log`
5. Что в логе? (connection refused?)
6. Запусти приложение — проверь снова

### Упражнение 7.4: tcpdump
**Задача:**
1. Запусти tcpdump: `sudo tcpdump -i lo port 80`
2. В другом терминале: `curl http://localhost`
3. Видишь пакеты в tcpdump?
4. Останови: `Ctrl+C`

### Упражнение 7.5: DevOps Think
**Задача:** «Пользователь пишет: сайт myapp.ru не работает, ERR_CONNECTION_TIMED_OUT. Ты на сервере — всё看起来正常. Что делаешь?»

Подсказки:
1. Проверь с сервера: `curl -v http://localhost` — работает?
2. Проверь извне: `curl -v http://myapp.ru` с другого компьютера
3. ufw: `sudo ufw status` — не блокирует ли?
4. DNS: `dig myapp.ru +short` — правильный IP?
5. Маршрутизация: `traceroute myapp.ru` от пользователя
6. Может проблема у провайдера пользователя?
