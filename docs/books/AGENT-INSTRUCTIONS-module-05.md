## Контекст проекта

Ученик прошёл Модули 1–4. У него есть работающий сервер с автодеплоем.

**Что он уже умеет:**
- Управляет Linux-сервером, настраивает systemd-сервисы
- Настраивал Nginx, HTTPS, ufw
- Пишет Dockerfile и docker-compose.yml, управляет образами
- Настроил CI/CD — push в main деплоит автоматически

**Чего пока нет:**
Всё что он делал — работает, пока не случилось ничего плохого. Диск переполнился, база данных упала и данные не восстановить, сервер перегружен и непонятно почему. Он не знает как это предотвратить и как восстановиться когда уже случилось.

**Что он хочет после этой книги:**
Сервер который живёт сам: PostgreSQL настроена правильно, переменные окружения управляются единообразно, бэкапы делаются автоматически, восстановление проверено. Уметь сказать "данные в безопасности" и не просто верить в это.

## Главная идея книги

**Сервер без бэкапа — это не сервер, это бомба с таймером.**

Три уровня надёжности:
```
1. Данные не теряются       → бэкапы + проверенное восстановление
2. Сервис восстанавливается → правильные restart policy, healthcheck
3. Ты знаешь что происходит → управляемые конфиги, логи, переменные
```

Книга строится снизу вверх: сначала привести в порядок конфигурацию,
потом защитить данные, потом научиться восстанавливаться.

## Что читатель настроит к концу книги

```
/opt/myapp/
├── docker-compose.yml
├── .env                    ← все секреты, в gitignore
├── nginx/
│   └── conf.d/app.conf
└── scripts/
    ├── backup.sh           ← бэкап БД + файлов, запускается по cron
    ├── restore.sh          ← восстановление из бэкапа
    └── health-check.sh     ← проверка что всё живо

/etc/cron.d/myapp-backup    ← ежедневный бэкап в 3:00
/var/backups/myapp/         ← локальные бэкапы
  └── 2026-04-11/
      ├── db.sql.gz
      └── uploads.tar.gz
```

Плюс PostgreSQL настроена по-человечески: отдельный пользователь,
права минимальны, параметры производительности выставлены.

### Часть 1: Порядок в конфигурации (Главы 1-2)

#### Глава 1: Управление переменными окружения

**Цель:** все секреты и конфигурация хранятся в одном месте, единообразно.

- Проблема: секреты разбросаны — в docker-compose.yml, в systemd unit, в скриптах
- Единая точка конфигурации: `.env` файл в `/opt/myapp/`
  ```bash
  # /opt/myapp/.env
  POSTGRES_PASSWORD=supersecret
  POSTGRES_USER=myapp
  POSTGRES_DB=myapp_prod
  SECRET_KEY=abc123
  DOMAIN=example.com
  BACKUP_DIR=/var/backups/myapp
  ```
- Как `.env` подхватывают разные инструменты:
  - `docker-compose` — автоматически читает `.env` рядом с `docker-compose.yml`
  - `systemd` — через `EnvironmentFile=/opt/myapp/.env`
  - Shell-скрипты — через `source /opt/myapp/.env`
  - Python-приложение — через `python-dotenv` или переменные окружения контейнера
- Права на `.env`: `chmod 600 .env`, владелец `deploy`
- Не один `.env` — три уровня:
  ```
  .env.example   ← в git (шаблон без значений)
  .env           ← на сервере (реальные значения, не в git)
  .env.local     ← на ноутбуке разработчика (не в git)
  ```
- Проверка: `docker-compose config` — показывает итоговый конфиг с подставленными переменными
- `envsub` / `envsubst` — подставить переменные в шаблон конфига (для nginx)
- **Упражнения:** собрать все переменные в один `.env`, убедиться что `docker-compose config` не содержит хардкодов, проверить права файла

> **Опасно:** `chmod 644 .env` — любой пользователь на сервере прочитает пароли. Всегда `chmod 600`.

#### Глава 2: Nginx — продвинутая настройка

**Цель:** Nginx настроен не "как-нибудь работает", а правильно — с заголовками безопасности, rate limiting и правильными таймаутами.

> **Важно:** Не повторять основы из Модуля 2. Читатель умеет настраивать `proxy_pass` и `ssl`. Здесь — следующий уровень.

- Заголовки безопасности (`add_header`):
  ```nginx
  add_header X-Content-Type-Options nosniff;
  add_header X-Frame-Options DENY;
  add_header Referrer-Policy strict-origin-when-cross-origin;
  add_header Strict-Transport-Security "max-age=31536000" always;
  ```
- Rate limiting — защита от брутфорса и DDoS:
  ```nginx
  limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
  limit_req zone=api burst=20 nodelay;
  ```
- Правильные таймауты (почему важны в связке с приложением):
  ```nginx
  proxy_connect_timeout 10s;
  proxy_read_timeout 60s;
  send_timeout 60s;
  ```
- `gzip` компрессия — меньше трафика, быстрее ответы
- Кастомные страницы ошибок: `error_page 502 /502.html`
- Разделение конфигов: `include /etc/nginx/snippets/ssl.conf`
- Логи: формат JSON для удобного парсинга
- `nginx -t && systemctl reload nginx` — сделать привычкой (алиас в `.bashrc`)
- **Упражнения:** добавить заголовки безопасности, проверить через `curl -I`, настроить rate limiting

### Часть 2: PostgreSQL (Главы 3-4)

#### Глава 3: PostgreSQL — правильная настройка

**Цель:** база данных настроена безопасно, с правильными правами и базовыми параметрами производительности.

- PostgreSQL в Docker vs на хосте — когда что:
  ```
  В Docker:   удобно, переносимо, легко поднять несколько версий
  На хосте:   лучше производительность (нет оверхеда сети), проще бэкапы системными инструментами
  ```
  Книга учит оба способа, основной акцент — Docker (как продолжение Модуля 3).

- **Пользователи и права** (самое важное):
  ```sql
  -- Плохо:
  DATABASE_URL=postgresql://postgres:pass@db/mydb
  -- (суперпользователь postgres имеет права на всё)

  -- Хорошо:
  CREATE USER myapp WITH PASSWORD 'secret';
  CREATE DATABASE myapp_prod OWNER myapp;
  GRANT CONNECT ON DATABASE myapp_prod TO myapp;
  -- Приложение не может удалить базу или создать новую
  ```
- `psql` — основные команды:
  - `\l` — список баз
  - `\c mydb` — подключиться к базе
  - `\dt` — список таблиц
  - `\du` — список пользователей
  - `\q` — выйти
- Подключиться к PostgreSQL в контейнере:
  ```bash
  docker-compose exec db psql -U myapp -d myapp_prod
  ```
- `pg_hba.conf` — кто может подключаться (минимальное понимание)
- Базовые параметры производительности в `postgresql.conf`:
  ```
  shared_buffers = 256MB       # ~25% от RAM
  effective_cache_size = 1GB   # ~75% от RAM
  max_connections = 100        # не ставить 1000
  ```
  Объяснять зачем, не просто скопировать.
- `pg_activity` — мониторинг запросов в реальном времени (как `htop` для PostgreSQL)
- **Упражнения:** создать отдельного пользователя, проверить что он не может удалить базу, выставить параметры памяти

> **Опасно:** Никогда не открывай порт `5432` наружу. Если PostgreSQL в Docker — не пиши `ports: "5432:5432"` в `docker-compose.yml`. Используй только внутренние сети.

#### Глава 4: Миграции базы данных

**Цель:** читатель умеет менять схему БД без потери данных и знает как это делать в пайплайне.

- Проблема: изменил модель в коде → база не обновилась → 500 ошибка
- Что такое миграция: версионированное изменение схемы БД
- Инструменты миграций:
  - Alembic (для SQLAlchemy / FastAPI)
  - Django migrations (встроенный)
  - Flyway / Liquibase (языко-независимые)
  Объяснять на примере Alembic — как наиболее распространённого для Python.
- Базовый workflow с Alembic:
  ```bash
  alembic revision --autogenerate -m "add users table"
  alembic upgrade head
  alembic downgrade -1  # откатить одну миграцию
  alembic history       # история миграций
  ```
- Миграции в CI/CD: запускать до перезапуска контейнера приложения:
  ```yaml
  script: |
    docker-compose run --rm app alembic upgrade head
    docker-compose up -d app
  ```
- Опасные миграции (и как делать безопасно):
  - Удаление колонки — сначала убрать из кода, потом из схемы
  - Переименование — добавить новую, скопировать данные, убрать старую
  - `NOT NULL` без дефолта — нельзя добавить если есть строки
- **Упражнения:** создать миграцию, применить, откатить, применить снова

> **Порядок важен:** сначала миграция, потом деплой нового кода. Наоборот — 500 ошибка пока деплой не закончится.

### Часть 3: Бэкапы (Главы 5-6)

#### Глава 5: Стратегия бэкапов

**Цель:** читатель понимает что бэкапить, куда и как часто. И почему "есть бэкап" ≠ "данные в безопасности".

- Правило 3-2-1:
  ```
  3 копии данных
  2 разных носителя (например: сервер + S3)
  1 копия вне площадки (offsite)
  ```
- Что бэкапить:
  ```
  ✓ База данных (дамп PostgreSQL)
  ✓ Загруженные файлы пользователей (uploads/)
  ✓ Конфиги (/opt/myapp/.env, nginx конфиги)
  ✗ Код (он в git)
  ✗ Docker-образы (они в registry)
  ✗ Временные файлы, кэш
  ```
- Типы бэкапов:
  - Полный (full) — всё каждый раз
  - Инкрементальный — только изменения (для больших данных)
  Для начала: только полный, инкрементальный упомянуть.
- Где хранить:
  - Локально на сервере (`/var/backups/`) — быстро, но если сервер умер — нет бэкапа
  - Удалённо (S3, Backblaze B2, sftp на другом сервере) — надёжно
  Стратегия: локально + удалённо.
- Ротация: сколько бэкапов хранить
  ```
  7 дневных + 4 недельных + 12 месячных
  ```
- **Главное правило:** бэкап не существует пока ты не проверил восстановление.
- **Упражнения:** составить список что нужно бэкапить на своём проекте, рассчитать сколько места займут бэкапы за месяц

#### Глава 6: Автоматические бэкапы — скрипты и cron

**Цель:** бэкапы делаются сами, без участия человека.

- Бэкап PostgreSQL:
  ```bash
  # pg_dump — создать дамп
  docker-compose exec -T db pg_dump -U myapp myapp_prod | gzip > db_$(date +%Y%m%d).sql.gz

  # pg_dumpall — все базы и пользователи
  # pg_dump --format=custom — бинарный формат, быстрее восстанавливается
  ```
- Бэкап файлов:
  ```bash
  tar -czf uploads_$(date +%Y%m%d).tar.gz /opt/myapp/uploads/
  ```
- Полный скрипт `backup.sh`:
  ```bash
  #!/bin/bash
  set -euo pipefail

  source /opt/myapp/.env
  BACKUP_DATE=$(date +%Y-%m-%d)
  BACKUP_PATH="${BACKUP_DIR}/${BACKUP_DATE}"

  mkdir -p "$BACKUP_PATH"

  # База данных
  docker-compose -f /opt/myapp/docker-compose.yml exec -T db \
    pg_dump -U "$POSTGRES_USER" "$POSTGRES_DB" \
    | gzip > "${BACKUP_PATH}/db.sql.gz"

  # Файлы
  tar -czf "${BACKUP_PATH}/uploads.tar.gz" /opt/myapp/uploads/

  # Конфиги
  cp /opt/myapp/.env "${BACKUP_PATH}/env.backup"

  echo "Backup completed: ${BACKUP_PATH}"

  # Удалить старые (оставить 7 дней)
  find "$BACKUP_DIR" -maxdepth 1 -type d -mtime +7 -exec rm -rf {} +
  ```
- `set -euo pipefail` — объяснить зачем (скрипт падает при первой ошибке)
- Настройка cron:
  ```bash
  # /etc/cron.d/myapp-backup
  0 3 * * * deploy /opt/myapp/scripts/backup.sh >> /var/log/myapp-backup.log 2>&1
  ```
- Проверка: `crontab -l`, `grep CRON /var/log/syslog`
- Отправка бэкапов на S3 / Backblaze:
  ```bash
  # rclone — универсальный инструмент для работы с облачными хранилищами
  rclone copy "$BACKUP_PATH" remote:mybucket/myapp/
  ```
  Установка и настройка rclone (базово).
- **Упражнения:** написать скрипт бэкапа, запустить вручную, проверить что файлы созданы, настроить cron

> **Запомни:** Скрипт бэкапа должен логировать результат. Иначе ты не узнаешь что он сломался три недели назад.

#### Глава 7: Восстановление из бэкапа

**Цель:** читатель хотя бы раз прошёл полное восстановление. Не теоретически — руками.

- **Главное правило главы:** не читай, делай. Восстановление нужно практиковать.
- Восстановление PostgreSQL:
  ```bash
  # Из gzip-дампа
  gunzip -c db.sql.gz | docker-compose exec -T db psql -U myapp myapp_prod

  # Из custom-формата (быстрее)
  docker-compose exec -T db pg_restore -U myapp -d myapp_prod < db.dump
  ```
- Восстановление файлов:
  ```bash
  tar -xzf uploads.tar.gz -C /opt/myapp/
  ```
- Полный сценарий: "сервер умер, поднимаем с нуля":
  1. Поднять новый сервер Ubuntu
  2. Установить Docker
  3. Склонировать репозиторий
  4. Создать `.env` из бэкапа
  5. `docker-compose up -d`
  6. Восстановить базу из дампа
  7. Восстановить файлы
  8. Проверить что всё работает
- Скрипт `restore.sh` — зеркало `backup.sh`
- Проверка целостности бэкапа:
  ```bash
  gunzip -t db.sql.gz && echo "OK" || echo "CORRUPTED"
  ```
- **Упражнение (обязательное):** поднять тестовый сервер (или отдельный Docker-контейнер), восстановить туда бэкап, проверить что приложение работает с восстановленными данными.

> **Запомни:** Бэкап который не проверялся — это не бэкап. Это надежда.

### Часть 4: Стабильность (Глава 8)

#### Глава 8: Управление ресурсами и стабильность сервера

**Цель:** сервер не ложится из-за переполненного диска или утечки памяти.

- **Диск:**
  - `df -h` — место на диске (делать привычкой при каждом входе)
  - `du -sh /var/lib/docker` — сколько занимает Docker
  - Docker мусор: образы, контейнеры, тома
    ```bash
    docker system prune        # удалить неиспользуемое (без томов)
    docker system prune -a     # включая образы без тега
    docker system df           # посмотреть что занимает место
    ```
  - Автоматическая чистка через cron:
    ```bash
    0 4 * * 0 root docker system prune -f >> /var/log/docker-prune.log 2>&1
    ```
  - Ротация логов Docker: `/etc/docker/daemon.json`
    ```json
    {
      "log-driver": "json-file",
      "log-opts": {
        "max-size": "10m",
        "max-file": "3"
      }
    }
    ```
  - `logrotate` для логов приложения (повторение из Модуля 1, применение на практике)

- **Память:**
  - `free -h` — состояние памяти
  - `docker stats` — потребление памяти контейнерами
  - Лимиты памяти в docker-compose:
    ```yaml
    deploy:
      resources:
        limits:
          memory: 512m
    ```
  - Swap: когда помогает, когда нет (SSD vs HDD)

- **Мониторинг ресурсов:** `htop`, `iotop`, `nethogs`

- **Автоперезапуск:**
  - `restart: unless-stopped` в docker-compose — уже знает
  - `depends_on` с `healthcheck` — уже знает
  - Что делать если контейнер в restart loop:
    ```bash
    docker-compose logs --tail=50 app   # смотреть логи
    docker inspect myapp                # детали статуса
    ```

- **Итоговый чеклист надёжности:**
  ```
  □ Бэкапы настроены и проверены
  □ Логи ротируются
  □ Docker мусор чистится автоматически
  □ Лимиты памяти выставлены
  □ restart: unless-stopped для всех сервисов
  □ healthcheck для базы данных
  □ .env с правами 600
  □ PostgreSQL не доступна снаружи
  □ Алерт если диск > 80%
  ```

- **Упражнения:** настроить ротацию логов Docker, настроить автоочистку, выставить лимит памяти для контейнера

### Глава 9: Итоговый проект

**Цель:** привести сервер из предыдущих модулей к production-ready состоянию.

Что нужно сделать:
1. Собрать все переменные окружения в один `.env` с правами `600`
2. Создать отдельного пользователя PostgreSQL с минимальными правами
3. Добавить заголовки безопасности и rate limiting в Nginx
4. Написать `backup.sh` — бэкап БД + файлов
5. Настроить cron на ежедневный бэкап в 3:00
6. Настроить отправку бэкапов в облако (rclone + Backblaze/S3)
7. Написать `restore.sh`
8. Провести полное учение: восстановить данные из бэкапа на чистой машине
9. Настроить ротацию логов Docker
10. Настроить автоочистку Docker-мусора раз в неделю

**Чеклист готовности:**
- [ ] `backup.sh` запускается без ошибок, файлы созданы
- [ ] Бэкап в облаке — проверено через UI
- [ ] `restore.sh` проверен на отдельном контейнере
- [ ] `docker-compose ps` — все сервисы `Up (healthy)`
- [ ] `df -h` — диск не > 70%
- [ ] `chmod 600 .env` — проверено
- [ ] `nc -zv localhost 5432` — соединение отклонено (порт закрыт снаружи)

### Приложения

#### Приложение A: Шпаргалка команд PostgreSQL
`psql`, `pg_dump`, `pg_restore`, `pg_dumpall` — синтаксис и примеры

#### Приложение B: Готовые скрипты
- `backup.sh` — полный скрипт бэкапа
- `restore.sh` — восстановление
- `health-check.sh` — проверка состояния стека
- `/etc/cron.d/myapp-backup` — cron для бэкапов

#### Приложение C: Типичные проблемы
- Диск переполнен → `docker system prune`, `find /var/log -size +100M`
- Бэкап создаётся но нулевого размера → `set -e` в скрипте не стоит, pg_dump упал тихо
- Восстановление: `ERROR: database already exists` → дропнуть и создать заново
- PostgreSQL не стартует после изменения конфига → `docker logs db` + проверить `postgresql.conf`

## Принципы написания

Все принципы предыдущих модулей, плюс специфичные:

### 1. Практика восстановления — обязательна
Глава о бэкапах без практического восстановления бесполезна.
Явно говорить читателю: "закрой книгу и сделай это прямо сейчас на тестовой машине".

### 2. Скрипты объясняются построчно
Каждая строка `backup.sh` — с комментарием зачем она нужна.
Особенно: `set -euo pipefail` и `date +%Y%m%d` — не очевидные вещи.

### 3. "Что будет если этого не сделать"
Для каждой настройки безопасности показывай реальный сценарий провала:
> "Если не выставить `chmod 600 .env` — любой пользователь на сервере
> может прочитать пароль от базы: `cat /opt/myapp/.env`"

### 4. ASCII-схемы для архитектуры бэкапов
```
Сервер → backup.sh → /var/backups/ (7 дней)
                   └→ rclone → Backblaze B2 (30 дней)
```

### 5. Никакой воды
- Без теории про RAID и репликацию PostgreSQL
- Без Barman, pgBackRest — только `pg_dump`
- Без Ansible, Terraform — это другой курс
- Без мониторинговых стеков (Prometheus, Grafana) — это Модуль 6
