# Глава 6: Процессы

> **Запомни:** Всё, что работает на сервере, — это процесс. Веб-сервер, база данных, твой скрипт — всё это процессы. Научись ими управлять.

---

## 6.1 Что такое процесс

**Процесс** — это запущенная программа.

Каждый процесс имеет:
- **PID** (Process ID) — уникальный номер
- **Родителя** (PPID) — кто его запустил
- **Владельца** — от какого пользователя работает
- **Состояние** — работает, спит, зомби
- **Приоритет** — сколько CPU получает

Когда ты запускаешь команду в терминале — создаётся процесс.
Когда закрываешь терминал — процесс умирает.

---

## 6.2 Посмотреть процессы: `ps`

`ps` = **p**rocess **s**tatus

### Базовый `ps`

```bash
ps
  PID TTY          TIME CMD
 1234 pts/0    00:00:00 bash
 5678 pts/0    00:00:00 ps
```

Показывает процессы **только в текущем терминале**. Не очень полезно.

### Все процессы: `ps aux`

```bash
ps aux
```

| Колонка | Значение |
|---------|----------|
| `USER` | Кто запустил |
| `PID` | ID процесса |
| `%CPU` | Сколько CPU использует |
| `%MEM` | Сколько памяти использует |
| `VSZ` | Виртуальная память (КБ) |
| `RSS` | Реальная память (КБ) |
| `TTY` | Терминал |
| `STAT` | Состояние |
| `START` | Когда запустился |
| `TIME` | Сколько CPU времени использовал |
| `COMMAND` | Что за команда |

### Найти конкретный процесс

```bash
ps aux | grep nginx
```

Это комбинация: `ps aux` показывает все процессы, `grep nginx` фильтрует по слову.

> **Запомни:** `ps aux | grep <имя>` — базовый и очень частый приём.
> Но `grep` покажет и сам себя. Когда привыкнешь, используй `pgrep -a <имя>` — вывод будет чище.

### Состояния процессов (STAT)

| Код | Значение |
|-----|----------|
| `R` | Running (работает) |
| `S` | Sleeping (ждёт чего-то) |
| `Z` | Zombie (умер но не удалён) |
| `T` | Stopped (остановлен) |

## 6.3 Мониторинг в реальном времени: `top`

`top` показывает процессы которые **меняются в реальном времени**.

```bash
top
```

Откроется экран который обновляется каждые 3 секунды:

```
top - 14:30:00 up 5 days,  3:22,  1 user,  load average: 0.00, 0.01, 0.05
Tasks:  98 total,   1 running,  97 sleeping,   0 stopped,   0 zombie
%Cpu(s):  0.3 us,  0.3 sy,  0.0 ni, 99.3 id,  0.0 wa
MiB Mem :   1982.0 total,    823.4 free,    412.6 used,    746.0 buff/cache
MiB Swap:   1024.0 total,   1024.0 free,      0.0 used.   1369.4 avail Mem

    PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
      1 root      20   0  169312  11872   8192 S   0.0   0.6   0:05.12 systemd
    234 root      20   0   17056   4864   3840 S   0.0   0.2   0:00.45 sshd
```

### Что смотреть

**Первая строка:**
- Сколько времени сервер работает (`up 5 days`)
- Load average: нагрузка (0.00 = пусто, 1.00 = один полный CPU)

**Вторая строка:**
- Сколько всего процессов
- Сколько спит, работает, зомби

**Третья строка:**
- Загрузка CPU
- `id` = idle (простой). Чем больше — тем лучше

**Четвёртая строка:**
- Память: сколько всего, свободно, использовано

### Управление в `top`

| Клавиша | Действие |
|---------|----------|
| `q` | Выйти |
| `M` | Сортировать по памяти |
| `P` | Сортировать по CPU |
| `k` | Убить процесс (спросит PID) |
| `h` | Помощь |

## 6.4 `htop`: улучшенный `top`

`htop` — это `top` но красивый и удобный.

```bash
sudo apt install htop
htop
```

**Почему лучше:**
- Цветной вывод
- Можно кликать мышкой (если есть)
- Полоски загрузки CPU и памяти
- Легче убивать процессы (F9)
- Проще читать

> **Совет:** Поставь `htop` сразу после установки системы. Это одна из первых команд которые ты установишь.

## 6.5 Остановить процесс: `kill`

Процесс завис? Зависает? Мешает? Убей его.

### По PID

```bash
kill 1234
```

Отправляет сигнал TERM (вежливая просьба остановиться).
Процесс может успеть сохраниться и корректно закрыться.

### Если не останавливается

```bash
kill -9 1234
```

`-9` = SIGKILL (мгновенная смерть).
Процесс не успевает ничего сохранить.

> **Совет:** Всегда начинай с обычного `kill`. Только если не помогло — `kill -9`.

### Узнать PID процесса

```bash
# Через ps
ps aux | grep nginx

# Через pidof
pidof nginx

# Через pgrep
pgrep nginx
```

---

## 6.6 `killall`: убить по имени

```bash
killall nginx
```

Убивает **все** процессы с именем `nginx`.

```bash
killall -9 python3
```

Убивает все процессы python3 принудительно.

> **Опасно:** `killall` убьёт ВСЕ процессы с этим именем. Убедись что тебе это нужно.

## 6.7 Фоновые задачи

Ты запустил скрипт. Он работает и занимает терминал. Ты не можешь делать ничего другого.

**Решение:** запусти в фоне.

### Запустить в фоне

Добавь `&` в конец команды:

```bash
python3 server.py &
[1] 5432
```

`&` = выполнять в фоне.
`[1]` = номер задачи.
`5432` = PID.

Терминал свободен — можешь работать дальше.

### Посмотреть фоновые задачи

```bash
jobs
[1]+  Running   python3 server.py &
```

### Вернуть на передний план

```bash
fg
```

`fg` = **f**ore**g**round (на первый план).

### Переместить работающий процесс в фон

1. Нажми `Ctrl+Z` — процесс остановлен
2. Набери `bg` — процесс продолжит в фоне

### Отвязать процесс от терминала

```bash
nohup python3 server.py &
```

`nohup` = no hang up (не убивать при закрытии терминала).

Без `nohup` процесс умрёт когда ты закроешь SSH.

> **Запомни:** `nohup ... &` = запустить в фоне и чтобы жил после выхода.
> Но это временное решение. Для постоянных сервисов используй `systemd` (Глава 7).

## 6.8 Приоритеты: `nice` и `renice`

Процесс может быть более или менее важным.

### Запустить с приоритетом

```bash
nice -n 10 python3 heavy_task.py
```

`-n 10` = приоритет от -20 (важный) до 19 (не важный).
По умолчанию: 0.

### Изменить приоритет работающего процесса

```bash
renice 15 -p 1234
```

Процесс 1234 теперь с приоритетом 15 (меньше ресурсов).

> **Совет:** Фоновые задачи и бэкапы запускай с `nice -n 19` — они не будут мешать основному сервису.

## 6.9 tmux: менеджеры терминалов

### Проблема

Ты подключился по SSH к серверу. Запустил долгую команду. Интернет пропал.

**Что случилось:**
- SSH соединение разорвалось
- Терминал закрылся
- Процесс умер

Твоя работа — потеряна.

> **Запомни:** Без менеджера терминала каждый разрыв SSH убивает твои процессы.

### Решение: tmux

**tmux** — мультиплексор терминала.

Он создаёт «виртуальный» терминал который живёт отдельно от твоего SSH.
- SSH разорвался → tmux сессия жива
- Подключился снова → tmux сессия на месте
- Всё работает как будто ничего не было

### Установка

```bash
sudo apt install -y tmux
```

### Создать сессию

```bash
tmux new -s mysession
```

`-s` = имя сессии. Дай осмысленное имя — потом найдёшь.

Ты внутри tmux. Внизу зелёная полоса — это статусная строка.

### Отключиться (detach)

Нажми **`Ctrl+b`**, отпусти, затем **`d`**.

Ты вернулся в обычный терминал. Но сессия tmux работает в фоне.

```bash
tmux ls
mysession: 1 windows (created Thu Apr  9 14:30:00 2026)
```

### Подключиться обратно (attach)

```bash
tmux attach -t mysession
```

`-t` = target (целевая сессия).

Ты снова внутри. Всё на месте.

### Краткий справочник

Все команды tmux вызываются через **`Ctrl+b`** (префикс):

| Комбинация | Действие |
|------------|----------|
| `Ctrl+b`, затем `d` | Отключиться (detach) |
| `Ctrl+b`, затем `"` | Разделить окно горизонтально |
| `Ctrl+b`, затем `%` | Разделить окно вертикально |
| `Ctrl+b`, затем `стрелка` | Перейти между панелями |
| `Ctrl+b`, затем `c` | Создать новое окно |
| `Ctrl+b`, затем `n` | Следующее окно |
| `Ctrl+b`, затем `p` | Предыдущее окно |
| `Ctrl+b`, затем `0-9` | Перейти к окну по номеру |
| `Ctrl+b`, затем `[` | Режим прокрутки (стрелки, q для выхода) |

### Разбиение экрана

`tmux` может разделить терминал на несколько панелей:

```
┌──────────────┬──────────────┐
│              │   Панель 2   │
│   Панель 1   ├──────────────┤
│              │   Панель 3   │
└──────────────┴──────────────┘
```

В каждой панели — свой терминал.

**Пример использования:**
- Панель 1: `tail -f /var/log/syslog` (следишь за логами)
- Панель 2: работаешь с файлами
- Панель 3: `htop` (мониторишь ресурсы)

### Переключение между окнами

Каждое `tmux` окно — как вкладка в браузере.

```
[0] logs  [1] editor  [2] htop
```

- `Ctrl+b c` — создать новое окно
- `Ctrl+b n` — следующее
- `Ctrl+b p` — предыдущее
- `Ctrl+b 0` — окно 0

### Полезные команды

```bash
# Список сессий
tmux ls

# Убить сессию
tmux kill-session -t mysession

# Убить все сессии
tmux kill-server

# Подключиться к последней сессии
tmux attach
```

### Типичный рабочий процесс

```bash
# 1. Подключился к серверу по SSH
ssh adelfos@server

# 2. Создал tmux сессию
tmux new -s work

# 3. Работаешь, смотришь логи, редактируешь файлы
# ... часы работы ...

# 4. Интернет пропал / нужно уйти
# SSH разорвался, но tmux жив!

# 5. Подключился снова
ssh adelfos@server
tmux attach -t work

# Всё на месте. Продолжаешь работать.
```

> **Совет:** tmux — базовый инструмент DevOps'а.
> Научись базовым командам и используй всегда.

### screen — альтернатива

`screen` — старая альтернатива tmux. Работает похоже.

```bash
screen -S mysession      # создать
Ctrl+a, d                 # отключиться
screen -ls               # список
screen -r mysession      # подключиться
```

> **Совет:** tmux лучше — больше функций, активнее развивается.
> Но screen может стоять на старых серверах где нет tmux.

### Упражнение 6.2: Мониторинг
**Задача:**
1. Запусти `top`
2. Отсортируй по памяти (нажми `M`)
3. Отсортируй по CPU (нажми `P`)
4. Выйди (нажми `q`)
5. Если есть `htop` — сравни вывод

### Упражнение 6.3: Запустить и убить процесс
**Задача:**
1. Запусти долгий процесс: `sleep 300 &`
2. Узнай его PID: `jobs -l` или `pgrep sleep`
3. Убей его: `kill <PID>`
4. Убедись что умер: `pgrep sleep` (должно быть пусто)

### Упражнение 6.4: Фоновые задачи
**Задача:**
1. Запусти: `python3 -c "import time; time.sleep(60)" &`
2. Проверь: `jobs`
3. Запусти ещё один: `sleep 120 &`
4. Проверь: `jobs` — оба работают?
5. Переведи первый на передний план: `fg %1`
6. Останови: `Ctrl+Z`
7. Продолжи в фоне: `bg`
8. Убей оба: `killall python3; killall sleep`

### Упражнение 6.5: nohup
**Задача:**
1. Запусти: `nohup sleep 600 &`
2. Закрой терминал и открой снова
3. Проверь: `ps aux | grep sleep` — процесс жив?
4. Убей его: `killall sleep`

### Упражнение 6.6: tmux — основы
**Задача:**
1. Установи tmux: `sudo apt install -y tmux`
2. Создай сессию: `tmux new -s test`
3. Внутри tmux запусти: `htop`
4. Отключись: `Ctrl+b`, затем `d`
5. Убедись что сессия жива: `tmux ls`
6. Подключись обратно: `tmux attach -t test`
7. htop всё ещё работает?
8. Убей сессию: `Ctrl+b`, затем `d`, потом `tmux kill-session -t test`

### Упражнение 6.7: tmux — панели
**Задача:**
1. Создай сессию: `tmux new -s panels`
2. Раздели горизонтально: `Ctrl+b "`
3. В верхней панели запусти: `tail -f /var/log/syslog`
4. Перейди в нижнюю: `Ctrl+b`, стрелка вниз
5. Запусти: `htop`
6. Раздели нижнюю вертикально: `Ctrl+b %`
7. В третьей панели запусти: `df -h`
8. Погуляй между панелями
9. Отключись и подключись снова — всё на месте?
10. Убей сессию

### Упражнение 6.8: DevOps Think
**Задача:** «Процесс nginx перестал отвечать. Найди и диагностируй»

Подсказки:
1. Работает ли процесс nginx? `ps aux | grep nginx`
2. Какой PID? `pgrep nginx`
3. Сколько памяти использует? `ps aux | grep nginx`
4. Посмотри логи: `sudo journalctl -u nginx -n 50`
5. Проверь порт: `ss -tlnp | grep 80`
6. Попробуй подключиться: `curl -I http://localhost`
