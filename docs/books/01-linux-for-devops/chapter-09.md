# Глава 9: Пользователи и группы

> **Запомни:** На сервере не должно быть ни одного пользователя без цели. Каждый пользователь — это потенциальная дыра в безопасности.

---

## 9.1 Зачем много пользователей

На домашнем компьютере ты один. На сервере — много программ и людей.

**Каждому сервису — свой пользователь:**

| Пользователь | Зачем |
|--------------|-------|
| `adelfos` | Ты (админ) |
| `www-data` | Веб-сервер (Nginx) |
| `postgres` | База данных |
| `redis` | Redis |
| `deploy` | Автодеплой из Git |

**Почему не все под root:**
- Если взломают Nginx — получат доступ только к файлам `www-data`
- Если все под root — взлом одного сервиса = полный доступ ко всему серверу

> **Запомни:** Принцип минимальных привилегий. Каждый процесс получает только то что ему нужно.

## 9.2 Где хранятся пользователи

### `/etc/passwd` — список пользователей

```bash
cat /etc/passwd
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
...
adelfos:x:1000:1000:Adelfos,,,:/home/adelfos:/bin/bash
```

Формат: `имя:пароль:UID:GID:описание:дом:оболочка`

| Поле | Значение |
|------|----------|
| `имя` | Логин |
| `пароль` | `x` = пароль в `/etc/shadow` |
| `UID` | ID пользователя (0 = root) |
| `GID` | ID основной группы |
| `дом` | Домашняя директория |
| `оболочка` | Терминал (`/bin/bash` или `/usr/sbin/nologin`) |

> **Запомни:** `/usr/sbin/nologin` = пользователь НЕ может войти в систему.
> Сервисные пользователи (www-data, postgres) имеют nologin — это правильно.

### `/etc/shadow` — хэши паролей

```bash
sudo cat /etc/shadow
root:$6$randomhash...:19000:0:99999:7:::
adelfos:$6$anotherhash...:19100:0:99999:7:::
```

Только root может читать. Содержит хэши паролей (не сами пароли).

### `/etc/group` — группы

```bash
cat /etc/group
root:x:0:
sudo:x:27:adelfos
www-data:x:33:
```

Формат: `имя:пароль:GID:участники`

## 9.3 Создать пользователя: `useradd`

```bash
sudo useradd -m -s /bin/bash deploy
```

| Опция | Значение |
|-------|----------|
| `-m` | Создать домашнюю директорию |
| `-s /bin/bash` | Установить оболочку bash |

### С паролем

```bash
sudo useradd -m -s /bin/bash deploy
sudo passwd deploy
```

`passwd` запросит новый пароль.

### Одной командой

```bash
sudo useradd -m -s /bin/bash -c "Deploy User" deploy
```

`-c` = комментарий (полное имя).

---

## 9.4 Изменить пользователя: `usermod`

### Добавить в группу

```bash
sudo usermod -aG sudo deploy
```

`-aG` = append to Group (добавить в группу, не убирая из других).

> **Опасно:** Без `-a` пользователь будет УДАЛЁН из всех других групп!
> Всегда используй `-aG`.

### Сменить оболочку

```bash
sudo usermod -s /usr/sbin/nologin olduser
```

Теперь пользователь не может войти в систему.

### Сменить домашнюю директорию

```bash
sudo usermod -d /new/home -m deploy
```

`-m` = переместить файлы.

---

## 9.5 Удалить пользователя: `userdel`

```bash
sudo userdel deploy
```

Удалит пользователя, но оставит домашнюю директорию.

### Удалить с домашней директорией

```bash
sudo userdel -r deploy
```

`-r` = удалить домашнюю директорию и почтовый ящик.

## 9.6 Управление группами

### Создать группу

```bash
sudo groupadd developers
```

### Добавить пользователя в группу

```bash
sudo usermod -aG developers adelfos
```

### Удалить пользователя из группы

```bash
sudo gpasswd -d adelfos developers
```

### Удалить группу

```bash
sudo groupdel developers
```

### Посмотреть группы пользователя

```bash
groups adelfos
adelfos : adelfos sudo developers

id adelfos
uid=1000(adelfos) gid=1000(adelfos) группы=1000(adelfos),27(sudo),1001(developers)
```

## 9.7 Частые группы в Ubuntu

| Группа | Зачем |
|--------|-------|
| `sudo` | Можно использовать sudo |
| `www-data` | Доступ к файлам веб-сервера |
| `docker` | Управление Docker без sudo |
| `ssh` | Доступ к SSH |
| `adm` | Чтение системных логов |

### Добавить в группу docker

```bash
sudo usermod -aG docker adelfos
```

После этого нужно перелогиниться.

## 9.8 SSH-подключение и ключи

### Подключение по паролю

Самый простой способ:

```bash
ssh adelfos@192.168.1.100
```

Формат: `ssh пользователь@сервер`

Сервер спросит пароль — введи пароль пользователя на сервере.

> **Запомни:** Пароль — для первого раза.
> Ключи — для постоянной работы.

### Почему ключи лучше паролей

| Проблема | Пароль | Ключ |
|----------|--------|------|
| Подбор | Можно брутфорсить | Практически невозможно |
| Ввод | Каждый раз вручную | Автоматически |
| Передача | Можно перехватить | Одноразовая передача |
| Отзыв | Поменять пароль везде | Удалить один ключ |
| Скрипты | Нужно хранить пароль | Работает автоматически |

> **Запомни:** На реальном сервере вход по паролю отключают.
> Только ключи. Приучайся сейчас.

### Генерация ключа

На своём компьютере (НЕ на сервере):

```bash
ssh-keygen -t ed25519 -C "adelfos@mylaptop"
```

| Опция | Значение |
|-------|----------|
| `-t ed25519` | Тип ключа (ed25519 — современный, безопасный) |
| `-C` | Комментарий (email или имя) |

Рекомендуемый вариант для обычной работы:

```
Enter file in which to save the key (~/.ssh/id_ed25519): [Enter]
Enter passphrase (recommended): [введи passphrase]
Enter same passphrase again: [повтори её]
```

> **Совет:** Для своего ноутбука и реальных серверов ставь passphrase на ключ.
> Чтобы не вводить её при каждом подключении, используй `ssh-agent`.
> Без passphrase можно только на учебной VM или одноразовом лабораторном стенде.

**Результат:**
```
~/.ssh/id_ed25519       ← приватный ключ (НИКОМУ НЕ ДАВАТЬ)
~/.ssh/id_ed25519.pub   ← публичный ключ (это можно давать)
```

### Копирование ключа на сервер

```bash
ssh-copy-id adelfos@192.168.1.100
```

Команда:
1. Берёт твой публичный ключ
2. Подключается к серверу (спросит пароль последний раз)
3. Добавляет ключ в `~/.ssh/authorized_keys`
4. Готово

Проверь:
```bash
ssh adelfos@192.168.1.100
```

Пароль пользователя сервера больше не спрашивается.
Если ты поставил passphrase на ключ — её может спросить SSH-клиент или `ssh-agent`. Это нормально.

### Вручную добавить ключ

Если `ssh-copy-id` нет (например на Mac):

```bash
# На сервере
mkdir -p ~/.ssh
chmod 700 ~/.ssh

# Скопируй содержимое id_ed25519.pub на сервер
nano ~/.ssh/authorized_keys
# Вставь одну строку публичного ключа

chmod 600 ~/.ssh/authorized_keys
```

> **Запомни:**
> - `~/.ssh` — права `700`
> - `~/.ssh/authorized_keys` — права `600`
> - Приватный ключ — НИКОМУ
> - Публичный ключ — можно передавать

### Алиасы: ~/.ssh/config

Надоело вводить IP каждый раз? Настрой алиасы:

```bash
nano ~/.ssh/config
```

```
Host devops
    HostName 192.168.1.100
    User adelfos
    IdentityFile ~/.ssh/id_ed25519

Host production
    HostName 203.0.113.50
    User deploy
    Port 2222
    IdentityFile ~/.ssh/production_key
```

Теперь вместо:
```bash
ssh adelfos@192.168.1.100
```

Просто:
```bash
ssh devops
```

> **Совет:** `~/.ssh/config` — один из самых полезных файлов.
> Настрой сразу для всех своих серверов.

### Полезные опции SSH

```bash
# Подключиться на другом порту
ssh -p 2222 user@server

# Выполнить одну команду и выйти
ssh devops "df -h"

# Скопировать файл на сервер
scp file.txt devops:/home/adelfos/

# Скопировать директорию
scp -r myproject/ devops:/home/adelfos/

# Скопировать с сервера
scp devops:/var/log/syslog ./
```

### Отключить вход по паролю

После настройки ключей — отключи пароль:

```bash
sudo nano /etc/ssh/sshd_config
# Найди и измени:
PasswordAuthentication no
PermitRootLogin no
```

```bash
sudo systemctl restart ssh
```

> **Опасно:** Проверь что ключи работают ПЕРЕД отключением паролей!
> Иначе потеряешь доступ к серверу.

## 9.9 `su` vs `sudo`

### `su` — сменить пользователя

```bash
su - root
```

Запросит пароль root. Станешь root.

```bash
su - deploy
```

Станешь пользователем `deploy`.

> **Совет:** `su -` (с дефисом) загружает окружение нового пользователя.
> `su` (без дефиса) — сохраняет твоё окружение. Всегда используй `su -`.

### `sudo` — выполнить одну команду от root

```bash
sudo apt update
```

Запросит ТВОЙ пароль. Выполнит команду от root. Вернёт тебя обратно.

### `sudo -i` — стать root

```bash
sudo -i
```

Станешь root с его окружением. Как `su - root` но через sudo.

## 9.10 Best practices: безопасность

### Правило 1: Не работай под root

Всегда используй обычного пользователя + `sudo`.

### Правило 2: Отключи вход по паролю для SSH

```bash
sudo nano /etc/ssh/sshd_config
# Найди и измени:
PasswordAuthentication no
```

```bash
sudo systemctl restart ssh
```

Теперь только по ключам.

### Правило 3: Удали ненужных пользователей

```bash
sudo userdel -r olduser
```

### Правило 4: Проверяй кто в группе sudo

```bash
grep sudo /etc/group
sudo:x:27:adelfos
```

Только те кто должен иметь root-доступ.

### Правило 5: Сервисы = nologin

```bash
sudo usermod -s /usr/sbin/nologin www-data
```

Сервисные пользователи не должны логиниться.

## 📝 Упражнения

### Упражнение 9.1: Создать пользователя для деплоя
**Задача:**
1. Создай пользователя: `sudo useradd -m -s /bin/bash -c "Deploy User" deploy`
2. Установи пароль: `sudo passwd deploy`
3. Проверь: `id deploy`
4. Проверь домашнюю директорию: `ls -la /home/deploy`

### Упражнение 9.2: Управление группами
**Задача:**
1. Создай группу: `sudo groupadd devteam`
2. Добавь себя: `sudo usermod -aG devteam $USER`
3. Добавь deploy: `sudo usermod -aG devteam deploy`
4. Проверь: `groups $USER`
5. Удали deploy из группы: `sudo gpasswd -d deploy devteam`

### Упражнение 9.3: SSH-ключи и подключение
**Задача:**
1. Проверь есть ли ключи: `ls -la ~/.ssh/`
2. Если нет — сгенерируй: `ssh-keygen -t ed25519 -C "you@laptop"`
3. Посмотри публичный ключ: `cat ~/.ssh/id_ed25519.pub`
4. Скопируй на виртуалку: `ssh-copy-id username@IP-виртуалки`
5. Подключись: `ssh username@IP-виртуалки` — пароль сервера больше не спрашивается?
6. Настрой алиас: `nano ~/.ssh/config`
7. Подключись по алиасу: `ssh devops`

### Упражнение 9.4: su и sudo
**Задача:**
1. Стань deploy: `sudo su - deploy`
2. Проверь кто ты: `whoami`
3. Проверь где ты: `pwd`
4. Вернись: `exit`

### Упражнение 9.5: Безопасность
**Задача:**
1. Посмотри всех пользователей: `cat /etc/passwd`
2. Посмотри у кого есть оболочка: `grep "/bin/bash" /etc/passwd`
3. Посмотри кто в sudo: `grep sudo /etc/group`
4. Проверь что у сервисных пользователей nologin: `grep "nologin" /etc/passwd`
