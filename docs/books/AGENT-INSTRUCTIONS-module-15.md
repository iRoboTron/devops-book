# Инструкция для ИИ-агента: Модуль 15 — Защита от внешних атак

> **Это первая книга части 3.**
> Она должна быть полностью самостоятельной и подробной.

---

## Контекст проекта

Читатель — программист, который понимает Linux на базовом уровне, но не строил защиту публичного сервера.

Он хочет понять:
- как выглядит поверхность атаки;
- что делает роутер, firewall, WAF, IDS/IPS;
- как защищают Linux-сервер на входе;
- как безопасно проверить, что защита работает.

---

## Что за книга

**Название:** "Защита от внешних атак: Публичный сервер, периметр и Linux hardening"

**Каталог:** `perimeter-security-devops`

**Для кого особенно полезна:**
- дом;
- VPS;

- всем, у кого есть публичный сервер.

**Объём:** 150-190 страниц

---

## Главная идея книги

Защита начинается до приложения:

`Интернет -> router/firewall/gateway -> reverse proxy/WAF -> Linux host -> app`

Читатель должен увидеть, что:
- один только Linux firewall не решает всё;
- одна "коробка" перед сервером тоже не решает всё;
- защита строится слоями;
- логи и проверки так же важны, как правила блокировки.

---

## Что читатель построит к концу книги

Минимальный результат:

- защищённый публичный Linux-сервер;
- SSH только по ключам;
- firewall policy по умолчанию deny;
- закрытая поверхность атаки;
- rate limiting и reverse proxy;
- fail2ban/CrowdSec;
- базовый IDS/IPS или его lab-модель;
- понимание anti-DDoS-слоёв;
- документированные тесты проверки защиты.

## Итоговый проект

Обязательно три варианта:

1. **Один публичный Linux-сервер**
   - SSH hardened
   - firewall
   - reverse proxy
   - fail2ban/CrowdSec
   - базовый WAF/rate limiting

2. **Gateway + DMZ + app server**
   - отдельный периметр
   - app server за gateway
   - только нужные направления трафика

3. **Lab со сценариями атак**
   - отдельные attacker/client/server VM
   - безопасные проверки:
     - `nmap`
     - перебор логина на своей lab-VM
     - шумный HTTP flood в рамках своей сети
     - проверка логов, банов, rate limits

---

## Обязательные безопасные демонстрации

- сканирование своих портов;
- тестовый brute-force против своей lab-VM;
- наблюдение логов SSH и nginx;
- проверка, что БД не торчит наружу;
- сравнение "до защиты" и "после защиты";
- проверка поведения при всплеске запросов;
- проверка, что DDoS нельзя "починить одной галочкой".

- Не сводить книгу к наборам `ufw allow`.
- Не делать enterprise-only текст.
