## 10.1 IP-адреса: простой аналог

**IP-адрес** — это как почтовый адрес компьютера.

```
192.168.1.100
```

Каждое устройство в сети имеет свой IP. Без него — не общается.

### Два типа IP

| Тип | Пример | Где используется |
|-----|--------|-----------------|
| Внешний (public) | `203.0.113.50` | Интернет |
| Внутренний (private) | `192.168.1.100` | Локальная сеть |

### Частные сети (не видны из интернета)

| Диапазон | Где |
|----------|-----|
| `10.0.0.0 – 10.255.255.255` | Большие корпоративные сети |
| `172.16.0.0 – 172.31.255.255` | Средние сети |
| `192.168.0.0 – 192.168.255.255` | Домашние сети |

> **Запомни:** Если IP начинается на `192.168.` или `10.` — это внутренний адрес.
> Сервер в интернете имеет внешний адрес.

## 10.2 Порты: двери в компьютере

**Порт** — это «дверь» в компьютере. Каждый сервис слушает свою дверь.

```
192.168.1.100:80
└──────────┘ └┘
   IP-адрес   Порт
```

| Порт | Сервис |
|------|--------|
| 22 | SSH |
| 80 | HTTP (веб) |
| 443 | HTTPS (защищённый веб) |
| 5432 | PostgreSQL |
| 3306 | MySQL |
| 8000 | Python (часто) |
| 3000 | Node.js (часто) |

> **Запомни:** IP = какой компьютер. Порт = какая программа на этом компьютере.

## 10.3 Протоколы: TCP и UDP

**Протокол** — правила общения.

### TCP (Transmission Control Protocol)

- Надёжный: гарантирует что данные дошли
- Медленнее: проверяет каждую посылку
- Для: веб, почта, базы данных, SSH

### UDP (User Datagram Protocol)

- Быстрый: шлёт без проверки
- Ненадёжный: часть может потеряться
- Для: видео, голос, игры, DNS-запросы

> **Для DevOps:** Почти всё что ты будешь настраивать — TCP.
> Веб-серверы, базы данных, SSH — всё TCP.

## 10.4 DNS: имена вместо цифр

**DNS** (Domain Name System) — переводит имена в IP.

```
example.com → 93.184.216.34
```

Без DNS пришлось бы запоминать IP-адреса.

### Как работает

1. Ты пишешь `example.com` в браузере
2. Компьютер спрашивает DNS-сервер: «Какой IP у example.com?»
3. DNS-сервер отвечает: `93.184.216.34`
4. Компьютер подключается к этому IP

### Проверить DNS

```bash
dig example.com
```

Или проще:
```bash
nslookup example.com
```

### Файл `/etc/hosts`

Локальный DNS. Можно сопоставить имя с IP вручную:

```bash
cat /etc/hosts
127.0.0.1   localhost
192.168.1.100   myserver.local
```

> **Совет:** `/etc/hosts` удобен для тестов когда ещё нет реального домена.

## 10.5 Узнать свой IP: `ip addr`

```bash
ip addr
1: lo: <LOOPBACK,UP,LOWER_UP>
    inet 127.0.0.1/8 scope host lo
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP>
    inet 192.168.1.100/24 brd 192.168.1.255 scope global eth0
```

### Разбор

| Интерфейс | Что это |
|-----------|---------|
| `lo` | Loopback (127.0.0.1) — обращение к самому себе |
| `eth0` | Основная сетевая карта (кабель) |
| `wlan0` | Wi-Fi (если есть) |
| `docker0` | Сетевой мост Docker |

### Проще: одна команда

```bash
ip -4 addr show eth0 | grep inet
inet 192.168.1.100/24
```

Или ещё проще:
```bash
hostname -I
192.168.1.100
```

## 10.6 Открытые порты: `ss`

`ss` = **s**ocket **s**tatistics (статистика сокетов)

### Все слушающие порты

```bash
ss -tlnp
State   Recv-Q  Send-Q   Local Address:Port    Peer Address:Port  Process
LISTEN  0       128            0.0.0.0:22           0.0.0.0:*      users:(("sshd",pid=234,fd=3))
LISTEN  0       128            0.0.0.0:80           0.0.0.0:*      users:(("nginx",pid=567,fd=6))
LISTEN  0       128          127.0.0.1:5432         0.0.0.0:*      users:(("postgres",pid=890,fd=5))
```

| Опция | Значение |
|-------|----------|
| `-t` | TCP |
| `-l` | Listening (слушают) |
| `-n` | Числа (не имена) |
| `-p` | Процесс (кто слушает) |

### Разбор результатов

| Адрес | Значение |
|-------|----------|
| `0.0.0.0:22` | SSH доступен отовсюду |
| `0.0.0.0:80` | Nginx доступен отовсюду |
| `127.0.0.1:5432` | PostgreSQL только с этого сервера |

> **Запомни:** `127.0.0.1` = только с этого сервера (localhost).
> `0.0.0.0` = отовсюду (включая интернет).
> 
> Если база данных на `0.0.0.0` — это дыра в безопасности!

### Найти конкретный порт

```bash
ss -tlnp | grep 80
```

## 10.7 Проверить доступность: `ping`

```bash
ping google.com
PING google.com (142.250.189.14) 56(84) bytes of data.
64 bytes from 142.250.189.14: icmp_seq=1 ttl=114 time=15.3 ms
64 bytes from 142.250.189.14: icmp_seq=2 ttl=114 time=14.8 ms
```

`Ctrl+C` чтобы остановить.

### Что смотреть

| Показатель | Значение |
|------------|----------|
| `time=15.3 ms` | Задержка (меньше = лучше) |
| `icmp_seq` | Номер запроса |
| `packet loss` | Потерянные пакеты (должно быть 0%) |

> **Совет:** `ping` проверяет что сервер «жив». Но не проверяет что сервис работает.
> Сервер может отвечать на ping но Nginx может быть упавшим.

## 10.8 HTTP-запросы: `curl`

`curl` — делает HTTP-запросы из терминала.

### Простой запрос

```bash
curl http://example.com
<!doctype html>
<html>
<head><title>Example Domain</title>...
```

Показывает HTML страницы.

### Только заголовки

```bash
curl -I http://example.com
HTTP/1.1 200 OK
Content-Type: text/html
Server: nginx
```

`-I` = только заголовки (HEAD request).

### Проверить HTTPS

```bash
curl -I https://example.com
```

### Записать в файл

```bash
curl -o page.html http://example.com
```

### Скачать файл

```bash
curl -O https://example.com/file.zip
```

### Отправить POST запрос

```bash
curl -X POST -H "Content-Type: application/json" \
     -d '{"name":"test"}' \
     http://localhost:8000/api/
```

> **Совет:** `curl` — главный инструмент для проверки веб-сервисов.
> Если `curl http://localhost:8000` работает — приложение запущено.

## 10.9 Скачать файл: `wget`

```bash
wget https://example.com/file.zip
```

В отличие от `curl` — `wget` сохраняет в файл по умолчанию.

### Рекурсивное скачивание

```bash
wget -r https://example.com/docs/
```

Скачает всю страницу со всеми файлами.

## 10.10 Диагностика соединений: `nc` (netcat)

`nc` = **n**et**c**at — универсальный сетевой инструмент.

### Проверить порт

```bash
nc -zv localhost 22
Connection to localhost 22 port [tcp/ssh] succeeded!
```

`-z` = scan mode (не отправлять данные)
`-v` = verbose (подробно)

### Проверить несколько портов

```bash
nc -zv localhost 22 80 443 5432
```

### Простой сервер

```bash
nc -l 8080
```

Слушает порт 8080. Всё что напишут — покажет.
Из другого терминала:
```bash
nc localhost 8080
Hello!
```

> **Совет:** `nc -zv` — самый быстрый способ проверить доступен ли порт.

## 10.11 Фаервол: `ufw`

`ufw` = **U**ncomplicated **F**ire**w**all (простой фаервол)

### Проверить статус

```bash
sudo ufw status
Status: inactive
```

### Включить

```bash
sudo ufw enable
```

> **Опасно:** Перед включением убедись что SSH (порт 22) разрешён!
> Иначе потеряешь доступ к серверу.

### Разрешить порт

```bash
sudo ufw allow 22/tcp     # SSH
sudo ufw allow 80/tcp     # HTTP
sudo ufw allow 443/tcp    # HTTPS
```

### Разрешить сервис по имени

```bash
sudo ufw allow OpenSSH
sudo ufw allow 'Nginx Full'
```

### Запретить порт

```bash
sudo ufw deny 3306/tcp    # Закрыть MySQL извне
```

### Удалить правило

```bash
sudo ufw status numbered
sudo ufw delete 3         # удалить правило номер 3
```

### Типичный набор для веб-сервера

```bash
sudo ufw allow OpenSSH
sudo ufw allow 'Nginx Full'
sudo ufw enable
sudo ufw status
```

Результат:
```
Status: active

To                         Action      From
--                         ------      ----
OpenSSH                    ALLOW       Anywhere
Nginx Full                 ALLOW       Anywhere
```

## 📝 Упражнения

### Упражнение 10.1: Узнать свою сеть
**Задача:**
1. Узнай IP сервера: `hostname -I`
2. Посмотри все интерфейсы: `ip addr`
3. Разберись: какой IP внешний, какой внутренний?

### Упражнение 10.2: Проверить порты
**Задача:**
1. Посмотри открытые порты: `ss -tlnp`
2. Какие сервисы слушают?
3. Есть ли что-то на `0.0.0.0`?
4. Проверь конкретный порт: `nc -zv localhost 22`

### Упражнение 10.3: curl
**Задача:**
1. Сделай запрос к Google: `curl -I https://google.com`
2. Какой статус код? (200 = ок)
3. Проверь свой Nginx (если установлен): `curl -I http://localhost`
4. Какой сервер указан в заголовке?

### Упражнение 10.4: ping
**Задача:**
1. Пропингуй Google: `ping -c 4 google.com`
2. Какая задержка?
3. Пропингуй свой сервер с другого устройства
4. Есть ли потеря пакетов?

### Упражнение 10.5: Фаервол
**Задача:**
1. Проверь статус: `sudo ufw status`
2. Если inactive — разреши SSH: `sudo ufw allow OpenSSH`
3. Включи: `sudo ufw enable`
4. Проверь: `sudo ufw status`
5. Разреши HTTP и HTTPS: `sudo ufw allow 'Nginx Full'`
