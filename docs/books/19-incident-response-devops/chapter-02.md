# Глава 2: Метрики и алерты

> **Запомни:** метрики нужны не ради красивого графика, а чтобы вовремя заметить ненормальное поведение и быстро сузить круг причин.

---

## 2.1 Контекст и границы

Инциденты часто видны сначала как отклонение в метрике: всплеск 401, рост latency, исчерпание диска, падение свободной памяти, необычный outbound traffic.

Плохой алертинг утомляет. Нужны сигналы, которые действительно связаны с риском или пользовательским impact, а не шумят на каждое колебание.

Эта глава особенно полезна для тех, у кого уже есть хотя бы базовые логи и несколько сервисов, за которыми нужно наблюдать.

---

## 2.2 Как выглядит риск

Типовые слабые места:
- нет метрик на auth, ошибки, ресурсы и сеть;
- есть сотни алертов, но никто не знает, какие из них важны;
- thresholds не учитывают baseline;
- алерты не содержат контекст;
- оператор узнает о проблеме только из жалобы пользователя.

### Где особенно важно
- Prometheus/Grafana
- cloud monitoring
- self-hosted exporters
- docker metrics

---

## 2.3 Что строит защитник

- минимальный baseline метрик: CPU, memory, disk, network, HTTP codes, auth failures;
- алерты на резкие и устойчивые отклонения;
- dashboard по сервису и инцидентному контуру;
- метки окружения, хоста и сервиса;
- runbook для каждого важного алерта.

### Практический результат главы
- ты понимаешь, какие метрики связаны с безопасностью и доступностью;
- умеешь связать алерт с логами и состоянием узла;
- можешь убрать шум из алертинга.

```bash
free -h
df -h
ss -s
```

---

## 2.4 Практика

### Шаг 1: Собери baseline
- для каждого сервиса выпиши ключевые технические и security-sensitive метрики;
- отдельно выдели auth failures, HTTP 5xx и disk usage.

```bash
free -h
df -h
ss -s
curl -s -o /dev/null -w "http_status=%{http_code}\n" https://HOST || true
sudo grep -c "Failed password\|Invalid user" /var/log/auth.log 2>/dev/null || true
```

### Шаг 2: Проверь текущие алерты
- если мониторинг уже есть, классифицируй алерты на useful, noisy и missing;
- для полезных зафиксируй runbook.

```bash
grep -RniE "alert:|expr:|threshold|severity" monitoring/ prometheus/ grafana/ 2>/dev/null | head -20
systemctl list-timers --all | grep -Ei 'alert|monitor|prometheus' || true
```

### Шаг 3: Сымитируй безопасное отклонение
- на своей lab создай контролируемый burst запросов или несколько ошибочных входов;
- проверь, что метрика заметна и алерт читаем.

```bash
curl -I https://HOST
```

### Что нужно явно показать
- какие базовые метрики уже собираются;
- какие алерты считаются приоритетными;
- какой runbook привязан к критичному алерту;
- как алерт приводит к расследованию.

---

## 2.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- проверь на своем стенде, что рост ошибок входа или HTTP 5xx действительно виден в мониторинге;
- убедись, что алерт содержит достаточный контекст для действия;
- зафиксируй минимум один полезный и один шумный алерт.

---

## 2.6 Типовые ошибки

- мерить только CPU и считать наблюдаемость достаточной;
- не иметь сигналов на auth и error rates;
- строить алерты без привязки к baseline;
- не писать runbook.
