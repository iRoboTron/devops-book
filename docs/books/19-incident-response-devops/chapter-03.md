# Глава 3: Host-level detection

> **Запомни:** сетевые сигналы полезны, но многие инциденты становятся понятны только на самом хосте: процессы, новые бинарники, sudo, cron, systemd и file changes.

---

## 3.1 Контекст и границы

На уровне хоста видны факты, которые не всегда проходят через сеть: запуск новых процессов, изменение unit-файлов, неожиданный listener, замена бинарника, странный sudo.

Даже без полноценного EDR инженер может собрать хороший baseline системными инструментами и научиться быстро замечать отклонения.

Эта глава особенно полезна для Linux-хостов, где нужно видеть локальные признаки компрометации или misconfig.

---

## 3.2 Как выглядит риск

Типовые слабые места:
- нет baseline процессов, сервисов и listening ports;
- sudo и systemd изменения не отслеживаются;
- неизвестно, кто и когда менял критичные файлы;
- инцидент расследуют только через top;
- все необычное списывается на deploy.

### Где особенно важно
- VPS
- bare metal
- lab VM
- docker host

---

## 3.3 Что строит защитник

- baseline по процессам, пакетам, listening ports и systemd services;
- аудит sudo, auth и изменений конфигурации;
- регулярный снимок installed packages и активных unit-файлов;
- минимальный file integrity подход для критичных каталогов;
- понимание нормального состояния узла.

### Практический результат главы
- ты умеешь быстро сравнить текущий хост с baseline;
- знаешь, где искать неожиданные процессы и сервисы;
- понимаешь, как связать host-level признаки с логами и сетью.

```text
ps auxf
ss -tulpn
systemctl list-units --type=service --state=running
last -a | head
```

---

## 3.4 Практика

### Шаг 1: Сними baseline узла
- зафиксируй список сервисов, пакетов, портов и пользователей;
- это будет точка сравнения перед и после упражнений.

```bash
dpkg -l | head
ss -tulpn
getent passwd | tail
```

### Шаг 2: Проверь host logs
- посмотри sudo, ssh, systemd и kernel сообщения;
- отметь события, которые нужно чаще мониторить.

```bash
sudo journalctl -u ssh -n 50 --no-pager
sudo journalctl -n 100 --no-pager
```

### Шаг 3: Сопоставь процесс и сеть
- для подозрительного listener найди PID, unit и бинарник;
- зафиксируй ownership и путь к файлу.

```bash
sudo lsof -i -P -n | grep LISTEN
readlink -f /proc/$(pgrep -n sshd)/exe
```

### Что нужно явно показать
- baseline listening ports и сервисов;
- журналы sudo, ssh и systemd;
- какие каталоги и unit-файлы считаются критичными;
- как находится владелец подозрительного процесса.

---

## 3.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- на своем стенде создай контролируемое изменение: новый unit-файл или новый listener на тестовом порту, затем зафиксируй, как ты это находишь;
- сравни снимок до и после изменения;
- убедись, что у тебя есть способ быстро отличить deploy от аномалии.

---

## 3.6 Типовые ошибки

- не иметь baseline нормального состояния;
- смотреть только на CPU и память;
- не знать, какие systemd units критичны;
- игнорировать неожиданные listeners и cron jobs.
