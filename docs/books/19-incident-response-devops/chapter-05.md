# Глава 5: Recovery и проверка восстановления

> **Запомни:** восстановление проверяется до инцидента. Бэкап, который ни разу не разворачивали, не считается надежным.

---

## 5.1 Контекст и границы

Recovery — это не просто “есть бэкап”. Нужны RPO и RTO, проверка restore-процесса, сценарий rebuild с нуля и понимание, какие зависимости тянет сервис: DNS, secret store, object storage, БД.

Очень многие команды умеют делать dump, но не умеют воспроизводимо поднимать систему в отдельном окружении.

Эта глава особенно полезна для всех сервисов с данными, конфигурацией и зависимостью от времени восстановления.

---

## 5.2 Как выглядит риск

Типовые слабые места:
- бэкапы есть, но не проверяются;
- непонятно, как восстановить секреты и конфиги;
- restore требует ручных знаний одного человека;
- нет приоритета по сервисам и данным;
- snapshot сохранен, но нет верификации целостности.

### Где особенно важно
- VPS
- docker-compose
- managed DB + app
- малый production

---

## 5.3 Что строит защитник

- четкие RPO и RTO;
- регулярные test restore;
- backup не только данных, но и конфигурации, секретов и инфраструктурного описания;
- инструкция rebuild с нуля;
- последующая проверка целостности и функциональности.

### Практический результат главы
- ты умеешь смотреть на backup как на часть recovery chain;
- знаешь, что нужно для полного восстановления сервиса;
- можешь доказать, что restore действительно работает.

```text
pg_restore --list backup.dump | head
sha256sum backup.tar.gz
```

---

## 5.4 Практика

### Шаг 1: Инвентаризация recovery chain
- выпиши, что нужно для поднятия сервиса с нуля: VM, DNS, proxy, app code, image, secrets, DB, storage;
- отметь, что уже бэкапится, а что нет.

```bash
printf 'vm
dns
proxy
app
secrets
database
'
```

### Шаг 2: Проверь один test restore
- возьми тестовую копию данных и восстанови ее в отдельной среде;
- после этого пройди smoke test приложения.

```bash
pg_restore --list backup.dump | head
```

### Шаг 3: Проверь документацию восстановления
- обнови пошаговый сценарий так, чтобы другой инженер мог пройти его без устных знаний;
- время восстановления зафиксируй.

```bash
printf 'document restore steps and timings
'
```

### Что нужно явно показать
- какие данные и конфиги входят в recovery chain;
- последний успешный test restore;
- какой smoke test подтверждает восстановление;
- какие пробелы остаются.

---

## 5.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- сделай test restore своей lab-БД или конфигурации в отдельную среду;
- после восстановления проверь login, API и основные пользовательские пути;
- зафиксируй фактическое время восстановления и ручные шаги.

---

## 5.6 Типовые ошибки

- не проверять восстановление на практике;
- бэкапить только данные, но не конфиг и секреты;
- не иметь smoke test после restore;
- держать восстановление только в голове одного человека.
