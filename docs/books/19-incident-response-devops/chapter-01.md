# Глава 1: Логи как сырье для защиты

> **Запомни:** расследование почти всегда начинается не с умного инструмента, а с вопроса: есть ли у нас вообще нужные логи и можно ли им доверять.

---

## 1.1 Контекст и границы

Логи — это первичное сырье. Если в них нет времени, идентификатора запроса, IP клиента и результата операции, то ни алертинг, ни расследование потом не соберутся.

Слишком подробные логи тоже вредны: можно утопить систему в шуме или случайно логировать секреты. Нужен баланс между полезной телеметрией и безопасностью.

Эта глава особенно полезна для любого инженера, у которого есть Linux, reverse proxy, приложение и желание понять, что произошло.

---

## 1.2 Как выглядит риск

Типовые слабые места:
- логи разбросаны по разным местам и ротируются слишком рано;
- нет единого формата времени и request id;
- auth, nginx и приложение не коррелируются;
- логи содержат токены, пароли или PII;
- нужные журналы уже потеряны к моменту разбора.

### Где особенно важно
- один VPS
- docker-compose
- малый сервис за nginx
- несколько VM

---

## 1.3 Что строит защитник

- единый формат времени и request id;
- сбор системных, proxy и приложенческих логов;
- rotation и retention по плану;
- запрет чувствительных данных в логах;
- простая схема поиска событий по request id, IP и user.

### Практический результат главы
- ты знаешь, какие логи действительно нужны для расследования;
- можешь быстро найти путь одного запроса через систему;
- понимаешь, чего нельзя логировать даже ради удобства.

```bash
sudo journalctl -u nginx -n 50 --no-pager
sudo journalctl -u myapp -n 50 --no-pager
sudo journalctl -u ssh -n 50 --no-pager
```

---

## 1.4 Практика

### Шаг 1: Собери список источников логов
- выпиши системные сервисы, reverse proxy, приложение, БД и security tools;
- для каждого зафиксируй путь или команду чтения и retention.

```bash
systemctl list-units --type=service --state=running
```

### Шаг 2: Проверь формат и полезность
- открой несколько записей по одному событию и проверь timestamp, IP, user и request id;
- если чего-то нет, добавь это в backlog логирования.

```bash
sudo tail -n 20 /var/log/nginx/access.log
```

### Шаг 3: Найди чувствительные данные
- поиск по token, password, authorization и secret в логах и коде форматирования;
- исключи такие поля из output.

```bash
rg -n "token|password|authorization|secret" /var/log app/ src/ || true
sudo journalctl -u myapp --no-pager | \
  grep -iE "token=|password=|secret=|Authorization: Bearer|api_key=" | \
  head -20
sudo grep -rE "password|token|secret|bearer" /var/log/nginx/ 2>/dev/null | head -10
```

### Что нужно явно показать
- где лежат ключевые журналы;
- какие поля есть в access и app logs;
- как один запрос ищется по всей системе;
- какие логи нуждаются в доработке.

---

## 1.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- сгенерируй на своем стенде один HTTP-запрос и найди его в proxy и приложении;
- сделай один неуспешный логин и найди его в auth logs;
- убедись, что чувствительные данные не логируются.

---

## 1.6 Типовые ошибки

- логировать всё подряд без структуры;
- не хранить логи достаточно долго;
- не иметь корреляции между источниками;
- писать в лог секреты и токены.
