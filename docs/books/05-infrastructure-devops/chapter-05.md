# Глава 5: Стратегия бэкапов

> **Запомни:** Бэкап который не проверялся на восстановление — это не бэкап. Это надежда. А надежда — не стратегия.

---

## 5.1 Правило 3-2-1

```
3 копии данных
2 разных носителя
1 копия вне площадки
```

### Пример

```
Копия 1: Данные на сервере (продакшн)
Копия 2: Бэкап на сервере (/var/backups/)
Копия 3: Бэкап в облаке (Backblaze B2 / S3)
```

| Критерий | Значение |
|----------|----------|
| 3 копии | Оригинал + 2 бэкапа |
| 2 носителя | Диск сервера + облачное хранилище |
| 1 offsite | Облако (не на том же сервере) |

> **Запомни:** Если бэкап только на сервере — сервер умер = бэкап умер.
> Offsite копия — обязательно.

---

## 5.2 Что бэкапить

### Бэкапить

| Что | Зачем | Как |
|-----|-------|-----|
| **База данных** | Все данные приложения | `pg_dump` |
| **Загруженные файлы** | Фото, документы пользователей | `tar` |
| **Конфиги** | `.env`, Nginx, docker-compose | `cp` |

### НЕ бэкапить

| Что | Почему |
|-----|--------|
| Код | Он в git |
| Docker-образы | Они в registry (ghcr.io) |
| Временные файлы | Можно пересоздать |
| Кэш | Не нужен |
| Логи | Не критичны для восстановления |

### Посчитать размер бэкапа

```bash
# Размер базы данных
docker compose exec db psql -U myapp -d myapp_prod -c \
  "SELECT pg_size_pretty(pg_database_size('myapp_prod'));"

# Размер файлов
du -sh /opt/myapp/uploads/

# Итого бэкап = БД + файлы
```

> **Совет:** Посчитай размер бэкапа × 30 дней.
> Узнаешь сколько места нужно в облаке.

## 5.3 Типы бэкапов

### Полный (full)

```
Каждый раз — полная копия всего.
```

| Плюс | Минус |
|------|-------|
| Простой | Занимает больше места |
| Восстановление = один файл | Долго создаётся |

### Инкрементальный

```
Первый раз — полный. Потом — только изменения.
```

| Плюс | Минус |
|------|-------|
| Меньше места | Восстановление сложнее |
| Быстрее | Нужна цепочка бэкапов |

> **Для начала:** только полные бэкапы.
> Инкрементальные — когда база > 10 ГБ.

## 5.4 Где хранить

### Локально (`/var/backups/`)

```
Сервер → backup.sh → /var/backups/myapp/
```

| Плюс | Минус |
|------|-------|
| Быстро | Если сервер умер — бэкап тоже |
| Дёшево | Ограничено диском сервера |

### Удалённо (S3, Backblaze B2)

```
Сервер → backup.sh → rclone → Backblaze B2
```

| Плюс | Минус |
|------|-------|
| Переживёт смерть сервера | Нужно настроить |
| Масштабируется | Небольшая стоимость |

### Стратегия: локально + удалённо

```
backup.sh
  ├──→ /var/backups/myapp/       (7 дней, быстро)
  └──→ rclone → Backblaze B2    (30 дней, надёжно)
```

> **Запомни:** Локально — для быстрого восстановления.
> Удалённо — для katastrophy recovery.

---

## 5.5 Ротация: сколько хранить

```
Ежедневные бэкапы:   7 дней
Еженедельные:        4 недели
Ежемесячные:         12 месяцев
```

### Реализация через `find`

```bash
# Удалить бэкапы старше 7 дней
find /var/backups/myapp/ -maxdepth 1 -type d -mtime +7 -exec rm -rf {} +
```

### Посчитать сколько места займут

```
Размер бэкапа: 500 МБ
Ежедневно:     500 МБ × 7 = 3.5 ГБ
Облако:        500 МБ × 30 = 15 ГБ
```

---

## 5.6 Главное правило

> **Бэкап не существует пока ты не проверил восстановление.**

Не "когда-нибудь проверю". Не "вроде работает".

**Закрой эту книгу и восстанови бэкап на тестовой машине.**

Если не можешь восстановить — бэкапа нет.

## 📝 Упражнения

### Упражнение 5.1: Аудит бэкапов
**Задача:** Составь список что нужно бэкапить на своём проекте:
1. Размер базы данных
2. Размер загруженных файлов
3. Размер конфигов
4. Итого бэкап в день
5. Сколько за месяц (× 30)

### Упражнение 5.2: Стратегия
**Задача:** Реши и запиши:
1. Как часто бэкапить? (ежедневно, ежечасно)
2. Где хранить локально? (`/var/backups/myapp/`)
3. Где хранить удалённо? (Backblaze B2, S3, другой сервер)
4. Сколько хранить? (7 дней локально, 30 дней в облаке)
5. Сколько это стоит? (посчитай объём × цену хранилища)

### Упражнение 5.3: DevOps Think
**Задача:** «У тебя есть бэкапы за 30 дней. Никогда не проверял восстановление. Сервер упал. Каковы шансы что данные восстановятся?»

Ответ:
- 50/50. Бэкап может быть:
  - Повреждён (диск начал сыпаться)
  - Неполный (pg_dump упал тихо)
  - Неприменим (другая версия PostgreSQL)
- Единственный способ узнать — проверить СЕЙЧАС
- Закрой книгу и восстанови бэкап
