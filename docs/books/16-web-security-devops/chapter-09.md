# Глава 9: Итоговый проект

> **Запомни:** итоговый проект по web security должен доказывать не только "я добавил headers", а что приложение прошло через controlled checks по основным trust boundaries.

---

## 9.1 Цель проекта

Взять учебное веб-приложение или свой небольшой сервис и привести его к defensible baseline.

Проект должен закрыть минимум эти зоны:

- логин и reset flow;
- cookie/session policy;
- экранирование пользовательского ввода;
- работа с БД без опасной конкатенации;
- CORS/CSRF/browser boundary;
- secrets и proxy configuration.

---

## 9.2 Стартовая точка

До начала должно быть:
- приложение в lab или на отдельном тестовом стенде;
- доступ к reverse proxy и логам приложения;
- возможность смотреть DevTools браузера;
- snapshot перед правками.

---

## 9.3 Фазы проекта

### Фаза 1: Карта trust boundaries

Запиши:
- какие формы и endpoints принимают внешний ввод;
- где есть login, reset, upload, внешние URL;
- какие cookies выдаёт система;
- какие секреты участвуют в работе.

### Фаза 2: Baseline review

Проверь:
- cookie flags;
- security headers;
- CORS;
- способ формирования SQL-запросов;
- хранение секретов.

### Фаза 3: Исправления

Минимум:
- параметризованные запросы;
- безопасный вывод пользовательских данных;
- `HttpOnly`/`Secure`/`SameSite` где нужно;
- разумные CORS-правила;
- `.env` не в git;
- безопасный proxy baseline.

### Фаза 4: Controlled checks

Только на своём стенде:
- проверка headers и cookies через DevTools/`curl`;
- controlled check на рендер пользовательского ввода;
- проверка login/reset flow;
- review SSRF/upload boundary.

---

## 9.4 Варианты проекта

### Основной вариант

Учебное web-приложение с формами, аутентификацией и базой данных.

### Альтернативный вариант

API-only сервис за reverse proxy:
- auth tokens;
- CORS;
- admin endpoints;
- secrets и headers.

---

## 9.5 Что считать успешным результатом

Проект считается выполненным, если ты можешь показать:
- карту trust boundaries;
- список исправленных слабых мест;
- результаты controlled checks;
- логически понятный final baseline для релиза.

---

## 9.6 Финальный чеклист

- [ ] Логин и reset flow не выдают лишнюю информацию
- [ ] Cookie flags настроены осознанно
- [ ] Пользовательский ввод не рендерится опасно в контролируемых местах
- [ ] SQL-запросы не строятся опасной конкатенацией
- [ ] CORS не открыт шире, чем нужно
- [ ] Секреты не лежат в git и не торчат в конфиге клиента
- [ ] Reverse proxy не доверяет лишним заголовкам
- [ ] Controlled checks проведены только в lab

---

## 9.7 Что сохранить после проекта

Сохрани:
- snapshot lab;
- чеклист baseline;
- список обязательных pre-release проверок для будущих проектов.
