# Глава 1: Аутентификация, сессии и password reset

> **Запомни:** веб-приложение ломается не только через уязвимость в коде, но и через слабую идентификацию, неаккуратные cookie и плохо продуманный reset-flow.

---

## 1.1 Контекст и границы

Защитник смотрит не только на форму логина, а на весь auth lifecycle: регистрация, вход, смена пароля, сброс, logout, срок жизни сессии, блокировки и журналирование.

Даже если пароль хэшируется нормально, систему легко ослабить через бесконечные попытки входа, user enumeration, слишком длинные reset-токены и cookie без нужных флагов.

Эта глава особенно полезна для всех, у кого есть логин, личный кабинет, API с токенами или админка.

---

## 1.2 Как выглядит риск

Типовые слабые места:
- логин отвечает по-разному для существующего и несуществующего пользователя;
- reset-ссылка живет слишком долго или не инвалидируется после использования;
- session cookie доступны JavaScript или уходят по HTTP;
- нет rate limiting на /login и /password-reset;
- в логах нет связки IP, user agent, username и кода ответа.

### Где особенно важно
- личные кабинеты
- внутренние панели
- B2B API с токенами
- админка на отдельном домене

Практический признак user enumeration:

```bash
# Запрос с существующим пользователем
curl -s -X POST https://HOST/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"real@example.com","password":"wrong"}'
# Плохо:
# {"error":"Invalid password"}

# Запрос с несуществующим пользователем
curl -s -X POST https://HOST/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"fake@example.com","password":"wrong"}'
# Плохо:
# {"error":"User not found"}
```

Если текст ошибки отличается, приложение помогает атакующему отличать реальные учётные записи от вымышленных. Правильный результат для обоих случаев один и тот же:

```
{"error":"Invalid credentials"}
```

## 1.3 Что строит защитник

- единая политика cookie: HttpOnly, Secure, SameSite=Lax или Strict, короткий TTL;
- argon2id или bcrypt для паролей и одноразовые reset-ссылки;
- невыдача лишней информации в ошибках логина и reset-flow;
- rate limiting и блокировки по burst-модели;
- централизованный аудит auth-событий на уровне reverse proxy и приложения.

### Практический результат главы
- ты умеешь проверить auth-flow целиком, а не только happy path;
- знаешь, какие флаги должны быть у cookie;
- можешь показать, где хранится session state и как она инвалидируется.

```python
# FastAPI / Starlette
response.set_cookie(
    key="session",
    value=session_id,
    httponly=True,    # JS не читает cookie
    secure=True,      # браузер шлёт только по HTTPS
    samesite="lax",   # не уходит в cross-site POST по умолчанию
    max_age=1800,     # 30 минут
    path="/",
)
```

## 1.4 Практика

### Шаг 1: Проверь HTTP-ответы логина и logout
- сделай несколько успешных и неуспешных попыток входа;
- в DevTools посмотри Set-Cookie, коды ответов и redirect chain;
- зафиксируй, очищается ли cookie на logout и меняется ли session id после входа.

```bash
curl -I https://HOST/login
curl -vk https://HOST/logout
```

Пример того, что нужно увидеть в ответе:

```
< HTTP/2 200
< set-cookie: session=abc123; Path=/; HttpOnly; Secure; SameSite=Lax; Max-Age=1800
```

Что считать проблемой:

```
< set-cookie: session=abc123; Path=/
```

Здесь нет `HttpOnly`, `Secure` и `SameSite`. Такая cookie легче крадётся через XSS, уходит по HTTP и хуже защищена от CSRF.

### Шаг 2: Проверь password reset как отдельный протокол
- создай тестового пользователя и инициируй reset;
- убедись, что повторное использование ссылки не работает;
- сравни срок жизни reset-токена с политикой проекта.

```bash
journalctl -u myapp -n 50 --no-pager
```

### Шаг 3: Поставь видимость на burst логинов
- на уровне nginx или приложения логируй username, IP и request id;
- сделай серию ошибочных входов и проверь, что логи пригодны для алерта.

```bash
sudo tail -f /var/log/nginx/access.log
```

Быстрая агрегация ошибок логина по IP:

```bash
sudo awk '$9 == 401' /var/log/nginx/access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head
```

Пример вывода:

```
     47 185.234.10.11
      3 192.168.1.10
```

Если один IP за короткое время собирает десятки `401`, это либо атака перебором, либо баг клиента.

### Что нужно явно показать
- где именно выставляются флаги cookie;
- как выглядит ошибка входа;
- как выглядит лог одной успешной и одной неуспешной авторизации;
- какой TTL у сессии и reset-токена.

## 1.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- сделай controlled check: несколько неверных логинов подряд на своем тестовом аккаунте и проверь, возникает ли rate limiting;
- посмотри, одинаково ли система отвечает на существующий и несуществующий логин;
- убедись, что reset-ссылка перестает работать после смены пароля.

---

## 1.6 Типовые ошибки

- доверять JWT как магической защите без контроля отзыва и срока жизни;
- оставлять админские cookie без Secure за прокси;
- логировать пароли или reset-токены;
- делать слишком подробные ошибки user not found и password invalid.
