# Глава 2: Initial access

> **Запомни:** начальный доступ часто приходит через уже существующую слабость процесса: открытый сервис, слабый auth, забытая панель или небезопасный workflow.

---

## 2.1 Контекст и границы

Initial access — это не обязательно сложная уязвимость. Часто это комбинация: лишний сервис наружу, старая админка, broad VPN access, устаревший компонент или слишком доверчивый reset и login flow.

Защитнику полезно описывать входные точки не с позиции как пробить, а с позиции что именно мы открыли и чем это подтверждается в логах.

Эта глава особенно полезна для защитника, которому важно понимать, какие входные точки надо закрывать в первую очередь.

---

## 2.2 Как выглядит риск

Типовые слабые места:
- ненужные сервисы опубликованы в интернет — лишний listener сразу становится входной точкой.
  Признак: неожиданные порты во внешнем `nmap`.
  Проверить: `nmap -Pn SERVER_IP`.
- админские панели и тестовые endpoints доступны публично — входной путь появляется не через CVE, а через забытый URL.
  Признак: `200` на `/admin`, `/.env`, `/phpmyadmin`.
  Проверить: `curl` по типовым путям.
- auth endpoint не ограничен по попыткам — одно и то же действие можно повторять без задержки и сигнала.
  Признак: burst неудачных логинов без rate limit и банов.
  Проверить: логи auth и proxy.
- внешние интеграции и file import создают новые входные точки — webhook, upload и import by URL получают тот же уровень доверия, что и обычный пользователь.
  Признак: входные точки не описаны в inventory.
  Проверить: список routes и background ingestion flows.
- нет списка доверенных входов в систему — команда не знает, какие поверхности вообще должны существовать.
  Признак: разные люди называют разные entrypoints.
  Проверить: собрать inventory и сравнить с реальной экспозицией.

### Где особенно важно
- веб-приложения
- SSH и VPN
- SaaS admin панели
- сторонние интеграции

---

## 2.3 Что строит защитник

- минимизация входных точек;
- отдельный management path;
- rate limiting и жесткий auth baseline;
- регулярный review опубликованных сервисов;
- связка входных точек с логами и алертами.

### Практический результат главы
- ты можешь перечислить все initial access surfaces своего стенда;
- понимаешь, какие из них наиболее вероятны и почему;
- можешь показать, как они контролируются и наблюдаются.

```bash
ss -tulpn
nmap -Pn SERVER_IP
```

---

## 2.4 Практика

### Шаг 1: Опиши все входные точки
- выпиши user-facing, admin, machine-to-machine и background ingestion surfaces;
- для каждой точки укажи auth, logging и owner.

```bash
ss -tulpn
```

### Шаг 2: Оцени вероятность и impact
- какие точки самые доступные извне;
- какие из них дают самый большой blast radius при компрометации.

```bash
for path in /admin /wp-admin /phpmyadmin /.env /config.php /_cpanel; do
  code=$(curl -s -o /dev/null -w "%{http_code}" "https://SERVER_IP$path")
  echo "$code $path"
done
```

### Шаг 3: Привяжи входную точку к сигналу
- для логина, SSH, API и панели покажи, где в логах будет видно злоупотребление;
- добавь недостающую телеметрию.

```bash
sudo journalctl -u ssh -n 50 --no-pager
```

### Что нужно явно показать
- полный список входных точек;
- какие из них публичные, а какие внутренние;
- как они защищены и мониторятся;
- какие точки требуют немедленного ужесточения.

---

## 2.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- на своем стенде убери одну лишнюю входную точку и сравни карту экспозиции до и после;
- проверь, что попытки входа на оставшиеся auth endpoints видны в логах;
- не проводи практические действия по получению доступа, цель здесь — defensive mapping.

---

## 2.6 Типовые ошибки

- думать только про веб, забывая SSH, VPN и import workflows;
- не иметь owners для access surfaces;
- не связывать входную точку с наблюдаемостью;
- оставлять старые тестовые панели публичными.
