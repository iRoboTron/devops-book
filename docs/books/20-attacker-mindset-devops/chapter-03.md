# Глава 3: Credential abuse

> **Запомни:** учетные данные часто используются не через взлом криптографии, а через слабые процессы: reuse, broad access, плохая ротация и отсутствие сигналов злоупотребления.

---

## 3.1 Контекст и границы

Credential abuse — это защитная тема про учетные данные как поверхность атаки. Важно понимать, какие типы учеток есть, как они выдаются, где живут, как отзываются и как выглядит их злоупотребление в логах.

Здесь особенно вредны shared accounts, long-lived tokens и отсутствие отличий между обычным и подозрительным использованием.

Эта глава особенно полезна для всех, у кого есть пароли, API keys, SSH keys, tokens и shared accounts.

---

## 3.2 Как выглядит риск

Типовые слабые места:
- shared admin accounts;
- отсутствие MFA там, где оно возможно;
- длинноживущие API keys без owner и срока жизни;
- нет сигналов на burst failed logins, новые ключи и необычные login times;
- SSH keys и tokens не ревьюятся годами.

### Где особенно важно
- SSH
- веб-логин
- CI/CD
- cloud API
- service accounts

---

## 3.3 Что строит защитник

- индивидуальные учетки;
- MFA и short-lived credentials, где применимо;
- инвентаризация ключей и токенов;
- журналирование auth success, failure и new credential events;
- регулярная ротация и отзыв старых учетных данных.

### Практический результат главы
- ты понимаешь, какие credentials в системе самые рискованные;
- можешь показать жизненный цикл ключа или токена;
- знаешь, какие аномалии должны поднимать расследование.

```text
ls -la ~/.ssh
cat ~/.ssh/authorized_keys
last -a | head
```

---

## 3.4 Практика

### Шаг 1: Инвентаризация учетных данных
- перечисли типы credentials: user passwords, SSH keys, CI tokens, cloud keys и app secrets;
- для каждого укажи owner, срок жизни и место хранения.

```bash
printf 'credential
owner
location
rotation
'
```

### Шаг 2: Проверь жизненный цикл
- посмотри, как новая учетка выдается и как старая отзывается;
- отметь shared accounts и long-lived tokens.

```bash
last -a | head
```

### Шаг 3: Привяжи к сигналам
- какие события по credentials видны в логах: failed logins, new keys, sudo, token use;
- где не хватает телеметрии.

```bash
sudo journalctl -u ssh -n 100 --no-pager
```

### Что нужно явно показать
- какие credentials существуют в инфраструктуре;
- какие из них shared или long-lived;
- как происходит отзыв доступа;
- какие события считаются подозрительными.

---

## 3.5 Lab-only проверка

Все проверки в этой главе выполняются только на своих VM, контейнерах, тестовых доменах и собственных сервисах.

- на своем стенде проведи безопасный review SSH keys или тестовых токенов и удали один устаревший доступ;
- проверь, как в логах выглядит несколько неуспешных логинов подряд на тестовую учетку;
- не автоматизируй перебор и не выходи за пределы собственных систем.

---

## 3.6 Типовые ошибки

- держать shared admin accounts;
- не знать владельца токена;
- не удалять старые SSH keys;
- не иметь сигналов на подозрительные auth события.
